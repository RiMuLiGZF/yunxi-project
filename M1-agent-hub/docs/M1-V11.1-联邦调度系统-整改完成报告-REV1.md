## 二、P0 级缺陷修复详情

### 2.1 P0-001：PII 脱敏逻辑不可达

**问题根源**：原实现中 `sanitize_content` 方法以 `risk_level == "high"` 作为唯一脱敏阈值，导致 low/medium 风险 PII 直接跳过脱敏分支。

**修复方案**：
- 重写 `_assess_risk_level` 方法，采用加权评分制
  - 风险分 = Σ(单类 PII 权重 × 数量) × 涉密等级权重
  - 风险等级：none / low / medium / high / critical（五级）
- 修改 `sanitize_content` 逻辑：**所有检测到 PII 的内容均进入脱敏分支**
- 根据风险等级采用不同脱敏强度（critical 强 → low 轻）

**验证**：5 个风险等级测试全部通过，low 风险 `was_modified=True`。

### 2.2 P0-002：API Key 明文存储

**问题根源**：`ExternalAgentRegistry._api_keys` 以明文 dict 存储，存在内存泄露风险。

**修复方案**：
1. 新增 `federation/crypto_utils.py` 加密工具模块
   - 基于 `cryptography.Fernet`（AES-128-CBC + HMAC-SHA256）
   - 主密钥从 `FEDERATION_MASTER_KEY` 环境变量读取
   - 开发模式自动生成并告警
2. 注册表改用 `_api_keys_encrypted` 存储加密后的 Key
3. `get_api_key()` 增加 `caller_id` 参数，实现调用者鉴权
   - 受信任组件（白名单 8 个）：返回明文
   - 未鉴权调用：返回脱敏预览
4. 日志中 API Key 自动脱敏（仅显示前 4 后 4 位）
5. 支持主密钥轮换 `rotate_all_keys()`

**验证**：8 个加密存储测试全部通过，内存中无明文，不受信任调用返回脱敏值。

## 三、P1 级缺陷修复详情

### 3.1 P1-001：PII 检测正则绕过

**问题根源**：正则仅匹配标准格式，可通过零宽字符、全角字符、[at]/[dot] 替换等手法绕过。

**修复方案**：新增 `_normalize_content` 预处理归一化层
- 移除零宽字符（U+200B, U+200C, U+200D, U+2060, U+FEFF 等）
- 全角字符转半角（0xFF01-0xFF5E 映射）
- [at] / (at) / [dot] / (dot) 还原
- 手机号空格/横杠分隔归一化

**额外增强**：
- 手机号正则增加前后数字断言，防止误匹配身份证中的数字段
- PII 有效性二次校验（身份证校验码、银行卡 Luhn、手机号长度）

**验证**：9 个检测增强测试全部通过，包括零宽字符、全角字符、[at] 替换等绕过手法均被有效检测。

### 3.2 P1-002：7 类 PII 脱敏未实现

**问题根源**：原实现仅实现了 email、phone_cn、custom_keyword 3 类脱敏。

**修复方案**：补全 7 类 PII 脱敏方法，总计 10 类全覆盖

| PII 类型 | 严重度 | 脱敏策略（high 强度） |
|----------|--------|----------------------|
| id_card_cn | critical | 18 星号全替换 |
| bank_card | critical | **** **** **** **** |
| api_key | critical | sk-\*\*\*\*[REDACTED]\*\*\*\* |
| password | critical | \*\*\*\*[PASSWORD]\*\*\*\* |
| token | high | Bearer \*\*\*\*[REDACTED]\*\*\*\* |
| private_key | critical | 完整替换为 REDACTED 标记 |
| url_internal | medium | [INTERNAL_URL_REDACTED] |

每类均支持 critical/high/medium/low 四级脱敏强度。

### 3.3 P1-003：接口权限校验缺失

**问题根源**：联邦调度所有 HTTP 接口可匿名访问，存在越权风险。

**修复方案**：新增 `_register_auth_middleware` 鉴权中间件

**权限矩阵**：

| 级别 | 接口 | 说明 |
|------|------|------|
| admin | 注册/删除 Agent、设置预算、审计日志 | 最高权限 |
| write | 调用 Agent、对比执行 | 写操作 |
| read | 列表查询、决策、健康检查、隐私扫描 | 读操作 |

**鉴权方式**：
1. **Admin API Key**：`Authorization: Bearer <key>`，密钥来自 `FEDERATION_ADMIN_KEY`
2. **内部调用签名**：`X-Internal-Call: true` + `X-Signature`（HMAC-SHA256）+ `X-Timestamp`（5 分钟窗口）

所有 `/v1/federation/` 下接口均受保护，非联邦接口不受影响。

### 3.4 P1-004：GPL 协议风险提示

**问题根源**：注册外部 Agent 时无协议合规检查，接入 GPL 类传染性协议 Agent 可能带来开源义务风险。

**修复方案**：
1. 新增 `LicenseType` 枚举（10 种协议类型）
2. `ExternalAgentProfile` 增加 `license` 字段
3. 注册时 GPL 类协议（GPL-2.0/GPL-3.0/AGPL/LGPL）必须设置 `confirm_license_risk=True`
   - 未确认：抛出 `ValueError`，包含风险说明
   - 已确认：正常注册，日志记录风险确认状态

### 7.2 生成主密钥

```python

print(Fernet.generate_key().decode())
```

```python

r = ExternalAgentRegistry()
result = r.rotate_all_keys(new_master_key="新的主密钥")

```
