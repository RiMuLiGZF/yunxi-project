### TC-CP-01：多场景快速切换并发测试

| 项目 | 内容 |
|------|------|
| **测试场景** | 模拟 M4 场景引擎在短时间内连续发起 6 种场景模式切换请求（模式A-F），每种模式并发 16~17 次，总计 100 次 `orchestrate.scene_switch` 调用 |
| **前置条件** | 1. M1 Orchestrator-Agent 已注册并就绪；2. 场景映射矩阵已加载（6种模式定义完备）；3. /health 和 /ready 返回正常；4. 联调阶段分身池配额：全局上限 50，单父 Agent 上限 5 |
| **执行步骤** | 1. 初始化 Orchestrator-Agent 实例；2. 构造 100 个 AgentTask（intent=orchestrate.scene_switch），按 6 种模式均匀分配，payload 包含 old_scene / new_scene / context_handover / involved_agents / new_scene_agents；3. 使用 asyncio.gather 并发提交全部 100 个 task；4. 等待全部 task 返回 AgentResult；5. 统计成功率、延迟分布、错误类型 |
| **判定标准** | 1. 100 个 scene_switch 全部返回 status=success（成功率 100%）；2. P50 延迟 ≤ 50ms，P99 ≤ 200ms；3. 每种模式的 released_agents 和 initialized_agents 数量与输入一致；4. 无 agent_id 冲突、无 DAG 注册表残留脏数据；5. 无安全违规（涉密标记未篡改） |
| **观测指标** | success_rate, p50_latency_ms, p99_latency_ms, avg_latency_ms, error_count_by_type, dag_registry_clean_count, security_violation_count |

### TC-CP-02：多模块混合任务并发测试

| 项目 | 内容 |
|------|------|
| **测试场景** | 模拟 100 个混合任务并发到达 M1，覆盖 M1→M2（Skill 调用）、M1→M3（推理委托）、M1→M5（记忆查询）三条跨模块链路，每条链路各占约 1/3 |
| **前置条件** | 1. 所有 8 个子 Agent 已初始化并 mount；2. Budget-Agent 日预算设为足够高（$1000）以避免熔断干扰；3. Bus-Agent MessageBus 单例已就绪；4. Discovery-Agent 至少有 3 个模拟 Agent 注册（覆盖 M2/M3/M5 能力） |
| **执行步骤** | 1. 注册 3 个模拟 Agent（m2.skill、m3.inference、m5.memory）到 AgentRegistry；2. 构造 100 个 AgentTask：<br>   - 34 个 target=agent.m2.skill（budget.request_quota + skill.invoke）<br>   - 33 个 target=agent.m3.inference（inference.chat）<br>   - 33 个 target=agent.m5.memory（memory.query，security_classification=INTERNAL）<br>3. 全部 task 设置不同的 priority（1~10 均匀分布）；4. 通过 Bus-Agent route_message 并发路由全部 100 个 task；5. 验证 BusMessage 中 security_classification 传递正确性（M5 消息应全部为 TOP_SECRET）；6. 统计路由成功率、优先级排序正确性、涉密标记准确率 |

| **观测指标** | route_success_rate, m5_top_secret_accuracy, non_m5_classification_preserve_rate, priority_order_correctness, msg_id_collision_count, avg_route_latency_ms |

### TC-CP-03：长时间持续负载压力测试

| 项目 | 内容 |
|------|------|
| **测试场景** | 以 20 QPS 稳定速率持续向 M1 提交任务，持续运行 300 秒（总计约 6000 个任务），验证系统在持续负载下的资源稳定性、内存不泄漏、调度不降级 |
| **前置条件** | 1. M1 全部子 Agent 就绪；2. 预热 100 个任务（不计入测试结果）；3. 监控进程内存和 CPU 采集工具就绪；4. Budget-Agent 日预算设为 $10000（避免熔断）；5. 分身池配额：全局上限 100，单父 Agent 上限 10（正常生产配置） |
| **执行步骤** | 1. 启动内存/CPU 采样（每 10 秒记录一次）；2. 启动 20 QPS 任务生成器，每 50ms 生成一个 AgentTask（intent 随机选择 orchestrate.build / bus.publish / budget.check / security.check_input / discovery.evaluate）；3. 每个任务通过 asyncio.gather 并发池（并发度上限 20）提交；4. 每 60 秒采样一次：检查 DAG 注册表大小、Bus-Agent 消息计数、Budget 预算余额、Security 审计日志条目数；5. 300 秒后停止生成，等待剩余任务完成；6. 检查最终状态：内存是否回落、DAG 注册表是否可正常查询、审计日志是否完整 |
| **判定标准** | 1. 6000 个任务成功率 ≥ 99%（允许 ≤ 60 个失败）；2. P50 延迟在 300 秒期间波动 < 20%（无持续劣化）；3. 内存增量 < 100MB（线性增长可接受，指数增长不可接受）；4. CPU 峰值 < 80%；5. DAG 注册表最终大小 = 成功构建的 DAG 数量（无残留脏数据）；6. 审计日志条目数 = security.check_input 调用次数（无丢失） |
| **观测指标** | total_tasks, success_rate, p50_latency_trend(数组), memory_peak_mb, memory_delta_mb, cpu_peak_pct, dag_registry_size, audit_log_count, error_rate_by_minute |

| **前置条件** | 1. M1 空闲状态（无 pending 任务）；2. Budget-Agent 日预算设为 $1（低预算，触发熔断验证）；3. Arbiter-Agent 预设 5 个等待边（模拟部分 Agent 已有等待关系）；4. Discovery-Agent 2 个模拟 Agent 注册（一高一低负载） |
| **执行步骤** | 1. 记录基线状态（内存、CPU、DAG 注册表大小、预算余额）；2. 瞬间并发提交 150 个任务（100 个 priority=1 + 50 个 priority=5），全部使用 asyncio.gather 无延迟提交；3. 监控 Budget-Agent 熔断触发情况（日预算 $1 应在若干任务后触发熔断）；4. 监控 Arbiter-Agent 死锁检测（预设等待边不应产生误报）；5. 等待全部 150 个任务返回；6. 统计：熔断任务数、正常完成数、优先级排序正确性、系统恢复时间 |
| **判定标准** | 1. 100 个高优先级任务中，前 80% 应优先于普通任务被处理（队列头部统计）；2. Budget-Agent 在预算耗尽后触发熔断，后续任务收到 circuit_open 状态（非崩溃）；3. Arbiter-Agent 不因高并发产生误报死锁（预设 5 边检测正确）；4. 系统在全部 150 任务完成后 5 秒内恢复空闲状态（内存回落至基线±50MB）；5. 无 msg_id 冲突、无未捕获异常 |
| **观测指标** | high_priority_first_ratio, circuit_triggered_count, circuit_rejected_count, deadlock_false_positive, recovery_time_ms, peak_memory_mb, total_error_count |

## 4. 降级与容错预案

### 4.1 压力等级定义

| 等级 | 并发数 | QPS | 持续时间 | 预期系统状态 |
|------|--------|-----|---------|------------|
| L1 正常 | 10 | 5 | 60s | 全功能正常，无降级 |
| L2 中等 | 30 | 15 | 120s | 全功能正常，延迟微增 |
| L3 较高 | 50 | 25 | 180s | 部分非核心功能降级 |
| L4 高压 | 100 | 50 | 300s | 多级降级启动，熔断可能触发 |
| L5 极端 | 200 | 100 | 60s | 全面降级，仅保留核心链路 |

### 4.2 分级降级策略

| 压力等级 | 降级动作 | 影响范围 |
|---------|---------|---------|
| L1-L2 | 无降级 | 全功能运行 |
| L3 | 1. 分身池全局上限降至 30，单父 Agent 上限降至 3<br>2. 日志采样率从 100% 降至 50%（仅保留 warning/error）<br>3. 快照频率从每次 DAG 变更降至每 5 次变更一次 | 分身创建受限、诊断粒度降低、快照延迟增加 |
| L4 | 1. 禁止新模式 F（断点续跑）和模式 E（仲裁）的新任务进入<br>2. Budget-Agent 仅检查 REQUEST 级别，跳过 SESSION/DAILY/MONTHLY 级检查<br>3. 涉密分级仅执行第一级正则匹配，跳过关键词密度分析<br>4. Bus-Agent DLQ 重试次数从 3 降至 1<br>5. 分身池全局上限降至 10 | 非核心场景暂停、预算管控简化、涉密检测降级、死信重试减少、分身几乎不可用 |
| L5 | 1. 仅保留模式 A（简单任务直调）和模式 B（DAG 编排）<br>2. 禁止所有 M5 记忆查询（涉密传输开销过大）<br>3. 禁止分身池创建<br>4. Arbiter-Agent 仅执行 AUTO_RESOLVE 一级仲裁，跳过 NEGOTIATE 和 HUMAN_ESCALATE<br>5. 端云调度强制 LOCAL_FIRST，禁止 CLOUD_FIRST | 仅核心调度运行、M5 不可用、无分身、仲裁简化、禁止云端推理 |

### 4.3 故障恢复机制

| 故障类型 | 检测方式 | 恢复动作 | 恢复时间目标 |
|---------|---------|---------|------------|
| DAG 注册表内存溢出 | DAG 注册表大小 > 10000 | 触发 Lifecycle-Agent 批量 ARCHIVE 已完成 DAG（completion_rate=100%），释放内存 | ≤ 5s |
| Budget-Agent 预算耗尽 | budget.check 返回 false | 返回 deny + 等待窗口（60s 后重试）；如为 REQUEST 级别则立即拒绝 | 即时拒绝 |
| Bus-Agent 消息堆积 | 队列深度 > 1000 | 启动批量消费（batch_size=50），提升消费速率；超过 5000 时触发队列截断（丢弃 priority > 7 的低优先级消息） | ≤ 10s |
| Security-Agent 审计日志溢出 | 审计日志 > 90000 条 | FIFO 淘汰最旧条目（正常机制）；超过 95000 时加速淘汰（每 100 条淘汰 10 条） | 即时淘汰 |
| Arbiter-Agent Wait-For 图过大 | 等待边数 > 500 | 触发 AUTO_RESOLVE 批量清理：超时 120s 的等待边自动标记为超时并移除 | ≤ 5s |
| Discovery-Agent 负载评分异常 | VRAM/CPU 返回值为负数或 NaN | 重置该 Agent 负载评分为默认值 0.5，标记 health=degraded，15s 后重新探测 | ≤ 15s |
| ClonePool 分身泄漏 | TTL 过期分身 > 20% | 立即触发 cleanup_expired 全量清理；同时将 cleanup_interval 从 60s 降至 10s | ≤ 1s |
