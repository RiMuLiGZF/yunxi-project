## 一、当前七层架构现状

```
V9 OrchestratorV9          [编排入口]  Guardrails + Ledger + Budget + Intent
  ↓
V8 OrchestratorV8          [执行委托]  Pass-through + Tracer + TraceToMemory
  ↓
V7 OrchestratorV7          [调度增强]  Ensemble + Budget-aware + Retry
  ↓
V5 OrchestratorV5          [扩展能力]  SemanticRouter + GuardrailsV2 + LoopGuard
  ↓
V4 OrchestratorV4          [事件驱动]  Streaming + A2A + LLMProvider
  ↓
V3 OrchestratorV3          [语义路由]  IntentClassification + ClassificationRouter
  ↓
V2 OrchestratorV2          [基础执行]  AgentRegistry + TaskDispatcher + MessageBus
```

### 各层职责与问题

| 层级 | 组件 | 职责 | 问题 |
|------|------|------|------|
| V9 | OrchestratorV9 | 编排入口、Guardrails、Ledger、Budget预检、GroupChat | 职责过重，集成所有高层能力 |
| V8 | OrchestratorV8 | V8透传、Tracer、TraceToMemory | 中间层，仅为历史兼容 |
| V7 | OrchestratorV7 | Ensemble多Agent投票、Budget-aware、RetryCoordinator | 与V9 BudgetManager功能重叠 |
| V5 | OrchestratorV5 | SemanticRouter、GuardrailsV2、LoopGuard、VectorMemory、PluginLoader、MCPServer | **严重越界**：VectorMemory/PluginLoader/MCPServer属其他模块 |
| V4 | OrchestratorV4 | StreamingEngine、A2AProtocol、LLMProvider | **轻微越界**：LLMProvider推理与M3重叠 |
| V3 | OrchestratorV3 | 意图分类、ClassificationRouter | 轻量级，可合并 |
| V2 | OrchestratorV2 | AgentRegistry、TaskDispatcher、MessageBus、DeadLetterQueue | 核心执行层，应保持独立 |

## 二、简化方案评估

### 方案A：三层架构（推荐）

```
┌─────────────────────────────────────────────┐
│           编排决策层 (Orchestration)          │
│  Guardrails + Ledger + Budget + GroupChat    │
│  + IntentClassification + Ensemble + Retry   │
└─────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│           调度执行层 (Scheduling)             │
│  AgentRegistry + TaskDispatcher + MessageBus │
│  + SemanticRouter + CircuitBreaker           │
└─────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│           扩展适配层 (Extension)              │
│  StreamingEngine + A2AProtocol + HTTPTransport│
│  + MemoryBridge + SkillsInterface            │
└─────────────────────────────────────────────┘
```

**优点：**
- 委托链从7层减少到3层，调用栈深度减少57%
- 每层职责清晰，符合单一职责原则
- 消除V8历史兼容层的维护成本
- 消除V5越界组件的架构合理性（已迁移标记）

**缺点：**
- 编排决策层仍然较重，未来可进一步拆分
- 需要重构OrchestratorV9的委托逻辑

### 方案B：四层架构（折中）

```
V9 编排层：Guardrails + Ledger + Budget + GroupChat
V5/V7 调度扩展层：SemanticRouter + Ensemble + Retry + GuardrailsV2
V4/V3 协议层：A2A + Streaming + IntentClassification
V2 执行层：Registry + Dispatcher + Bus
```

**优点：**
- 保留了一定的分层粒度
- 迁移工作量较小

**缺点：**
- 仍然比三层复杂
- V5越界组件问题未彻底解决

## 三、Benchmark对比（模拟评估）

基于代码结构分析，评估简化前后的性能影响：

| 指标 | 七层架构 | 三层架构 | 提升 |
|------|---------|---------|------|
| 简单查询调用栈深度 | 7层 | 3层 | **-57%** |
| 简单查询函数调用数 | ~14次 | ~6次 | **-57%** |
| 复杂任务调用栈深度 | 7层 | 3层 | **-57%** |
| 复杂任务函数调用数 | ~35次 | ~20次 | **-43%** |
| 每请求委托开销 | ~0.5ms | ~0.2ms | **-60%**（估算） |
| 代码文件数 | 40+ | 30+ | **-25%** |
| 维护复杂度 | 高 | 中 | **显著降低** |

**说明：**
- 当前简单查询已通过 `_is_simple_query` 预筛短路，不受七层影响
- 复杂请求（多Agent协作、Ledger跟踪、Retry等）每次委托涉及函数调用+可能的上下文切换
- 三层架构下，OrchestratorV9直接调用TaskDispatcher，跳过V8/V7/V5/V4/V3中间层

## 四、V10.0 实施结论

### 已完成的架构清理

1. **R01 [P0]** VectorMemory → 标记DEPRECATED，待迁移至模块5
2. **R02 [P0]** PluginLoader + MCPServer → 标记DEPRECATED，待迁移至模块2
3. **R03 [P0]** LLMProvider推理执行 → 标记DEPRECATED，保留InferenceRouter路由决策
4. **R08 [P2]** 七层简化评估 → 提供三层架构方案与benchmark

### 未实施的架构重构（建议V11.0进行）

- 将OrchestratorV9中的V8/V7/V5/V4/V3委托链合并为"编排决策层"
- 移除V8历史兼容层
- 将EnsembleEngine、RetryCoordinator、SemanticRouter统一纳入"调度扩展层"

### 风险与建议

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 三层合并导致编排层过重 | 中 | V11.0可拆分为"编排策略"+"安全检查"两个子层 |
| V8移除破坏向后兼容 | 低 | V8透传方法已标记，内部项目可批量替换 |
| 性能提升不如预期 | 低 | Python函数调用开销极小，主要收益在维护复杂度 |

**结论：建议V11.0实施三层架构重构，V10.0先完成职责剥离和接口补全。**
