# 第一部分：云汐系统全景

## 1.1 系统架构总览

云汐系统采用**模块化微服务架构**，11 个模块（M1-M11）各自独立部署，通过 HTTP API 通信。每个模块有自己的端口、数据存储和技术栈，但都遵循统一的 M8 纳管协议（`/m8/health`, `/m8/metrics`, `/m8/config`）。

### 11 个模块一览

| 模块 | 名称 | 端口 | 核心定位 | 一句话说明 |
|------|------|------|----------|-----------|
| **M1** | 多 Agent 集群调度 | 8001 | 大脑中枢 | 联邦调度 8 个子 Agent + 16+ 外部 Adapter，智能路由任务 |
| **M2** | 技能集群 | 8002 | 能力扩展层 | 18 个内置技能 + MCP/A2A 协议接入外部能力 |
| **M3** | 端云协同内核 | 8003 | 混合调度层 | 本地/云端智能路由，离线降级，数据同步 |
| **M4** | 业务场景引擎 | 8004 | 场景调度层 | 工作/学习/生活/休闲场景自动识别与切换 |
| **M5** | 潮汐分层记忆 | 8005 | 记忆系统 | 四层潮汐模型（L0-L3），向量检索，睡眠巩固 |
| **M6** | 穿戴硬件外设 | 8006 | 硬件接入层 | 6 种设备模拟，SSE 实时数据推送 |
| **M7** | 积木编排平台 | 8007 | 工作流层 | DAG 可视化工作流编排，拖拽式自动化 |
| **M8** | 管理控制塔 | 8008 | 管理枢纽 | 用户认证、模块纳管、算力调度、系统监控 |
| **M9** | 编程开发 | 8009 | 开发工具层 | VSCode 管理、多语言代码执行、项目管理 |
| **M10** | 系统卫士 | 8010 | 运维守护 | 资源监控、进程管理、四级防护、审计日志 |
| **M11** | MCP 总线服务 | 8011 | 工具总线 | 统一管理所有 MCP 服务的注册、发现、路由、调用 |

### 关键设计理念

1. **联邦而非单体**：M1 不直接"拥有"能力，而是调度各个 Agent 和模块协同完成任务
2. **记忆独立**：M5 作为独立模块存在，所有模块通过 API 访问记忆，而非各自存储
3. **M8 纳管协议**：所有模块必须实现 `/m8/health`、`/m8/metrics`、`/m8/config` 三个标准接口，M8 通过这套协议统一监控和管理
4. **MCP 总线**：M11 是工具调用的标准总线，所有工具能力通过 MCP 协议注册和发现
5. **端云协同**：M3 确保系统在断网时也能降级运行，保护隐私的同时利用云端算力

## 1.3 数据流和请求链路

理解一个系统最好的方式，是跟着一个请求走一遍完整的生命周期。

### 典型请求链路：用户发一条消息

```
用户输入
  │
  ▼
前端 (index.html)
  │  POST /api/chat
  ▼
M8 管理控制塔
  │  认证鉴权 → 路由转发
  ▼
M1 多 Agent 集群调度
  │
  ├─→ 意图识别 (intent_classifier)
  │     识别用户意图：闲聊 / 查询 / 任务 / 技能调用
  │
  ├─→ 联邦调度 (federation_scheduler)
  │     选择最合适的 Agent / 模型处理
  │
  ├─→ M5 潮汐记忆检索
  │     查询相关历史记忆 → 注入 Prompt
  │
  ├─→ （如需工具）M11 MCP 总线
  │     工具发现 → 路由调用 → 结果返回
  │
  ├─→ （如需技能）M2 技能集群
  │     技能匹配 → 执行 → 结果返回
  │
  └─→ LLM 生成回复
        │
        ▼
M1 → M8 → 前端 → 用户
        │
        └─→ M5 记忆写入
             新对话存入 L0 沙滩层 → 逐步巩固到 L1/L2/L3
```

### 关键链路节点详解

**1. M8 控制塔：入口与认证**
- 用户请求首先到达 M8，通过 JWT Token 验证身份
- M8 根据请求路径路由到对应模块的业务接口
- 所有操作记录审计日志

**2. M1 调度中心：大脑的决策**
- `intent_classifier.py`：意图分类器，判断用户要什么
- `semantic_router.py`：语义路由，根据意图选择处理路径
- `orchestrator_v9.py`：编排器，协调整个处理流程
- `llm_provider.py`：统一的 LLM 调用入口，支持多模型切换

**3. M5 记忆系统：信息的存取**
- 读路径：根据对话语义 → 向量检索 → 返回相关记忆片段
- 写路径：新对话 → L0 内存层 → 定时巩固到 L1/L2 → 睡眠时压缩到 L3
- 权限：三级域隔离（私有/共享/全局）+ 四级密级标记

**4. M11 MCP 总线：工具的标准化**
- 所有工具能力通过 MCP 协议注册到 M11
- M1 需要工具时，通过 M11 统一调用
- M11 负责鉴权、限流、审计、故障转移

**5. M10 系统卫士：幕后的守护者**
- 全程监控各模块的 CPU / 内存 / 进程状态
- 异常时自动降级或重启
- 四级防护策略（正常→关注→警告→紧急）

# 第二部分：核心技术详解

## 2.1 FastAPI 与异步 Python Web

### 是什么

FastAPI 是一个现代、高性能的 Python Web 框架，基于标准 Python 类型提示，自动生成 OpenAPI 文档。它的核心特点：

- **异步原生**：基于 ASGI（Uvicorn），原生支持 `async/await`
- **类型驱动**：用 Pydantic 做数据验证，Python 类型提示即 API 契约
- **自动文档**：代码写完，Swagger UI 和 ReDoc 自动生成
- **性能接近 Node.js / Go**：在 Python Web 框架中性能顶尖

### 在云汐中的应用

云汐所有 11 个模块的后端服务都基于 FastAPI + Uvicorn 构建。典型结构（以 M5 为例）：

```
M5-tide-memory/
├── server.py          # 启动入口：uvicorn.run(...)
└── src/tide_memory/
    ├── api/           # API 路由（APIRouter）
    ├── middleware/    # 中间件
    └── core/          # 配置与模型（Pydantic）
```

**关键应用模式：**

1. **M8 标准接口**：每个模块都实现 `/m8/health`、`/m8/metrics`、`/m8/config` 三个接口，作为 M8 纳管的统一契约
2. **APIRouter 模块化**：路由按业务领域拆分到不同文件，通过 `include_router` 挂载
3. **依赖注入**：通过 `Depends()` 管理数据库会话、认证、配置等依赖
4. **Pydantic v2 模型**：所有请求/响应都有明确的模型定义，自动验证

### 学习建议

| 维度 | 建议深度 | 说明 |
|------|---------|------|
| 基础使用 | **精通** | 路由、请求模型、响应模型、依赖注入，必须熟练 |
| 异步编程 | **熟练** | `async/await`、并发控制、异步数据库驱动 |

## 2.2 SQLAlchemy ORM 与 SQLite

### 是什么

**SQLAlchemy** 是 Python 生态最成熟的 ORM（对象关系映射）框架，让你用 Python 类和对象操作数据库，而不是写原生 SQL。

**SQLite** 是一个轻量级的嵌入式关系数据库，整个数据库就是一个文件，无需独立服务器进程，非常适合中小规模应用和本地部署。

### 在云汐中的应用

**SQLite 数据库分布：**

| 数据库 | 位置 | 主要表 |
|--------|------|--------|
| M8 主库 | `M8-control-tower/backend/data/m8.db` | users, audit_logs, system_settings, compute_sources, workflows... |
| M5 记忆库 | `data/memory/l1_shallow.db`, `l2_deep.db` | 记忆条目、情绪标签、审计日志 |
| M6 硬件库 | `data/m6.db` | 设备、传感器数据 |
| M11 MCP 库 | `~/.yunxi/m11_bus.db` | MCP 服务、工具、调用记录、API Key |

**SQLAlchemy 使用模式（以 M8 为例）：**

```python
# models.py — 数据模型定义
class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True)
    username = Column(String, unique=True, index=True)
    password_hash = Column(String)
    role = Column(String, default="viewer")

# repositories/ — Repository 模式，封装数据访问
class UserRepository:
    def get_by_username(self, db, username):
        return db.query(User).filter(User.username == username).first()
```

**M5 的特殊设计：**
- L0（沙滩层）：纯内存，不落地
- L1（浅水层）：SQLite 存储结构化记忆
- L2（深水层）：SQLite + 向量索引（FAISS）
- L3（深海层）：文件系统 + AES-256-GCM 加密

### 学习建议

| 维度 | 建议深度 | 说明 |
|------|---------|------|
| 基本 CRUD | **精通** | 查询、过滤、排序、分页、增删改 |
| 关系映射 | **熟练** | 一对一、一对多、多对多、级联操作 |
| 会话管理 | **掌握** | Session 生命周期、事务、连接池 |
| 性能优化 | **了解** | 索引、N+1 问题、eager loading |
| 迁移工具 | **了解** | Alembic 数据库迁移（当前云汐未使用，后续可能接入） |

**关于 SQLite 的注意事项：**
- SQLite 是单文件数据库，并发写入性能有限
- 云汐目前各模块独立数据库，天然分片，问题不大
- 如果后续并发量上去，需要考虑迁移到 PostgreSQL

## 2.3 向量数据库与 Embedding

### 是什么

**Embedding（嵌入/向量化）**：把文本、图像等非结构化数据转换成高维空间中的向量（比如 768 维或 1536 维的浮点数数组）。语义相近的内容，在向量空间中的距离也更近。

**向量数据库**：专门存储和检索向量的数据库，核心能力是**相似度搜索**——给定一个查询向量，快速找出库中最相似的向量。

常用技术：
- **sentence-transformers**：Python 库，基于预训练模型生成文本 Embedding
- **FAISS**：Facebook 开源的向量相似度搜索库，性能极强
- **ChromaDB**：轻量级向量数据库，封装了 Embedding + 存储 + 检索

### 在云汐中的应用

向量检索是 M5 潮汐记忆系统的核心。没有向量检索，记忆系统就只是一个简单的对话日志。

**记忆检索流程：**

```
用户输入 "我上次说的那个项目怎么样了？"
        │
        ▼
  生成查询向量 (sentence-transformers)
        │
        ▼
  FAISS 向量搜索（L2 层记忆库）
        │
        ▼
  Top-K 相似记忆片段
        │
        ▼
  注入 LLM Prompt → 生成基于记忆的回复
```

**M5 四层模型中的向量角色：**
- **L0 沙滩层**：无向量，纯内存最新对话
- **L1 浅水层**：结构化存储，关键词检索为主
- **L2 深水层**：向量化存储，FAISS 语义检索
- **L3 深海层**：加密归档，按需加载后向量化

**关键设计点：**
1. **增量索引**：新记忆定时批量生成 Embedding 并加入 FAISS 索引
2. **混合检索**：关键词匹配 + 向量相似度 + 时间衰减，加权排序
3. **情绪向量**：基于效价-唤醒度模型的情绪推断，辅助记忆关联

### 学习建议

| 维度 | 建议深度 | 说明 |
|------|---------|------|

| sentence-transformers | **熟练** | 模型选择、生成向量、批量处理 |

## 2.4 大语言模型 API 与 Prompt Engineering

### 是什么

**大语言模型 API**：通过 HTTP 接口调用云端大模型（OpenAI GPT、DeepSeek、Anthropic Claude 等），传入 Prompt，返回生成的文本。核心能力包括对话补全、函数调用（Function Calling）、Embedding 等。

**Prompt Engineering（提示工程）**：通过精心设计输入文本，引导大模型输出期望结果的技术。不是简单的"问问题"，而是一门涉及系统提示、少样本学习、思维链、角色设定等技巧的工程实践。

### 在云汐中的应用

**多模型架构：**

云汐不是绑定单一模型，而是通过 `shared/llm_client.py` 提供统一入口，支持多模型切换：

| Provider | 用途 | 特点 |
|----------|------|------|
| **DeepSeek** | 主力模型 | 性价比高，中文能力强，支持 Function Calling |
| **OpenAI** | 备选 / 高质量任务 | 能力最强，但成本高 |
| **Ollama** | 本地部署 | 隐私保护，离线可用，能力相对弱 |
| **Mock** | 降级 / 测试 | 无网络时的兜底模式 |

**M8 算力调度中台更进一步：**
- 管理多个算力源（DeepSeek / Anthropic / Qwen / Zhipu / Moonshot / Gemini）
- 智能路由策略（auto / fastest / cheapest / priority）
- 协议自动转换（OpenAI 格式 ↔ 各厂商格式）
- 故障自动转移

**Prompt Engineering 的应用场景：**

1. **系统提示词（System Prompt）**：定义 Agent 的角色、能力边界、输出格式
   - M1 的各个子 Agent 都有独立的系统提示词
   - M2 的每个技能有技能提示词

2. **记忆注入 Prompt**：把 M5 检索到的相关记忆片段插入 Prompt，让 LLM 基于记忆回答

3. **Function Calling Prompt**：描述工具的名称、参数、用途，让 LLM 决定何时调用工具

4. **思维链（CoT）**：在复杂推理任务中，引导模型一步步思考

5. **少样本学习（Few-shot）**：在 Prompt 中给出几个示例，引导输出格式

## 2.5 MCP 协议（Model Context Protocol）

### 是什么

**MCP（Model Context Protocol）** 是 Anthropic 提出的开放协议，定义了大语言模型与外部工具/数据源交互的标准方式。简单说，MCP 就是 AI 时代的"USB 接口"——让不同的工具和服务能以标准方式接入 LLM。

**核心概念：**
- **MCP Server**：提供工具的服务端，暴露一组 tools（工具）、resources（资源）、prompts（提示模板）
- **MCP Client**：使用工具的客户端（通常是 LLM 应用或 Agent）
- **Transport**：传输层，支持 stdio、HTTP、SSE 等方式

**为什么重要：**
- 统一了工具调用的接口标准，不用每个工具单独集成
- 工具可以独立开发、独立部署，通过 MCP 协议接入
- 生态正在快速形成，未来会有大量 MCP 兼容的服务

### 在云汐中的应用

**M11 MCP 总线服务**是云汐 MCP 生态的核心枢纽：

```
┌─────────────┐
│  M1 调度中心  │  MCP Client
└──────┬──────┘
       │ 统一调用
┌──────▼──────┐
│ M11 MCP 总线 │  注册 / 发现 / 路由 / 审计
└──────┬──────┘
       │
  ┌────┼────┐
  ▼    ▼    ▼
┌───┐ ┌───┐ ┌───┐
│M2 │ │M5 │ │M9 │  MCP Servers
└───┘ └───┘ └───┘
```

**M11 的核心能力：**
1. **服务注册**：MCP 服务启动后向总线注册，提交工具清单
2. **工具聚合**：统一查询所有已注册服务的工具
3. **请求路由**：根据工具名自动路由到对应服务执行
4. **统一鉴权**：API Key 管理，细粒度权限控制
5. **监控审计**：全链路调用日志、成功率、耗时分析

**M2 技能集群的 MCP 桥接：**
- M2 本身也实现了 MCP 兼容接口（`/mcp/v1/tools/list`, `/mcp/v1/tools/call`）
- 18 个内置技能可以作为 MCP 工具被其他模块调用

### 学习建议

| 维度 | 建议深度 | 说明 |
|------|---------|------|
| 协议原理 | **掌握** | MCP 的核心概念、消息格式、交互流程 |
| FastMCP 使用 | **熟练** | Python MCP 框架，快速开发 MCP Server |
| M11 总线设计 | **精通** | 云汐 MCP 总线的架构和扩展方式 |
| 工具开发 | **熟练** | 如何开发一个新的 MCP 工具服务 |
| 协议演进 | **了解** | MCP 与 Function Calling、A2A 的关系 |

**学习路径：**
1. MCP 官方文档 → 理解协议基础（2-3 天）
2. FastMCP 教程 → 动手写一个简单的 MCP Server（3-5 天）
3. 读 M11 代码 → 理解总线设计（1 周）
4. 实践：把一个现有技能改造成 MCP 服务

- **MCP 官方文档**：[modelcontextprotocol.io](https://modelcontextprotocol.io/)

## 2.6 微服务架构与服务治理

### 是什么

**微服务架构**：将一个大型应用拆分为多个小型、独立部署的服务，每个服务有自己的数据库和技术栈，通过 API 通信。好处是独立开发、独立部署、故障隔离，代价是分布式系统的复杂性。

**服务治理**：微服务多了之后，需要解决服务发现、配置管理、负载均衡、熔断降级、链路追踪、监控告警等问题，这一套东西统称服务治理。

### 在云汐中的应用

云汐是一个"轻量级微服务"架构——11 个模块各自独立，但部署在同一台机器上，通过本地 HTTP 调用通信。比单体复杂，但比大规模微服务简单。

**当前架构特点：**

1. **模块即服务**：每个模块是一个独立的 FastAPI 进程，监听不同端口
2. **点对点调用**：模块之间直接通过 HTTP 调用（比如 M1 直接调 M5）
3. **M8 纳管**：M8 作为管理中枢，但不是网关——业务流量不一定经过 M8
4. **M8 标准接口**：所有模块实现统一的健康检查、指标、配置接口

**服务治理的实现方式：**

| 治理能力 | 实现方式 | 所在模块 |
|----------|---------|---------|
| 健康检查 | `/m8/health` 接口，M10 进程监控 | M10 + 各模块 |
| 配置管理 | 环境变量 + .env 文件，M8 配置中心 | M8 + 各模块 |
| 熔断降级 | Circuit Breaker（熔断器）模式 | M1 / M3 |
| 重试机制 | Retry Coordinator（重试协调器） | M1 |
| 限流 | 部分模块实现 | M8 / M11 |
| 审计日志 | 操作审计、调用记录 | M8 / M10 / M11 |
| 监控指标 | `/m8/metrics` 接口 | 各模块 + M10 |

**M1 的调度治理：**
- `circuit_breaker.py`：熔断器，服务失败次数达到阈值后自动熔断
- `retry_coordinator.py`：重试协调器，指数退避重试
- `health_monitor.py`：健康监控，定期检测 Agent 状态
- `budget_manager.py`：预算管理，Token 成本控制

### 学习建议

| 维度 | 建议深度 | 说明 |
|------|---------|------|
| 微服务基础 | **掌握** | 核心概念、拆分原则、优缺点、适用场景 |
| 服务间通信 | **精通** | HTTP API 设计、RESTful、超时、重试、幂等性 |
| 容错模式 | **熟练** | 熔断器、降级、限流、隔离、故障转移 |
| 可观测性 | **掌握** | 日志、指标、链路追踪三件套 |
| 进阶主题 | **了解** | 服务网格、Kubernetes、分布式事务 |

**学习路径：**
1. 理解微服务核心概念和设计模式（1 周）
2. 读云汐 M1 的调度和治理代码（circuit_breaker, retry_coordinator）（1 周）
3. 学习 Docker 和容器化基础（2 周）

- **书籍**：《微服务设计》（Sam Newman）— 微服务领域的经典
- **书籍**：《Release It!》— 生产环境的稳定性模式（熔断器、隔板等）

## 2.9 RAG（检索增强生成）

### 是什么

**RAG（Retrieval-Augmented Generation，检索增强生成）**：让大模型在生成回答之前，先从知识库中检索相关信息，然后把检索到的内容作为上下文一起输入给模型，让模型基于检索到的信息生成回答。

RAG 解决的核心问题：
- **知识截止**：大模型的训练数据有截止日期，不知道最新信息
- **幻觉问题**：模型会编造不存在的事实
- **私有数据**：模型没有你个人的记忆和数据
- **成本效率**：比微调（Fine-tuning）便宜得多，更新也更快

### 在云汐中的应用

RAG 是 M5 潮汐记忆 + LLM 的核心结合方式。云汐的对话系统本质上就是一个**个人化的 RAG 系统**。

**云汐 RAG 流程：**

```
用户提问
  │
  ▼
┌─────────────────────────────┐
│   1. 查询理解                  │
│   - 提取关键词                 │
│   - 生成查询向量               │
│   - 识别时间、情绪等维度        │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│   2. 多路径检索                │
│   - 向量检索（语义相似度）      │  ← L2 层 FAISS
│   - 关键词检索（BM25）         │  ← L1/L2 层
│   - 时间过滤（最近优先）        │
│   - 情绪匹配（相关情绪增强）    │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│   3. 重排序（Rerank）         │
│   - 混合评分：向量 + 关键词 + 时间 │
│   - 去重和多样性控制           │
│   - Top-K 选择（通常 5-20 条） │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│   4. Prompt 构建              │
│   - 系统提示词 + 角色设定       │
│   - 检索到的记忆片段（格式化）  │
│   - 用户原始问题               │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│   5. LLM 生成回答             │
│   - 基于记忆内容回答           │
│   - 标注引用来源（可选）       │
│   - 不知道就说不知道           │
└─────────────────────────────┘
```

**云汐 RAG 的特色设计：**

1. **分层检索**：L0 → L1 → L2 → L3，越深层越精确但速度越慢
2. **三级域隔离**：Agent 私有域 / 协作共享域 / 全局核心域，检索范围受权限控制
3. **情绪维度**：效价-唤醒度模型，情绪相似的记忆更容易被检索到
4. **睡眠巩固**：类似人类睡眠时的记忆整理，定时对记忆做压缩、合并、索引重建

### 学习建议

| 维度 | 建议深度 | 说明 |
|------|---------|------|
| RAG 基础 | **精通** | 原理、流程、核心挑战、评估方法 |
| 向量检索 | **熟练** | Embedding、相似度搜索、索引优化 |
| 高级 RAG | **掌握** | 混合检索、重排序、查询改写、分块策略 |

## 2.10 Agent 与多智能体系统

### 是什么

**Agent（智能体）**：能自主感知环境、做出决策、执行行动的 AI 系统。一个 Agent 通常具备：感知能力、记忆能力、推理能力、行动能力（工具调用）、目标驱动。

**多智能体系统（Multi-Agent System）**：多个 Agent 协作完成任务，每个 Agent 有不同的角色和能力，通过通信和协作解决单个 Agent 难以处理的问题。

### 在云汐中的应用

M1 多 Agent 集群调度是云汐的"大脑中枢"，采用**联邦调度架构**。

**M1 的 8 个内部子 Agent：**

| Agent | 代号 | 职责 | 类比 |
|-------|------|------|------|
| **Arbiter** | 仲裁者 | 任务分发、结果仲裁、质量评估 | 项目经理 |
| **Budget** | 预算官 | 成本管控、预算分配、用量统计 | 财务 |
| **Bus** | 消息总线 | 消息传递、事件订阅、广播通知 | 通信员 |
| **Discovery** | 发现者 | Agent 注册、能力发现、路由决策 | 人事 |
| **Lifecycle** | 生命周期 | 实例管理、状态追踪、优雅启停 | 运维 |
| **Security** | 安全官 | 权限校验、数据脱敏、安全审计 | 安全 |
| **Snapshot** | 快照师 | 状态快照、版本回滚、记忆备份 | 档案员 |
| **Voice** | 语音师 | 语音交互、TTS/ASR、情感语音 | 客服 |

**联邦调度架构：**

```
                    ┌─────────────────────┐
                    │   API Gateway        │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │  Federation Scheduler│
                    │  （联邦调度器）       │
                    └──────────┬──────────┘
          ┌────────────────────┼────────────────────┐
┌─────────▼────────┐ ┌────────▼────────┐ ┌─────────▼────────┐
│  Internal Agents │ │  External Agents │ │  Clone Pool       │
│  （8个子Agent）   │ │  （16+ Adapter） │ │  （分身池）       │
└──────────────────┘ └──────────────────┘ └──────────────────┘
```

**核心调度机制：**

1. **意图分类**：`intent_classifier_v2.py` — 先判断用户要什么
2. **语义路由**：`semantic_router.py` — 根据意图选择最合适的 Agent
3. **任务分发**：`task_dispatcher.py` — 把任务分配给具体 Agent
4. **结果仲裁**：Arbiter Agent 评估多个结果，选择最优
5. **反馈闭环**：`feedback_loop.py` — 收集使用反馈，优化调度策略

**编排器的演进：**
- M1 的 `orchestrator_v2.py` → `orchestrator_v9.py` 经历了 8 个大版本迭代
- 从简单的顺序执行 → 并行调度 → 联邦调度 → 自适应路由
- 反映了云汐 Agent 系统的演进路径

### 学习建议

| 维度 | 建议深度 | 说明 |
|------|---------|------|
| Agent 基础 | **精通** | Agent 的定义、组成、ReAct 框架、规划能力 |
| 多 Agent 协作 | **熟练** | 角色分工、通信机制、协作模式、冲突解决 |
| 调度算法 | **掌握** | 意图识别、路由策略、负载均衡 |
| Agent 框架 | **了解** | LangGraph / AutoGen / CrewAI 等框架的设计思想 |
| 前沿方向 | **了解** | Agentic Workflow、多智能体社会、AGI 路径 |

**学习路径：**
1. 理解 Agent 基本概念和 ReAct 模式（3-5 天）
2. 学习一个 Agent 框架（LangGraph 或 AutoGen）（1-2 周）
3. 读 M1 的调度和 Agent 代码，理解云汐的设计（2 周）
4. 实践：设计并实现一个新的 Agent 角色

# 第五部分：主理人技术决策指南

作为主理人，你最重要的技术能力不是写代码，而是**做决策**。技术选型、架构调整、优先级判断……这些决策的质量，直接决定云汐能走多远。

## 5.1 如何做技术选型

### 选型的核心原则

**1. 够用就好，不要追新**
- 新不等于好，稳定、生态成熟、团队熟悉才重要
- 例子：云汐选择 SQLite 而不是 PostgreSQL，因为当前规模完全够用，维护成本低得多
- 判断标准：未来 6 个月会不会遇到瓶颈？不会就先用简单方案

**2. 与现有技术栈一致**
- 尽量选择团队已经掌握的技术，减少学习成本和维护成本
- 例子：所有模块都用 FastAPI，而不是每个模块用不同框架
- 引入新技术要有充分理由，不能因为"听说很酷"就上

**3. 社区活跃、文档完善**
- 开源项目要看 star 数、issue 响应速度、最近 commit 频率
- 文档质量很重要，出了问题能不能快速找到答案
- 避免选择"作者一个人维护"的关键依赖

**4. 可替换性**
- 关键组件要考虑"如果这个项目死了怎么办"
- 通过封装和抽象，降低对具体技术的耦合
- 例子：`shared/llm_client.py` 封装了多模型切换，不绑定单一提供商

### 常见技术选型场景

| 场景 | 推荐方案 | 备选方案 | 决策依据 |
|------|---------|---------|---------|
| 小型 Web 服务 | FastAPI + SQLite | Flask + PostgreSQL | 规模、团队熟悉度 |
| 向量检索 | FAISS（本地） | ChromaDB / Milvus | 数据量、部署复杂度 |
| 大模型调用 | DeepSeek（主力） | OpenAI / Qwen | 成本、能力、稳定性 |
| 工作流引擎 | 自研 DAG 引擎 | Airflow / Temporal | 复杂度、定制需求 |
| 部署方式 | 裸机进程 | Docker / K8s | 团队能力、运维成本 |
| 消息队列 | （当前不需要） | Redis / RabbitMQ | 异步任务量 |

### 选型流程建议

1. **明确需求**：要解决什么问题？约束条件是什么？
2. **列候选方案**：至少列 2-3 个，包括"不做"这个选项
3. **做原型验证**：每个方案花 1-2 天做个最小原型
4. **评估打分**：从功能、性能、成本、学习曲线、生态等维度打分
5. **做决策并记录**：写下来为什么选这个，未来回头看

## 5.2 如何评估代码质量

作为主理人，你不一定每天写代码，但你需要能判断代码写得好不好。

### 看代码的 7 个维度

**1. 可读性**
- 变量名、函数名是否清晰？
- 注释是否恰当（不多也不少）？
- 函数长度是否合适（一般不超过 50 行）？
- 有没有"魔法数字"（直接写死的常量）？

**2. 模块化**
- 职责是否清晰？一个模块/函数是不是只做一件事？
- 模块之间的依赖关系是否合理？
- 有没有"上帝类"（什么都管的超大类）？

**3. 错误处理**
- 异常有没有被正确捕获和处理？
- 错误信息是否清晰，能定位问题？
- 有没有资源泄漏的风险（文件、数据库连接）？
- 边界条件有没有考虑到？

**4. 可测试性**
- 代码好不好写单元测试？
- 有没有硬编码的依赖导致无法 Mock？
- 测试覆盖率有多少？

**5. 性能意识**
- 有没有明显的性能陷阱（N+1 查询、循环内数据库操作）？
- 关键路径有没有缓存？
- 异步代码有没有阻塞操作？

**6. 安全性**
- 用户输入有没有验证和过滤？
- 敏感数据（密码、API Key）有没有加密存储？
- 有没有 SQL 注入、XSS 等常见漏洞？
- 权限控制是否到位？

**7. 可维护性**
- 配置是不是硬编码的？
- 有没有文档和注释说明设计思路？
- 代码风格是否一致？
- 改一个功能会不会牵扯一堆地方？

### Code Review 清单

每次 Review 代码时，可以对照这个清单快速检查：

- [ ] 功能是否符合需求？
- [ ] 有没有明显的 Bug？
- [ ] 边界条件处理了吗？
- [ ] 错误处理完善吗？
- [ ] 代码可读性如何？
- [ ] 命名是否清晰？
- [ ] 有没有重复代码可以抽取？
- [ ] 测试覆盖了吗？
- [ ] 有没有性能隐患？
- [ ] 有没有安全问题？

## 5.3 如何管理技术债务

技术债务是不可避免的——为了快速上线，你总会写一些"先跑起来再说"的代码。关键是**要有意识地管理债务，而不是被债务压垮**。

### 什么是技术债务

- **代码质量债务**：命名混乱、重复代码、缺少注释
- **架构债务**：模块边界模糊、依赖关系混乱
- **测试债务**：缺少单元测试、集成测试
- **文档债务**：缺少设计文档、API 文档
- **依赖债务**：使用过时的库版本、有安全漏洞的依赖

### 管理策略

**1. 记录债务，不要回避**
- 用 TODO / FIXME 注释标记代码中的问题
- 在项目管理工具中维护"技术债务清单"
- 定期回顾，排期偿还

**2. 分类处理**
- **必须马上还的**：安全漏洞、严重性能问题、影响迭代的架构问题
- **可以慢慢还的**：代码风格、注释完善、测试补充
- **可以不还的**：即将重构的模块、外围工具代码

**3. 还债节奏**
- 每个迭代拿出 10-20% 的时间处理技术债务
- 不要等到债务堆成山才想起来还
- 重构要小步快跑，不要搞"大爆炸式重构"

**4. 预防新债务**
- Code Review 严格把关
- 编码规范和最佳实践文档化
- 新功能必须带测试和文档

### 云汐当前的技术债务（示例）

以下是一些可能需要关注的方向，供你参考：

1. **测试覆盖不足**：部分模块测试较少，核心模块测试较完善
2. **数据库迁移工具缺失**：当前是手动管理表结构，没有 Alembic 之类的迁移工具
3. **配置管理分散**：各模块有自己的 .env 文件，缺少统一的配置中心
4. **部署方式原始**：目前是脚本启动进程，没有容器化和编排
5. **部分模块代码结构待优化**：比如 M6 的核心逻辑还在 server.py 中

## 5.4 如何规划技术演进路线

技术演进不是"想到哪做到哪"，需要有清晰的路线图和优先级。

### 演进的驱动力

技术演进应该由业务需求驱动，而不是技术好奇心驱动。常见的演进驱动力：

1. **规模增长**：用户量、数据量上来了，现有架构扛不住
2. **功能复杂度**：新功能越来越多，代码越来越难维护
3. **团队变化**：团队扩大，需要更规范的协作方式
4. **技术趋势**：重大技术变革，不跟进就落后
5. **成本压力**：现有方案成本太高，需要优化

### 技术路线图的制定方法

**1. 明确目标（1 年愿景）**
- 1 年后系统要达到什么状态？
- 支撑多少用户？多少数据量？
- 核心能力是什么？

**2. 拆解里程碑（季度级）**
- Q1：解决什么问题？交付什么能力？
- Q2：……
- 每个里程碑有明确的验收标准

**3. 排优先级**
- 用 RICE 或 MoSCoW 方法排序
- RICE：Reach（影响面）× Impact（影响程度）× Confidence（信心） / Effort（工作量）
- MoSCoW：Must have / Should have / Could have / Won't have

**4. 定期复盘调整**
- 每个月回顾路线图，根据实际情况调整
- 不要把路线图当宗教，灵活调整但保持方向

### 云汐可能的演进方向（供参考）

**短期（0-6 个月）：夯实基础**
- 完善测试覆盖，建立 CI/CD 流程
- 统一配置管理和模块间通信规范
- M11 MCP 总线成熟化，更多模块接入
- 核心模块的代码质量优化

**中期（6-12 个月）：能力扩展**
- 容器化部署（Docker Compose 起步）
- 数据库迁移到 PostgreSQL（如果数据量上来）
- 更完善的可观测性（日志、指标、链路追踪）
- Agent 能力深化（更智能的调度和协作）

**长期（1-2 年）：架构升级**
- Kubernetes 容器编排（如果需要多机部署）
- 服务网格和更完善的服务治理
- 多模态能力（图像、语音深度整合）
- 端侧 AI 能力增强（本地小模型 + 云端大模型协同）

### 主理人的角色

在技术演进中，主理人的角色不是"画路线图的人"，而是：

1. **方向把控者**：确保技术演进服务于产品目标，不走偏
2. **资源协调者**：争取足够的时间和人力投入技术建设
3. **风险管理者**：识别演进中的风险，避免盲目激进
4. **团队赋能者**：帮助团队成长，提升整体技术能力

> **技术是手段，不是目的。云汐系统的价值不在于用了多少酷炫的技术，而在于它能不能真正帮到用户、帮到你。**
