## 一、现状调研

### 1.1 M10 系统守护模块现状

M10 系统卫士（`M10-system-guard`）核心代码约 **6,345 行**，定位是**系统运维守护（System Guard），而非网络安全防护（Security Protection）**。

**已实现的安全相关能力：**

- Token 鉴权（X-M10-Token / Authorization: Bearer）
- 阈值防护引擎（CPU / 内存 / 温度 / 磁盘四类指标四级拦截）
- 过载自动限流（根据防护级别动态调整并发）
- 重型任务熔断（资源紧张时自动暂停）
- 启动安全检查（任务执行前资源预检）
- 审计日志（内存存储，支持 JSON / CSV 导出）

**缺失的关键安全能力：**

- 无 WAF / Web 应用防火墙
- 无 SQL 注入、XSS、CSRF 等 Web 攻击防护
- 无 IP 黑白名单、地理位置访问控制
- 无 API 级别的速率限制（仅内部任务并发限制）
- 无入侵检测 / 异常行为分析
- 无漏洞扫描 / 依赖安全检查
- 无请求内容过滤与输入校验
- 无数据加密 / TLS 传输加密强制
- 审计日志仅内存存储，无持久化和防篡改

### 1.2 各模块安全能力碎片化

| 安全领域 | 实现模块 | 成熟度 |
|:--------|:--------|:------:|
| API Key 鉴权 | M11 + shared.auth.py | 中等（两套实现） |
| JWT 用户认证 | M8 中控塔 | 中等 |
| RBAC 角色权限 | M8 中控塔 | 较好（五级角色） |
| 速率限制 | M11 + shared | 基础 |
| 资源过载防护 | M10 系统守护 | 较好 |
| 数据加密 | M8（Fernet + bcrypt） | 较好 |
| 审计日志 | M8 + M10 | 基础（文件 / 内存） |
| 紧急制动 | M8 中控塔 | 基础 |
| WAF / IDS | 无 | 完全缺失 |
| IP 访问控制 | 无 | 完全缺失 |
| HTTPS / TLS | 无 | 完全缺失 |

### 1.3 主要安全风险点

| 风险等级 | 风险点 | 影响范围 |
|:--------|:------|:--------|
| **高危** | 默认弱凭据（admin / admin123456） | M8 管理台 |
| **高危** | 鉴权机制碎片化，各自为政 | 全系统 |
| **高危** | 全链路 HTTP 明文传输 | 全系统 |
| **高危** | CORS 全开放（\*） | M10 等多数模块 |
| 中危 | 开发环境默认 Token 硬编码 | M6 / M7 / M10 |
| 中危 | 无输入安全校验，注入风险 | 所有 API 接口 |
| 中危 | 审计日志无防篡改、易丢失 | M8（文件）/ M10（内存） |
| 中危 | 安全代码重复，维护成本高 | M6 / M7 / M10 鉴权中间件 |

## 二、三种方案设计

### 方案 A：轻量级安全网关（M12 Security Gateway）

**定位**：独立的 API 网关 + 轻量级 WAF，作为云汐系统的统一入口。

**架构示意：**

```
客户端 → M12 安全网关 → 分发到 M1-M11 各模块
                  ↑
           安全策略配置
```

**核心功能：**

1. **统一鉴权入口**：集中管理所有模块的认证
2. **速率限制**：IP 级别 + 用户级别多层限流
3. **IP 黑白名单**：手动配置 + 自动封禁
4. **请求过滤**：SQL 注入检测、XSS 过滤、异常请求头检查
5. **基础 WAF 规则**：OWASP Top 10 基础规则集
6. **审计日志**：全流量请求日志
7. **CORS 安全策略统一配置**

**技术与规模：**

| 项目 | 说明 |
|------|------|
| 技术栈 | FastAPI + Starlette 中间件 + 自研规则引擎 |
| 部署方式 | 反向代理模式，部署在所有模块之前 |
| 开发工作量 | 约 3 - 4 周（核心功能） |
| 预估代码量 | 约 3,000 - 4,000 行 |

**优点：**

- 轻量独立，不侵入现有代码
- 部署简单，只需调整入口流量
- 快速见效，基础安全能力一步到位
- 单点管控，安全策略统一配置

**缺点：**

- WAF 规则相对简单，深度防护不足
- 无入侵检测、漏洞扫描等高级功能
- 仅防护入口流量，内部调用无防护
- 无法做代码级安全检查

---

### 方案 B：全功能安全中心（M12 Security Center）

**定位**：综合安全管理平台，云汐系统的安全大脑。

**架构示意：**

```
┌──────────────────────────────────────────┐
│         M12 安全中心（中心化）            │
│  ┌─────────┬─────────┬─────────┬───────┐ │
│  │ 鉴权中心 │访问控制 │审计中心 │入侵检测│ │
│  ├─────────┼─────────┼─────────┼───────┤ │
│  │漏洞扫描 │策略管理 │安全仪表板│告警中心│ │
│  └─────────┴─────────┴─────────┴───────┘ │
└──────┬───────────┬───────────┬───────────┘
       │ SDK       │ SDK       │ SDK
    ┌──▼──┐     ┌───▼──┐    ┌──▼──┐
    │ M1  │     │ M8   │    │ M11 │  ...
    └─────┘     └──────┘    └─────┘
```

**核心功能：**

1. **统一鉴权中心**：OAuth2.0 / JWT / API Key、SSO 单点登录、MFA 多因素认证
2. **访问控制引擎**：RBAC + ABAC 混合、细粒度 API 级权限、动态策略
3. **审计与合规**：全量操作审计（防篡改）、合规报表、日志归档
4. **入侵检测系统（IDS）**：规则检测 + 异常行为分析、实时告警 + 自动封禁
5. **漏洞扫描**：依赖库扫描、API 安全扫描、配置基线检查
6. **安全策略管理**：可视化编辑器、版本管理、灰度发布
7. **安全仪表板**：实时安全态势、威胁分析、安全评分

**技术与规模：**

| 项目 | 说明 |
|------|------|
| 技术栈 | 独立 FastAPI 服务 + 各模块 Python SDK + Redis + SQLite / PostgreSQL |
| 部署方式 | 中心化服务 + 各模块轻量 Agent / SDK |
| 开发工作量 | 约 10 - 12 周 |
| 预估代码量 | 约 15,000 - 20,000 行 |

**优点：**

- 功能全面，覆盖安全全生命周期
- 统一管理，安全态势一目了然
- 可扩展，支持持续迭代增强
- 合规友好，审计与报表完善

**缺点：**

- 开发量大，周期长
- 各模块需接入 SDK，有侵入性
- 部署复杂度高，需考虑高可用
- 运维成本高，需要专人维护

---

### 方案 C：嵌入式安全框架（内嵌式）

**定位**：shared 库安全能力增强 + 各模块逐步接入，**不单独编号为 M12**。

**架构示意：**

```
┌──────────────────────────────────────┐
│         shared 库（增强版）            │
│  ┌──────────┬──────────┬────────────┐ │
│  │安全中间件 │ 安全工具  │ FastAPI Dep│ │
│  └──────────┴──────────┴────────────┘ │
└──────┬──────────┬──────────┬──────────┘
       │ 引入     │ 引入     │ 引入
    ┌──▼──┐   ┌───▼──┐   ┌──▼──┐
    │ M6  │   │ M7   │   │ M10 │  ... 各模块按需接入
    └─────┘   └──────┘   └─────┘
```

**核心功能**（增强 `shared.auth.py`）：

1. **统一鉴权中间件**：替换 M6 / M7 / M10 重复代码
2. **安全工具函数集**：输入清洗、XSS 过滤、CSRF Token、敏感数据脱敏、密码强度校验
3. **速率限制中间件**：IP 限流 + 用户限流、多种算法、Redis 分布式支持
4. **审计日志 SDK**：统一格式、异步上报、装饰器式接入
5. **IP 访问控制**：黑白名单中间件
6. **FastAPI Depends 生态**：`require_auth` / `require_role` / `require_permission` / `rate_limit`

**技术与规模：**

| 项目 | 说明 |
|------|------|
| 技术栈 | Python 标准库 + FastAPI Depends + 可选 Redis |
| 部署方式 | 各模块内嵌，升级 shared 库版本即可 |
| 开发工作量 | 约 4 - 5 周（shared 库增强） |
| 预估代码量 | 约 3,000 - 5,000 行（shared 库增量） |

**优点：**

- 无额外服务，架构不变
- 接入灵活，各模块按需逐步接入
- 开发成本中等
- 代码复用，消除重复实现
- 低耦合，不影响各模块独立性

**缺点：**

- 分散管理，无统一控制台
- 安全策略配置分散在各模块
- 缺少集中式安全监控
- 无入侵检测、漏洞扫描等高级功能

## 四、推荐建议

### 4.1 推荐路线：方案 A + 方案 C 组合，分阶段演进

#### 第一阶段（1 - 2 个月）：方案 C 先行 + 方案 A 核心功能

**1. 优先落地 shared 库安全增强（方案 C 核心部分）**

- 统一鉴权中间件，替换 M6 / M7 / M10 重复代码
- 补充输入清洗、XSS 过滤等基础安全工具
- 统一审计日志格式与 SDK
- 修复默认弱凭据、CORS 全开放等高风险问题

**2. 同步建设轻量级安全网关（方案 A 核心功能）**

- 作为 M8 管理台的前置网关
- 实现统一鉴权、IP 限流、基础 WAF 规则
- 提供安全日志聚合入口

#### 第二阶段（3 - 6 个月）：方案 A 深化，评估方案 B 必要性

- 完善 WAF 规则库，覆盖 OWASP Top 10
- 增强入侵检测规则
- 建设安全仪表板原型
- 根据业务发展和安全需求，评估是否需要升级到方案 B

### 4.2 选择理由

1. **风险优先**：先通过方案 C 快速修复当前已知高危漏洞
2. **成本可控**：组合方案开发量约 6 - 8 周，远低于方案 B
3. **渐进增强**：不搞大跃进，先基础后高级，根据实际需求迭代
4. **架构低侵入**：方案 C 是库级增强，方案 A 是旁路网关，都不改变现有模块架构
5. **回退成本低**：如不满足需求，可平滑升级到方案 B，前期投入不浪费

### 4.3 M10 与 M12 的定位边界

| 模块 | 定位 | 核心职责 |
|------|------|---------|
| **M10 系统守护** | 系统稳定性与资源管理（运维守护） | 阈值防护、过载限流、任务熔断、资源预检 |
| **M12 安全防护**（方案 A/B） | 网络安全与信息安全（安全防护） | WAF、鉴权、IDS、访问控制、安全审计 |
| 方案 C（嵌入式） | 库级安全能力增强 | 统一鉴权中间件、安全工具集、审计 SDK |

> M10 与 M12 为互补关系，而非替代关系。若选择方案 C（嵌入式），则暂不设立 M12 独立模块编号。
