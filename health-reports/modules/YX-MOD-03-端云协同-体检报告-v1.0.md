## 二、功能实现情况

### 核心功能（已实现）

1. **双执行器架构（本地 + 云端）**
   - 本地执行器：端侧设备本地推理，保障隐私与低延迟
   - 云端执行器：云端大模型调用，提供更强的计算能力
   - 执行器统一抽象接口，上层业务无感知切换

2. **智能路由决策**
   - 基于任务类型的路由：简单任务本地执行，复杂任务上云
   - 基于数据敏感度的路由：隐私数据本地处理，脱敏数据可上云
   - 基于网络状态的路由：网络良好时优先云端，网络差时优先本地
   - 基于设备性能的路由：GPU 显存充足时本地执行，否则上云

3. **端云数据双向同步**
   - 增量同步：仅同步变更数据，减少带宽消耗
   - 全量同步：支持全量数据对账与恢复
   - 同步队列：异步队列保障数据可靠传输
   - 断点续传：网络中断后可从中断点继续同步

4. **冲突检测与解决**
   - 版本向量（Version Vector）冲突检测
   - 基于时间戳的自动合并策略
   - 冲突记录留存，支持人工介入裁决

5. **离线模式支持**
   - 断网时自动切换至纯本地模式
   - 离线操作队列化，联网后自动同步
   - 离线状态监控与通知

6. **GPU 显存监控**
   - 实时监控本地 GPU 显存使用情况
   - 显存不足时自动拒绝本地执行请求，路由至云端
   - 显存阈值可配置

7. **熔断器 / 限流器**
   - 云端接口熔断保护，防止级联故障
   - 本地执行限流，防止设备过载
   - 自适应熔断恢复机制

### 重要功能（已实现）

1. **设备管理（注册 / 状态 / 心跳）**
2. **配置动态更新**
3. **同步状态查询与监控**

5. **异步文件处理（aiofiles）**

### 一般功能（已实现）

## 三、系统定位与作用

端云协同（M3）是整个云汐系统的**混合调度层**，负责在端侧隐私保护与云端算力优势之间取得平衡。

### 在系统中的角色

- **混合调度器**：根据任务特征、数据敏感度、网络状态等多维度因素，智能决策任务在本地还是云端执行
- **数据同步枢纽**：管理端侧与云端之间的数据双向同步，确保数据一致性
- **离线保障层**：在网络不可用情况下，保障端侧系统的基本可用能力
- **资源协调者**：监控端侧资源（GPU、内存、CPU），合理分配工作负载

### 与其他模块的关系

```
M1 调度中心 ──→ M3 端云协同
                    │
                    ├──→ 本地执行器（端侧 AI 模型）
                    ├──→ 云端执行器（云端大模型 / M1 远端）
                    ├──→ 数据同步 → M5 潮汐记忆（记忆数据同步）
                    └──→ 设备管理 → 端侧设备群
```

- **上游**：M1 调度中心，接收任务调度请求
- **下游**：本地 AI 模型（端侧）、云端服务（M1 远端/云服务）、M5 潮汐记忆（数据同步）
- **横向**：管理多个端侧设备，形成端云一体的计算网络

```

├── devices/                   # 设备管理

│   └── mock_device.py         # Mock 设备
```

> 注：8 个核心组件均有对应的 mock/fallback 实现，Mock 模式覆盖率高。

### 核心组件

| 组件 | 职责 | 技术实现 |
|------|------|----------|
| LocalExecutor | 本地执行器 | 本地 AI 模型推理 + GPU 加速 |
| CloudExecutor | 云端执行器 | HTTP/gRPC 调用云端服务 |
| Router | 智能路由 | 多因子决策引擎（任务类型 + 数据敏感度 + 网络 + 资源） |
| SyncManager | 数据同步管理 | 增量同步 + 队列 + 断点续传 |
| ConflictResolver | 冲突解决器 | 版本向量 + 时间戳策略 |
| OfflineQueue | 离线队列 | SQLite 持久化队列 + 重试机制 |
| DeviceManager | 设备管理 | 设备注册 + 心跳 + 状态管理 |
| GPUMonitor | GPU 监控 | psutil + GPU 驱动 API |
| CircuitBreaker | 熔断器 | 滑动窗口 + 半开探测 |
| RateLimiter | 限流器 | 令牌桶算法 |

### 数据流

```
任务请求
    │
    ▼
智能路由决策
    │
    ├─── 数据敏感？──── 是 ──→ 本地执行器
    │        │
    │        否
    │        │
    ├─── 网络可用？─── 否 ──→ 本地执行器 + 离线队列
    │        │
    │        是
    │        │
    ├─── GPU 充足？─── 是 ──→ 本地执行器（低延迟）
    │        │
    │        否
    │        │
    └──────────────────────→ 云端执行器（强算力）
             │
             ▼
执行结果返回
    │
    ▼

    │
    ▼
冲突检测 → 自动解决 / 人工介入
```

## 六、存在的问题

### 严重问题

1. **大量 Mock 模式，真实能力存疑**
   - **描述**：8 个核心组件（执行器、同步、路由、设备、GPU 等）均有对应的 mock/fallback 实现
   - **影响**：难以判断系统真实能力，测试时可能因 Mock 掩盖真实问题
   - **风险**：如果 Mock 代码混入生产路径，可能导致功能不可用
   - **疑点**：云端执行器可能仅为 Mock 实现，缺乏真实云端对接能力

2. **API 路径不统一，三套前缀并存**
   - **描述**：同时存在 `/api/v3/`、`/api/v1/`、`/m8/` 三套 API 路径前缀
   - **影响**：接口混乱，调用方接入困难，维护成本高
   - **原因推测**：历史版本迭代与模块迁移遗留，未及时清理

### 重要问题

3. **版本号严重不一致**
   - **描述**：README 标注 v3.0，实际代码版本 v2.1.2，出现版本号"倒退"
   - **影响**：版本管理混乱，无法通过版本号判断功能演进
   - **疑点**：可能是 README 更新不及时，或模块经历过重构导致版本号重置

4. **云端执行器可能是空壳**
   - **描述**：云端执行器的实现深度不明确，可能仅定义了接口但缺乏真实对接
   - **影响**：端云协同名不副实，实际只有本地执行器在工作
   - **验证建议**：检查云端执行器的实际调用链路与错误处理

5. **冲突解决策略简单**
   - **描述**：当前冲突解决主要依赖时间戳策略，缺乏更智能的合并机制
   - **影响**：复杂数据冲突场景下可能丢失用户数据或产生不一致
   - **场景**：端云同时修改同一条记忆数据时，简单的时间戳策略可能丢失一方修改

### 一般问题

6. **Mock 代码与生产代码混放**
   - **描述**：Mock 实现与真实实现在同一目录下，目录结构不够清晰
   - **影响**：代码阅读时容易混淆，新成员可能误将 Mock 当真实实现

7. **设备管理功能偏弱**
   - **描述**：设备管理仅支持基础的注册和心跳，缺乏设备分组、标签、远程配置等高级功能
   - **影响**：大规模设备部署时管理能力不足

8. **监控指标不够完善**
   - **描述**：缺少同步延迟、端云切换次数、本地/云端执行占比等关键业务指标
   - **影响**：难以评估端云协同的实际效果与资源利用率

## 七、优化建议

### 高优先级

1. **验证云端执行器真实能力**
   - **建议**：对云端执行器进行深度审查，确认其真实对接状态
   - **步骤**：
     1. 检查 CloudExecutor 的实现代码，确认是否有真实的云端调用
     2. 进行端到端测试，验证云端执行路径
     3. 如为空壳，尽快补齐实现；如已实现，清理 Mock 代码
   - **预期收益**：明确系统真实能力，避免虚假功能

2. **API 路径统一与清理**
   - **建议**：以 v3 为主版本，逐步淘汰 v1 和 m8 前缀
   - **步骤**：
     1. 梳理所有调用方，确认 v1 和 m8 接口的使用情况
     2. 提供 v3 版本的等价接口，引导迁移
     3. 设置废弃期限，到期后删除旧前缀
   - **预期收益**：接口清晰统一，维护成本降低

3. **Mock 代码整理与隔离**
   - **建议**：将所有 Mock 实现统一移入 `tests/mocks/` 或 `_mocks/` 目录
   - **方案**：生产代码目录只保留真实实现，Mock 仅用于测试
   - **预期收益**：代码结构清晰，减少混淆

### 中优先级

4. **统一版本号体系**
   - **建议**：建立单一版本号来源，README 与代码版本保持一致
   - **原则**：遵循语义化版本（SemVer），禁止版本号"倒退"
   - **预期收益**：版本管理规范化

5. **增强冲突解决策略**
   - **建议**：引入 CRDT（无冲突复制数据类型）或更精细的字段级合并
   - **方案**：
     - 简单数据：保留时间戳策略作为兜底
     - 复杂数据：字段级合并 + 操作转换（OT）
     - 无法自动解决：推送人工裁决界面
   - **预期收益**：降低数据冲突导致的用户数据丢失风险

6. **完善监控指标**
   - **建议**：增加端云协同专属监控指标
   - **指标清单**：
     - 本地 / 云端执行次数与占比
     - 端云切换原因分布
     - 同步延迟 P50 / P95 / P99
     - 冲突发生率与解决率
     - 离线时长分布
   - **预期收益**：可量化评估端云协同效果

### 低优先级

7. **增强设备管理能力**
   - **建议**：增加设备分组、标签管理、远程配置推送、批量升级等功能
   - **预期收益**：提升大规模部署时的运维效率

8. **引入端云协同评测体系**
   - **建议**：建立端云协同效果评估框架，量化延迟、成本、隐私保护三者的平衡
   - **预期收益**：为路由策略优化提供数据支撑
