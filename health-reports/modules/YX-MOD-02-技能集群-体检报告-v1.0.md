## 二、功能实现情况

### 核心功能（已实现）

1. **可插拔技能框架**
   - 基于插件化架构设计，技能独立封装，支持动态加载与卸载
   - 技能元数据标准化定义（名称、描述、参数、返回值、标签）
   - 支持技能热加载（watchdog 监听文件变化自动重载）

2. **18 个内置技能（8 大类）**
   - 信息检索类：网页搜索、文档解析、知识库查询
   - 文件处理类：PDF 处理、表格操作、图像识别
   - 数据分析类：统计分析、图表生成、数据清洗
   - 工具类：计算器、单位转换、时间处理
   - 编程类：代码执行、代码审查、API 调用
   - 创作类：文案生成、内容润色
   - 办公类：邮件发送、日程管理
   - 系统类：命令执行、文件管理

3. **智能路由系统**
   - 语义路由：基于技能描述与用户输入的语义相似度匹配
   - 贝叶斯路由：基于历史调用数据的贝叶斯概率推理
   - 多臂老虎机（MAB）：探索-利用平衡的在线学习路由
   - 多级路由策略组合，支持 fallback 链路

4. **技能流水线（Pipeline）**
   - 支持多技能串行/并行组合执行
   - 技能间数据传递与格式转换
   - 流水线编排配置，支持可视化定义

5. **MCP 协议桥接 + A2A 总线**
   - 完整实现 MCP（Model Context Protocol）协议
   - A2A（Agent-to-Agent）通信总线
   - 支持外部 MCP 工具接入与技能对外暴露

6. **沙箱隔离**
   - 技能执行环境隔离，防止恶意代码影响主系统
   - 资源限制（CPU、内存、执行时间）
   - 权限分级控制

## 三、系统定位与作用

技能集群（M2）是整个云汐系统的**能力扩展层**，为 M1 调度中心和 M7 等上游模块提供统一的技能工具集。

### 在系统中的角色

- **能力工具箱**：封装各类通用与专用技能，供上层模块按需调用
- **技能运行时**：提供技能加载、执行、隔离、监控的完整运行环境
- **协议转换层**：通过 MCP 协议桥接外部工具生态，扩展系统能力边界
- **路由决策层**：根据用户意图智能推荐和匹配最合适的技能组合

### 与其他模块的关系

```
M1 调度中心 ──→ M2 技能集群 ──→ M5 潮汐记忆（技能状态持久化）
                    │
                    ├──→ 内置技能（18 个，8 大类）
                    ├──→ MCP 外部工具
                    └──→ A2A 总线上的其他 Agent
```

- **上游**：M1 调度中心、M7 等需要技能能力的模块
- **下游**：M5 潮汐记忆（技能调用记录、状态持久化）
- **横向**：通过 MCP/A2A 协议与外部工具和 Agent 生态互联互通

## 四、架构设计

### 目录结构现状

```
M2-skills-cluster/
├── start_server.py            # 服务入口
├── api_v2/                    # API v2 路由层（第一套）

│   ├── routers/
│   └── schemas/
├── skills/                    # 技能实现
│   ├── base.py                # 技能基类
│   ├── registry.py            # 技能注册表
│   ├── loader.py              # 技能加载器
│   ├── info_retrieval/        # 信息检索类
│   ├── file_processing/       # 文件处理类
│   ├── data_analysis/         # 数据分析类
│   ├── tools/                 # 工具类
│   ├── programming/           # 编程类
│   ├── creation/              # 创作类
│   ├── office/                # 办公类
│   └── system/                # 系统类
├── routing/                   # 路由引擎
│   ├── semantic_router.py     # 语义路由
│   ├── bayesian_router.py     # 贝叶斯路由
│   └── mab_router.py          # 多臂老虎机路由
├── pipeline/                  # 技能流水线
├── mcp/                       # MCP 协议实现
│   ├── bridge.py
│   └── tools/
├── a2a/                       # A2A 总线
├── sandbox/                   # 沙箱隔离
├── config/                    # 配置管理
├── utils/                     # 工具函数
└── tests/                     # 测试文件
```

### 核心组件

| 组件 | 职责 | 技术实现 |
|------|------|----------|
| SkillRegistry | 技能注册与发现 | 内存注册表 + 元数据索引 |
| SkillLoader | 技能动态加载 | importlib + watchdog 热重载 |
| SkillBase | 技能基类 | 抽象基类 + 模板方法模式 |
| RouterEngine | 智能路由引擎 | 语义路由 + 贝叶斯 + 多臂老虎机 三层路由 |
| PipelineEngine | 技能流水线 | DAG 编排 + 异步执行 |
| MCPBridge | MCP 协议桥接 | MCP 协议编解码 + 工具映射 |
| A2ABus | A2A 通信总线 | 消息队列 + 发布订阅模式 |
| SandboxManager | 沙箱管理器 | 进程隔离 + 资源限制 |

### 数据流

```
技能调用请求
    │
    ▼
API 接入层（api_v2 / http_api）
    │
    ▼
路由引擎（语义匹配 → 贝叶斯排序 → MAB 最终选择）
    │
    ├──→ 单技能直接执行
    ├──→ 多技能流水线编排执行
    ├──→ MCP 桥接调用外部工具
    └──→ A2A 总线转发至其他 Agent
    │
    ▼
沙箱隔离执行环境
    │
    ▼
结果格式化 / 后处理
    │
    ▼
返回调用方 + 异步写入 M5 记忆
```

## 六、存在的问题

### 严重问题

1. **API 层重复实现**
   - **描述**：api_v2 和 http_api 两套 API 层并存，功能高度重叠
   - **影响**：维护成本翻倍，接口行为可能不一致，调用方接入混乱
   - **风险**：两套 API 长期并行，功能偏差会越来越大

2. **技能加载数量与文档不符**
   - **描述**：README 宣称 18 个内置技能，但启动时实际仅加载 10 个
   - **影响**：对外承诺与实际能力不符，影响系统可信度
   - **原因推测**：部分技能尚未完成或存在兼容性问题，被临时禁用

### 重要问题

3. **技能路由算法过多且层次不清**
   - **描述**：同时实现了语义路由、贝叶斯路由、多臂老虎机三种路由算法，三层叠加
   - **影响**：路由决策链路复杂，可解释性差，性能开销大
   - **问题**：三种算法的适用场景、优先级、组合策略不明确

4. **缺乏持久化机制**
   - **描述**：技能注册表为纯内存结构，无独立数据库，依赖 M5 记忆
   - **影响**：服务重启后技能状态丢失，路由算法的历史数据无法持久化
   - **风险**：贝叶斯路由和 MAB 路由的学习效果无法累积

5. **Windows 兼容 hack 泛滥**
   - **描述**：代码中存在大量 Windows 平台兼容的特殊处理
   - **影响**：代码可读性差，分支逻辑复杂，容易引入平台相关 bug
   - **典型场景**：文件路径处理、进程管理、沙箱隔离

### 一般问题

6. **版本号不一致**
   - **描述**：README 标注 v2.0，实际代码版本 v3.10.2，差距较大
   - **影响**：版本追溯困难，文档与代码不同步

7. **依赖体积过大**
   - **描述**：pandas、matplotlib、PyMuPDF 等重型依赖导致安装包体积大
   - **影响**：部署慢、镜像大、冷启动时间长
   - **建议**：按技能类别拆分可选依赖

8. **沙箱隔离强度不足**
   - **描述**：当前沙箱实现相对简单，主要依赖进程级隔离
   - **影响**：面对恶意技能代码的防护能力有限
   - **风险**：存在沙箱逃逸、资源滥用的安全隐患

## 七、优化建议

### 高优先级

1. **API 层合并统一**
   - **建议**：保留一套 API 层（建议保留 api_v2，命名更规范），移除 http_api
   - **步骤**：
     1. 比对两套 API 的功能差异，合并至 api_v2
     2. 将 http_api 的调用方全部迁移至 api_v2
     3. 标记 http_api 为废弃，下个版本移除
   - **预期收益**：代码量减少约 25%，维护成本降低，接口一致性提升

2. **补齐缺失技能或修正文档**
   - **建议**：
     - 方案 A：补齐剩余 8 个技能的实现，达到 README 宣称的 18 个
     - 方案 B：如技能暂不可用，修正 README 文档，如实反映当前能力
   - **原则**：文档与实际能力必须一致
   - **预期收益**：提升系统可信度，避免调用方预期偏差

3. **路由算法分层与简化**
   - **建议**：
     - 第一层（快速筛选）：语义路由，快速召回 Top-K 候选技能
     - 第二层（精排选择）：保留贝叶斯或 MAB 其中一种作为精排
     - 第三层（兜底）：规则引擎 fallback
   - **理由**：三层算法叠加收益递减，但复杂度和开销显著增加
   - **预期收益**：路由性能提升 30%-50%，可解释性增强

### 中优先级

4. **引入轻量持久化**
   - **建议**：增加 SQLite 本地持久化，存储技能调用日志、路由历史数据
   - **范围**：不替代 M5 的长期记忆，仅保存 M2 内部状态与路由学习数据
   - **预期收益**：路由算法效果可持续优化，服务重启不丢失学习成果

5. **Windows 兼容层抽象**
   - **建议**：将 Windows 兼容逻辑统一收敛到 `utils/platform.py` 中
   - **方案**：提供统一的平台抽象接口，上层业务代码不感知平台差异
   - **预期收益**：业务代码更简洁，平台相关问题集中修复

6. **依赖分组优化**
   - **建议**：按技能类别拆分 extras_require，支持按需安装
   - **示例**：
     ```
     pip install skills-cluster[pdf]      # 仅安装 PDF 处理相关
     pip install skills-cluster[data]     # 仅安装数据分析相关
     pip install skills-cluster[full]     # 全量安装
     ```
   - **预期收益**：轻量化部署，减少不必要的依赖

### 低优先级

7. **强化沙箱安全**
   - **建议**：评估引入 Docker 容器或 WASM 沙箱的可行性
   - **方案**：对高风险技能（如代码执行类）使用强隔离沙箱
   - **预期收益**：提升系统安全性

8. **技能市场 / 插件中心**
   - **建议**：建立技能注册中心，支持第三方技能提交与审核
   - **预期收益**：扩展技能生态，降低技能接入门槛
