## 2. 功能实现情况

### 2.1 核心功能

- **全局配置管理（YunxiConfig）**：单例模式的全局配置中心，管理 10 个模块的端口、Token、URL 等配置信息，是各模块获取配置的统一入口。
- **统一日志框架**：封装结构化日志能力，为各模块提供一致的日志输出格式和配置。
- **LLM 客户端**：多后端切换的大语言模型调用客户端，屏蔽不同 LLM 服务商的 API 差异。
- **模块注册中心（ModuleRegistry）**：维护系统内所有模块的注册信息，提供模块发现和元数据查询。
- **模块调用客户端（ModuleClient）**：封装模块间 HTTP 调用，内置重试、超时、鉴权等横切逻辑。
- **进程管理器（ProcessManager）**：管理子进程的启动、停止、监控，支持模块进程的生命周期管理。

### 2.2 重要功能

- **数据库基础封装**：db.py / db_config.py 提供 SQLAlchemy 数据库连接的基础封装和配置，供各模块复用。
- **语音引擎（VAD）**：基于 silero-vad 的语音活动检测能力，为语音相关模块提供 VAD 支持。
- **异步文件操作**：aiofiles 封装，提供异步文件读写能力。
- **加密工具**：cryptography 相关工具封装，提供加解密基础能力。

### 2.3 一般功能

- **版本管理**：version.py 提供系统版本信息。
- **HTTP 工具**：基于 httpx 的 HTTP 客户端封装。
- **系统工具**：psutil 等系统监控工具的封装。

## 3. 系统定位与作用

shared 公共基础库是云汐系统的**基础依赖层**，承担以下角色：

- **横切关注点统一实现**：将配置、日志、鉴权、重试、监控等横切关注点集中实现，避免各模块重复造轮子。
- **基础设施库**：提供数据库、HTTP 客户端、进程管理、加密等基础设施能力，各模块在此基础上构建业务功能。
- **系统一致性保障**：通过统一的配置、日志、模块调用等机制，保障各模块在行为规范上的一致性。
- **模块间协作桥梁**：模块注册中心和模块调用客户端是模块间发现和通信的基础设施。
- **AI 能力底座**：LLM 客户端和语音引擎为各模块提供 AI 基础能力。

**与其他模块的关系**：
- 被所有模块（M1-M11）依赖和导入
- 是系统中最底层的公共依赖
- 各模块的数据库连接、模块间通信、配置管理等都经由 shared 提供
- 不依赖任何业务模块，保持单向依赖关系

## 4. 架构设计

### 4.1 技术栈

| 类别 | 技术选型 |

### 4.2 核心组件

- **YunxiConfig / get_config()**：全局配置单例，管理 10 模块的端口、Token、URL。
- **ModuleRegistry / get_registry()**：模块注册中心单例，硬编码 9 个模块的元信息。
- **ProcessManager / get_process_manager()**：进程管理器单例，管理模块进程生命周期。
- **LLMClient**：多后端 LLM 调用客户端，支持不同服务商切换。
- **ModuleClient**：模块间调用客户端，内置重试、超时、鉴权。
- **Logger**：统一日志配置和输出。
- **VoiceEngine**：语音 VAD 引擎。
- **db.py / db_config.py**：数据库连接基础封装。

### 4.3 数据流

```
各业务模块 → import shared.*
              ↓
    ┌─────────┼─────────┐
    ↓         ↓         ↓
  配置中心  模块注册  日志框架
    ↓         ↓         ↓
  配置读取  模块发现  统一日志输出
              ↓
         模块调用客户端
              ↓

              ↓
         目标模块 API
```

## 5. 代码质量评估

### 5.1 代码规范性

- **规模适中**：19 个文件、约 4,200 行，作为基础库规模合理。
- **命名规范**：单例模式的 get_*() 命名风格一致，整体符合 Python 惯例。
- **导出不完整**：`__init__.py` 导出不完整，部分功能需要直接从子模块导入，使用体验不一致。

### 5.2 可维护性

- **代码重复**：根目录、core/、clients/ 三套实现存在代码重复，增加了维护成本和出错概率。
- **版本管理分散**：版本信息分散在各处，没有统一的版本管理机制。
- **硬编码问题**：ModuleRegistry 硬编码了 9 个模块的信息，新增模块需要修改 shared 库代码。

### 5.3 测试覆盖率

- 未发现针对 shared 库的专门测试。
- **评估**：作为基础库，缺少单元测试是较大的风险，因为所有上层模块都依赖这些基础能力。

### 5.4 依赖管理

- 有独立的 requirements.txt。
- 依赖项较多（FastAPI、Pydantic、SQLAlchemy、structlog、psutil、cryptography、httpx、aiofiles、silero-vad、numpy 等），作为基础库依赖较重。
- 缺少依赖版本锁定。

## 6. 存在的问题

### 6.1 严重问题

**1. 代码重复（根目录/core/clients 三套实现）**
- 描述：shared 库内存在根目录、core/、clients/ 三套功能类似的实现，代码重复严重。同一份逻辑可能在多处存在不同版本，导致维护混乱、bug 修复不彻底、行为不一致。
- 影响：维护成本成倍增加，各模块可能因为 import 路径不同而使用不同实现，引发难以排查的问题。
- 优先级：高

### 6.2 重要问题

**2. __init__.py 导出不完整**
- 描述：shared 包的 `__init__.py` 没有完整导出所有公共 API，部分类和函数需要用户知道具体子模块路径才能导入。
- 影响：使用体验差，导入方式不统一，增加使用者的认知成本。
- 优先级：中

**3. 版本管理分散**
- 描述：版本信息分散在 version.py 和其他文件中，各模块也有各自的版本号，缺少统一的版本管理策略。
- 影响：发布时版本号同步困难，容易出现各模块版本不一致的问题。
- 优先级：中

**4. ModuleRegistry 硬编码 9 个模块**
- 描述：模块注册中心硬编码了 9 个模块的信息，新增或调整模块需要修改 shared 库代码并重新发布。
- 影响：扩展性差，违背了注册中心应有的动态性原则。
- 优先级：中

### 6.3 一般问题

**5. 缺少类型 stubs 和完整文档**
- 描述：作为公共基础库，缺少类型存根文件（.pyi）和完整的 API 文档，使用者需要阅读源码才能了解接口。
- 影响：开发效率低，IDE 智能提示不完整，容易误用 API。
- 优先级：低

**6. 缺少单元测试**
- 描述：作为所有模块的基础依赖，缺少单元测试保障。基础库的 bug 会影响所有上层模块。
- 影响：基础能力的正确性无法验证，回归风险高。
- 优先级：低

**7. 依赖较重**
- 描述：shared 库引入了 silero-vad、numpy 等较重的依赖，所有依赖 shared 的模块都会被动引入这些依赖，即使模块本身不需要语音功能。
- 影响：安装包体积大，依赖安装时间长，轻量模块被迫引入重型依赖。
- 优先级：低

## 7. 优化建议

### 7.1 高优先级

1. **消除代码重复，统一实现**：梳理根目录、core/、clients/ 三套代码的功能重叠，制定合并计划。
   - 第一步：对比三套实现，绘制功能映射表
   - 第二步：确定"唯一真相来源"，将重复功能统一到一处实现
   - 第三步：在废弃的路径上提供兼容别名（指向新实现），避免破坏现有调用
   - 第四步：在后续版本中逐步移除兼容别名

### 7.2 中优先级

2. **完善 `__init__.py` 导出**：在 shared 包的 `__init__.py` 中统一导出所有公共 API，使用户只需 `from shared import YunxiConfig, ModuleRegistry, ProcessManager, ...` 即可使用所有核心功能。同时保留子模块导入的兼容性。

3. **统一版本管理机制**：建立全局版本管理策略，以 shared.version 中的版本号为系统主版本，各模块从 shared 导入版本号，避免硬编码和不一致。考虑使用语义化版本号（SemVer）。

4. **ModuleRegistry 动态化改造**：
   - 短期：将硬编码的模块列表改为从配置文件读取，新增模块只需修改配置
   - 中期：支持模块自注册机制，各模块启动时主动向注册中心上报信息
   - 长期：结合 M8 模块纳管能力，实现完全动态的服务发现

### 7.3 低优先级

5. **补充类型存根和 API 文档**：为公共 API 编写完整的类型注解和 docstring，考虑生成 .pyi 文件。使用 pdoc 或 sphinx 生成 API 文档网站。

6. **建立基础库测试体系**：为配置管理、模块调用客户端、进程管理器等核心组件编写单元测试，目标覆盖率 70% 以上。引入 CI 确保每次修改都通过测试。

7. **可选依赖拆分**：将语音 VAD、numpy 等重型依赖设为可选依赖（extras_require），需要的模块单独安装 `shared[voice]`，减轻不需要语音功能模块的依赖负担。
