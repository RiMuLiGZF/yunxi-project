## 二、功能实现情况

### 核心功能（已实现）

1. **四层潮汐记忆模型**
   - **L0 沙滩（即时记忆）**：内存存储，高读写速度，短生命周期（秒-分钟级）
   - **L1 浅水（短期记忆）**：SQLite 存储，中等速度，中生命周期（小时-天级）
   - **L2 深水（长期记忆）**：SQLite + 向量索引，中低速度，长生命周期（月年级）
   - **L3 深海（永久记忆）**：文件系统 + AES 加密归档，低速，永久生命周期
   - 记忆自动在四层间流转（潮汐升降）

2. **三级域权限隔离**
   - **系统域**：系统级记忆，所有用户共享
   - **用户域**：用户级记忆，仅用户本人可访问
   - **会话域**：会话级记忆，仅当前会话可见
   - 域间严格隔离，支持跨域授权访问

3. **四级密级标记**
   - **公开（Public）**：可对外披露
   - **内部（Internal）**：内部使用
   - **机密（Confidential）**：敏感信息
   - **绝密（Top Secret）**：最高密级
   - 记忆写入时标注密级，读取时根据权限过滤

4. **EI 情绪推断**
   - 基于记忆内容的情绪分析与推断
   - 情绪标签与记忆关联存储
   - 支持情绪维度的记忆检索
   - 情绪变化趋势追踪

5. **睡眠记忆巩固**
   - 低峰期自动执行记忆巩固（类似睡眠中的记忆整理）
   - 重要记忆从浅层向深层迁移
   - 冗余记忆清理与合并
   - 记忆摘要与索引优化

6. **向量 + 关键词双路召回**
   - **向量召回**：基于语义相似度的向量搜索（FAISS 索引）
   - **关键词召回**：基于关键词匹配的全文检索
   - **双路融合**：两路结果加权融合排序
   - 支持混合检索与可配置权重

7. **完整审计日志**
   - 所有记忆读写操作全程审计
   - 审计日志不可篡改
   - 支持按操作类型、时间、用户维度审计查询
   - 满足合规审计要求

### 重要功能（已实现）

1. **记忆统计与监控**
2. **记忆归档与恢复**
3. **记忆搜索 API**

## 三、系统定位与作用

潮汐记忆（M5）是整个云汐系统的**记忆核心**，为所有上游模块提供统一的记忆存储与检索服务，是系统的"大脑海马体"。

### 在系统中的角色

- **统一记忆存储**：集中管理系统所有记忆数据，提供标准化的读写接口
- **记忆层级管理**：通过四层潮汐模型，在性能、容量、成本间取得平衡
- **安全与合规**：三级域隔离 + 四级密级 + 审计日志，保障数据安全与合规
- **智能检索引擎**：向量 + 关键词双路召回，提供精准的记忆检索能力
- **记忆生命周期管理**：通过睡眠巩固机制，自动管理记忆的流转与淘汰

### 与其他模块的关系

```
M1 调度中心 ──┐
M2 技能集群 ──┤
M3 端云协同 ──┼──→ M5 潮汐记忆（统一记忆层）
M4 场景引擎 ──┘
              │
              ├──→ L0 沙滩（内存，即时记忆）
              ├──→ L1 浅水（SQLite，短期记忆）
              ├──→ L2 深水（SQLite + 向量，长期记忆）
              └──→ L3 深海（文件 + AES，永久记忆）
```

- **上游**：M1、M2、M3、M4 等所有需要记忆能力的模块
- **下游**：无（M5 是最底层的记忆存储服务）
- **横向**：通过统一 API 为所有模块提供记忆服务，是系统的核心基础设施

│   │   ├── consolidator.py    # 记忆巩固器

### 核心组件

| 组件 | 职责 | 技术实现 |
|------|------|----------|
| TideMemory | 潮汐记忆主入口 | 门面模式，统一四层记忆接口 |
| L0BeachStore | L0 沙滩存储 | 内存字典 + TTL 过期 |
| L1ShallowStore | L1 浅水存储 | SQLite（直接使用 sqlite3，非 SQLAlchemy ORM） |
| L2DeepStore | L2 深水存储 | SQLite + FAISS 向量索引 |
| L3AbyssStore | L3 深海存储 | 文件系统 + AES-256 加密 |
| MemoryConsolidator | 记忆巩固器 | 定时任务 + 记忆迁移策略 |
| EmotionInferencer | EI 情绪推断 | 规则引擎 + 可选模型推理 |
| HybridSearchEngine | 混合搜索引擎 | 向量召回 + 关键词召回 + 融合排序 |
| DomainAuth | 域权限管理 | 三级域 + 权限校验中间件 |
| ClassificationManager | 密级管理 | 四级密级 + 标签传播 |
| AuditLogger | 审计日志 | 追加式写入 + 完整性校验 |

### 数据流

```
记忆写入请求
    │
    ▼
鉴权（JWT） + 域权限校验 + 密级标记
    │
    ▼
写入 L0 沙滩（内存，高速）
    │
    ├──→ 高频访问 → 保留在 L0 / L1
    ├──→ 中频访问 → 迁移至 L1（SQLite）
    └──→ 低频但重要 → 迁移至 L2（向量索引）
              │
              └──→ 超长期归档 → 迁移至 L3（加密文件）

记忆读取请求
    │
    ▼
鉴权 + 域权限 + 密级过滤
    │
    ▼
双路召回（向量 + 关键词）
    │
    ▼
结果融合排序
    │
    ▼
按密级过滤后返回

睡眠巩固（低峰期异步）
    ├──→ 记忆重要性评估
    ├──→ 层级迁移（升层 / 降层）
    ├──→ 冗余记忆合并
    └──→ 向量索引重建
```

## 六、存在的问题

### 严重问题

1. **API 风格不统一，自实现路由类**
   - **描述**：API 层使用自实现的路由类，而非 FastAPI 标准的 APIRouter
   - **影响**：
     - 与 FastAPI 生态不兼容，无法使用其依赖注入、参数校验、自动文档等特性
     - 新成员学习成本高，需要理解自定义路由框架
     - 可能存在框架本身的 bug 或性能问题
     - 升级 FastAPI 版本时兼容性风险高
   - **风险等级**：高——这是架构层面的技术债

2. **L3 深海层实现风险**
   - **描述**：L3 深海层使用文件系统 + AES 加密实现永久记忆存储
   - **风险**：
     - 文件系统的并发读写控制复杂，容易出现数据损坏
     - 加密密钥管理是难点，密钥丢失 = 数据永久丢失
     - 文件系统的备份和恢复方案复杂
     - 性能较低，大量归档记忆检索时延迟不可控
   - **建议**：评估使用对象存储（如 MinIO）替代纯文件系统

### 重要问题

3. **数据库访问方式不统一**
   - **描述**：部分存储使用 SQLAlchemy ORM，部分使用原生 sqlite3 接口
   - **影响**：代码风格不统一，事务管理、连接池、错误处理等机制不一致
   - **范围**：L1/L2 存储层主要使用原生 sqlite3，与其他模块的 SQLAlchemy 风格不一致

4. **向量搜索依赖过重**
   - **描述**：依赖 sentence-transformers、faiss-cpu、scikit-learn、numpy 等多个重型包
   - **影响**：
     - 安装包体积大（可能达数 GB）
     - 部署时间长，冷启动慢
     - 内存占用高
     - 依赖冲突风险高
   - **建议**：考虑将向量搜索服务独立部署，M5 通过 API 调用

5. **记忆巩固机制未自动化**
   - **描述**：睡眠记忆巩固功能需要手动触发（POST /api/v1/memory/consolidate）
   - **影响**：记忆层级流转不及时，L0/L1 可能堆积大量过期记忆
   - **风险**：长期不巩固会导致内存占用高、检索效率下降

6. **测试覆盖率低**
   - **描述**：单元测试和集成测试覆盖不足
   - **影响**：重构和功能迭代风险高，容易引入隐性 bug
   - **重点缺失**：加密正确性、四层流转、并发读写、边界场景

### 一般问题

7. **可选依赖未真正可选**
   - **描述**：chromadb 标记为可选，但代码中可能存在硬引用
   - **影响**：轻量部署模式不可行，必须安装全部依赖
   - **建议**：使用 try-except 导入 + 功能降级模式

8. **入口文件结构与 M4 类似**
   - **描述**：根目录 server.py 仅做转发，实际入口在 src/main.py
   - **影响**：多一层跳转，与其他模块（M1/M2/M3）的入口结构不一致

## 七、优化建议

### 高优先级

1. **API 层迁移至标准 FastAPI APIRouter**
   - **建议**：将自实现路由类逐步迁移至 FastAPI 标准 APIRouter
   - **步骤**：
     1. 新建 `api/v1/` 目录，使用标准 APIRouter 重写核心接口
     2. 保持接口路径和行为不变，确保向后兼容
     3. 新旧接口并行运行，逐步迁移所有调用方
     4. 最终移除自实现路由类
   - **预期收益**：与 FastAPI 生态对齐，自动文档、依赖注入、中间件等特性可用

2. **数据库访问统一至 SQLAlchemy ORM**
   - **建议**：将原生 sqlite3 调用全部迁移至 SQLAlchemy ORM
   - **范围**：L1 浅水层、L2 深水层
   - **预期收益**：统一的事务管理、连接池、类型安全，代码可维护性提升

3. **记忆巩固自动化**
   - **建议**：引入定时任务调度器（如 APScheduler），自动执行记忆巩固
   - **策略**：
     - 低峰期（如凌晨 2-4 点）执行全量巩固
     - 每小时执行轻量巩固（L0 → L1 迁移）
     - 可配置巩固频率和强度
   - **预期收益**：记忆层级自动流转，无需人工干预

### 中优先级

4. **向量搜索服务解耦**
   - **建议**：将向量搜索能力从 M5 中独立出来，作为独立服务或使用现有向量数据库
   - **方案**：
     - 方案 A：独立部署向量搜索服务，M5 通过 API 调用
     - 方案 B：使用成熟的向量数据库（如 Milvus、Qdrant）替代自建 FAISS
     - 方案 C：保留 FAISS 但优化为懒加载模式
   - **预期收益**：降低 M5 的依赖重量，部署更灵活

5. **L3 深海层架构优化**
   - **建议**：评估使用对象存储替代纯文件系统
   - **方案**：
     - 本地部署：MinIO 对象存储
     - 云端部署：S3 / OSS / COS
     - 统一抽象 S3 兼容接口
   - **配套**：完善密钥管理方案（KMS / Vault）
   - **预期收益**：提升可靠性和可扩展性，降低实现复杂度

6. **提升测试覆盖率**
   - **建议**：制定测试覆盖目标，重点补充以下测试：
     - 四层记忆流转的单元测试
     - 加密存储的正确性与安全性测试
     - 并发读写的集成测试
     - 记忆巩固的效果验证测试
     - 边界场景与异常注入测试
   - **目标**：核心路径覆盖率达到 80% 以上

### 低优先级

7. **可选依赖真正可选化**
   - **建议**：使用 extras_require 和 try-except 导入模式，让向量相关依赖真正可选
   - **模式**：
     ```python
     try:
         import faiss
         HAS_FAISS = True
     except ImportError:
         HAS_FAISS = False
     ```
   - **预期收益**：支持轻量化部署，降低入门门槛

8. **统一入口结构**
   - **建议**：与其他模块对齐，统一入口文件结构
   - **方案**：评估是否需要保留根目录 server.py 转发层
