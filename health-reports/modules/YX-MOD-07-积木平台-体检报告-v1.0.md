## 2. 功能实现情况

### 2.1 核心功能

- **可视化工作流编排（DAG）**：支持通过拖拽积木节点构建有向无环图工作流，定义自动化流程的执行路径。
- **8 种内置积木**：开始、结束、HTTP 请求、条件判断、变量设置、代码执行、延时等待、数据转换、子工作流，覆盖常见的流程控制和数据操作场景。
- **工作流运行与调试**：支持工作流的触发执行、运行状态跟踪、调试信息输出。
- **工作流模板库**：内置 5 个工作流模板，提供开箱即用的流程示例，降低用户上手门槛。
- **M2 技能积木扩展**：支持与 M2 技能集群集成，将 M2 技能作为积木节点接入工作流。

### 2.2 重要功能

- **工作流 CRUD**：GET/POST /api/workflows 提供工作流的创建、查询、更新、删除。
- **工作流执行**：POST /api/workflows/{id}/run 触发指定工作流运行。
- **积木目录**：GET /api/blocks 获取可用积木列表及其配置项。
- **模板管理**：GET /api/templates 模板列表查询与应用。
- **运行记录**：GET /api/runs 查看历史执行记录与状态。

### 2.3 一般功能

- **子工作流嵌套**：支持工作流嵌套调用，实现复杂流程的模块化组织。
- **变量系统**：工作流内变量的设置、传递与读取。
- **条件分支**：基于条件判断的流程分支选择。
- **HTTP 集成**：通过 HTTP 请求积木与外部系统交互。

## 3. 系统定位与作用

M7 积木平台是云汐系统的**可视化工作流编排工具**，是用户构建自动化流程的核心平台，承担以下角色：

- **流程编排中心**：用户通过拖拽积木的方式，以低代码形式构建各种自动化工作流，无需编写完整代码。
- **能力连接器**：将系统内各模块的能力（M2 技能、MCP 工具、HTTP 服务等）以积木形式封装，实现跨模块能力组合。
- **自动化引擎**：执行用户编排的工作流，按 DAG 拓扑顺序驱动各节点运行。
- **模板复用载体**：通过模板库沉淀最佳实践，加速常见场景的流程构建。

**与其他模块的关系**：
- 向上：通过 M8（管理控制塔）向用户提供可视化编排界面
- 向下/向外：调用 M2（技能集群）的技能作为积木节点
- 向外：通过 HTTP 积木调用外部服务或其他模块 API
- 依赖：使用 shared 公共基础库的配置、日志、模块调用等能力

## 6. 存在的问题

### 6.1 严重问题

暂无。

### 6.2 重要问题

**1. API 路径无 v1 前缀，与其他模块不统一**
- 描述：M7 的 API 路径为 `/api/workflows`、`/api/blocks` 等，缺少 `/api/v1/` 版本前缀，与 M6、M9、M10、M11 等模块的 `/api/v1/*` 命名规范不一致。
- 影响：API 网关统一管理和前端调用时需要特殊处理，增加集成成本。
- 优先级：高

**2. JSON 存储与 Repository 模式并存，存在冗余**
- 描述：同时存在 JSON 文件存储和 SQLAlchemy Repository 两种数据持久化方式，职责重叠，代码冗余，维护成本高。
- 影响：数据一致性难以保证，新增功能时需要同时维护两套逻辑，容易引入 bug。
- 优先级：高

**3. 开发模式下环境加载重复**
- 描述：开发模式下，环境变量存在重复加载的情况，可能导致配置覆盖或性能浪费。
- 影响：开发体验不佳，极端情况下可能引起配置不一致。
- 优先级：中

### 6.3 一般问题

**1. routers/__init__.py 内容极简**
- 描述：路由包的初始化文件内容过于简单，未统一导出所有路由，也缺少路由注册的统一入口。
- 影响：新增路由时需要手动在多处注册，容易遗漏。
- 优先级：低

**2. 版本号未明确标注**
- 描述：模块没有明确的版本标识，无法追踪迭代历史和兼容性。
- 影响：版本管理和发布跟踪困难。
- 优先级：低

**3. 缺少自动化测试**
- 描述：工作流引擎、积木执行逻辑等核心模块缺少单元测试和集成测试。
- 影响：重构风险高，回归问题难以发现。
- 优先级：低

## 7. 优化建议

### 7.1 高优先级

1. **统一 API 路径前缀**：将所有 API 路由调整为 `/api/v1/` 前缀，与 M6、M9、M10、M11 等模块保持一致。建议提供旧路径的重定向兼容，避免影响现有调用方。

2. **统一数据持久化方案**：废弃 JSON 文件存储方式，全面迁移至 SQLAlchemy + SQLite 的 Repository 模式。整理现有的数据访问逻辑，统一到 Repository 层，消除冗余代码。

### 7.2 中优先级

3. **修复环境变量重复加载**：梳理 server.py 和 src/main.py 中的配置加载逻辑，确保环境变量只加载一次，使用单例模式管理配置实例。

4. **完善 routers/__init__.py**：在路由包的 `__init__.py` 中统一导出所有路由模块，提供 `include_routers(app)` 之类的注册函数，简化路由注册流程。

5. **补充版本号管理**：在模块中添加明确的版本号标识（如 `__version__` 变量），并与 shared.version 保持联动。

### 7.3 低优先级

6. **补充核心逻辑测试**：为工作流引擎（DAG 调度、节点执行、异常处理）和关键积木编写单元测试，目标覆盖率 60% 以上。

7. **新增积木开发文档**：编写积木扩展指南，说明如何注册新积木、定义配置项、处理输入输出，降低第三方扩展门槛。
