## 2. 功能实现情况

### 2.1 核心功能

- **系统资源监控**：监控 CPU、内存、磁盘、网络、温度、GPU、电池共 7 类系统资源，提供全面的运行时状态感知。
- **进程管理**：监控各模块进程状态，支持异常进程的自动拉起和恢复，保障系统可用性。
- **四级防护策略**：INFO / WARNING / CRITICAL / EMERGENCY 四级告警机制，分级响应不同严重程度的异常事件。
- **启动安全检查**：系统启动时执行完整性和安全性检查，确保启动环境安全可靠。
- **潮汐引擎（GPU 智能调度）**：GPU 资源的智能调度与分配，优化算力使用效率。

### 2.2 重要功能

- **审计日志**：记录系统关键操作和安全事件，提供追溯能力。
- **报告生成**：生成系统健康报告和运维统计报表。
- **沙箱任务调度**：管理沙箱环境中的任务执行与资源分配。
- **状态查询 API**：GET /api/v1/status 提供系统实时状态概览。
- **进程管理 API**：GET/POST /api/v1/process 进程查询与控制。
- **防护策略 API**：GET /api/v1/guard 防护状态与策略查询。
- **审计日志 API**：GET /api/v1/audit/logs 审计记录查询。

### 2.3 一般功能

- **潮汐引擎 API**：GET /api/v1/tide/* GPU 调度相关接口。
- **资源历史趋势**：基于历史数据的资源使用趋势分析。
- **告警通知**：触发告警时的通知机制。

## 3. 系统定位与作用

M10 系统守护是云汐系统的**基础设施守护模块**，扮演 7x24 小时运维工程师的角色，承担以下职责：

- **系统监控中心**：全方位监控系统资源使用情况，是系统运行状态的"眼睛"。
- **故障自愈引擎**：进程异常自动检测和恢复，保障系统高可用性。
- **安全防护屏障**：四级防护策略和启动安全检查，主动发现和应对安全威胁。
- **GPU 调度器**：潮汐引擎负责 GPU 资源的智能分配，是算力调度的重要组成部分。
- **审计与合规**：记录系统操作日志，满足安全审计和问题追溯需求。

**与其他模块的关系**：
- 监控对象：M1-M9、M11 所有模块的进程状态和资源使用
- 向上：向 M8（管理控制塔）提供监控数据和告警信息
- 横向：与 M8 的算力调度模块在 GPU 调度上存在职责交叉
- 依赖：使用 shared 公共基础库的配置、日志、进程管理等能力

## 4. 架构设计

### 4.1 技术栈

| 类别 | 技术选型 |

### 4.2 核心组件

- **进程守护器**：进程状态监控、异常检测、自动拉起。
- **防护策略引擎**：四级告警规则匹配与响应执行。
- **启动检查器**：系统启动时的安全扫描和完整性校验。
- **审计日志服务**：操作记录的存储与查询。
- **报告生成器**：健康报告和统计报表的生成。
- **潮汐引擎**：GPU 智能调度核心模块。
- **沙箱调度器**：沙箱任务的资源分配与执行管理。

### 4.3 数据流

```
系统资源 → 采集器 → 状态聚合 → 防护策略引擎 → 告警/自愈
                    ↓
              审计日志存储
                    ↓
              报告生成服务
                    ↑
  GPU任务 → 潮汐引擎 → 调度决策 → 资源分配
```

## 6. 存在的问题

### 6.1 严重问题

**1. 新旧版本并存，架构不清晰**
- 描述：同时存在 m10_system_guard/ 新实现和 backend/ 旧实现两套代码，职责划分不明确，不清楚哪套是当前运行版本。
- 影响：维护成本翻倍，新人上手困难，容易出现修改了错误版本的问题。功能迭代时需要考虑两套代码的兼容性。
- 优先级：高

**2. 缺少统一数据库层**
- 描述：新版持久化方式不明确，旧版使用 data/yunxi_m10.db。没有统一的数据访问层，监控数据、审计日志、进程信息等数据的存储方式分散。
- 影响：数据一致性难以保证，数据查询和分析困难，扩展新功能时数据存储方案不明确。
- 优先级：高

### 6.2 重要问题

**3. requirements 太简单与实际不匹配**
- 描述：requirements.txt 中声明的依赖项过少，与代码中实际 import 的依赖不匹配。
- 影响：新环境部署时依赖安装不全，导致运行失败。依赖管理不可信。
- 优先级：中

**4. README API 路径与代码不一致**
- 描述：README 文档中描述的 API 路径与实际代码实现的路径存在差异。
- 影响：文档可信度低，API 调用方可能因路径错误导致集成失败。
- 优先级：中

**5. 潮汐引擎职责与 M10 定位交叉**
- 描述：潮汐引擎（GPU 智能调度）作为 M10 的子模块存在，但 GPU 调度属于算力调度范畴，与 M10"系统守护"的核心定位（监控、防护、自愈）存在职责交叉。M8 也有算力调度模块，可能存在重复建设。
- 影响：职责边界模糊，GPU 调度的归属不清晰，可能导致维护分散和功能重叠。
- 优先级：中

### 6.3 一般问题

**6. 版本号未明确标注**
- 描述：模块缺少明确的版本号标识。
- 影响：版本管理和迭代追踪困难。
- 优先级：低

**7. 缺少自动化测试**
- 描述：作为系统守护的关键基础设施模块，缺少自动化测试保障。
- 影响：监控逻辑、告警策略、自愈机制的正确性无法验证。
- 优先级：低

## 7. 优化建议

### 7.1 高优先级

1. **清理旧版代码，统一架构**：明确当前使用的版本，将旧版 backend/ 目录标记为废弃或迁移完成后删除。将所有功能统一到 m10_system_guard/ 目录下，建立清晰的模块边界。如果新旧版功能存在差异，制定迁移计划逐步合并。

2. **建立统一数据库层**：设计统一的数据模型，将监控数据、审计日志、进程信息、告警记录等全部纳入统一的数据库管理。基于 SQLAlchemy 建立 Repository 层，提供一致的数据访问接口。明确新版数据库的位置和命名规范。

### 7.2 中优先级

3. **完善 requirements.txt**：扫描代码中的所有 import 语句，对照 requirements.txt 补全缺失的依赖项，指定合理的版本范围。建议使用 pipreqs 或类似工具自动生成。

4. **更新 API 文档**：对照实际代码中的路由定义，逐一核对 README 中的 API 路径描述，确保文档与实现一致。建议使用 FastAPI 的自动文档（/docs）作为主要 API 参考，README 只提供概览和链接。

5. **明确潮汐引擎的归属**：重新评估潮汐引擎的定位，如果 GPU 调度是 M10 的核心职责之一，则明确其在 M10 内的独立子模块地位；如果更适合归属于 M8 的算力调度体系，则制定迁移计划将潮汐引擎迁移到 M8 或独立为新模块。避免重复建设和职责交叉。

### 7.3 低优先级

6. **补充核心功能测试**：为资源采集、进程检测、告警策略匹配等核心逻辑编写单元测试，确保守护逻辑的正确性。

7. **补充版本号和变更日志**：添加版本号管理，建立 CHANGELOG.md 记录每次迭代的变更内容。
