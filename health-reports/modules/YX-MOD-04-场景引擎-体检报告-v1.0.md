## 二、功能实现情况

### 核心功能（已实现）

1. **多场景管理**
   - 支持多场景并行存在，每个场景拥有独立的上下文空间
   - 场景生命周期管理（创建、激活、挂起、销毁）
   - 场景元数据配置（名称、描述、触发条件、关联技能）

2. **场景智能识别**
   - 关键词匹配：基于预定义关键词的快速场景识别
   - LLM 增强识别（可选）：通过大模型进行语义理解的场景识别（默认关闭）
   - 置信度评分：多候选场景排序，取最高置信度
   - 模糊匹配：支持近义词、变体词匹配

3. **场景暖切换**
   - 场景切换时保留上下文，支持快速回退
   - 切换过渡动画与状态同步
   - 切换历史记录，支持场景回溯
   - 跨场景上下文共享与隔离策略

4. **独立上下文存储**
   - 每个场景拥有独立的上下文数据库表（scene_contexts）
   - 上下文结构化存储（键值对 + 结构化数据）
   - 上下文过期自动清理
   - 上下文导入导出

5. **MCP 工具接入**
   - 场景引擎能力通过 MCP 协议对外暴露
   - 支持外部系统通过 MCP 工具查询/切换场景
   - MCP 工具自动注册与发现

6. **VSCode 集成**
   - 提供 VSCode 扩展对接接口
   - 支持编辑器内场景状态显示与切换
   - 开发场景下的实时预览与调试

### 重要功能（已实现）

1. **场景配置管理**
2. **场景识别测试接口**
3. **上下文读写 API**
4. **健康检查与基本监控**
5. **配置文件驱动（pydantic-settings + dotenv）**

## 三、系统定位与作用

场景引擎（M4）是整个云汐系统的**业务调度层**，根据用户行为和对话内容自动识别并切换工作场景，提供沉浸式的场景化体验。

### 在系统中的角色

- **场景识别器**：实时分析用户输入，判断当前所处的工作场景
- **上下文管家**：管理各场景的独立上下文，确保场景切换时数据不混乱
- **体验调度器**：通过场景暖切换，提供流畅的多任务并行体验
- **能力组织者**：根据场景自动组织相关技能和 Agent，形成场景化能力包

### 与其他模块的关系

```
M1 调度中心 ──→ M4 场景引擎 ──→ M5 潮汐记忆（场景上下文持久化）
                    │
                    ├──→ M2 技能集群（按场景组织技能集）
                    ├──→ MCP 工具暴露
                    └──→ VSCode 集成
```

- **上游**：M1 调度中心，接收场景识别和切换指令
- **下游**：M5 潮汐记忆（场景上下文持久化）、M2 技能集群（场景技能组织）
- **横向**：通过 MCP 协议与外部工具、IDE 集成

│   │   └── switcher.py        # 场景切换器
│   ├── models/                # 数据模型

> 注：目录命名为全小写 `m4-scene-engine`，与其他模块的 Pascal 命名风格不一致。

### 核心组件

| 组件 | 职责 | 技术实现 |
|------|------|----------|
| SceneManager | 场景生命周期管理 | 状态机模式 + 注册表 |
| SceneRecognizer | 场景智能识别 | 关键词匹配 + 可选 LLM 增强 |
| ContextStore | 上下文存储 | SQLAlchemy + SQLite |
| SceneSwitcher | 场景切换器 | 暖切换 + 上下文迁移 |
| MCPTools | MCP 工具接入 | MCP 协议 + 工具注册 |
| VSCodeIntegration | VSCode 集成 | WebSocket + 扩展协议 |

### 数据流

```
用户输入 / 系统事件
    │
    ▼
场景识别器（关键词匹配 → 可选 LLM 增强）
    │
    ├──→ 场景未变化 → 保持当前场景
    │
    └──→ 场景变化 → 场景切换器
             │
             ▼
        暖切换流程：
        1. 保存当前场景上下文
        2. 加载目标场景上下文
        3. 同步关联技能 / Agent
        4. 触发切换事件通知
             │
             ▼
        新场景激活 + 上下文读写
             │
             ▼
        持久化至 M5 潮汐记忆
```

## 六、存在的问题

### 严重问题

1. **场景定义硬编码**
   - **描述**：场景定义以硬编码方式写在配置文件或代码中，不支持动态创建和管理
   - **影响**：新增场景需要修改代码并重启服务，灵活性差
   - **风险**：场景数量增长后，配置管理将变得困难

2. **LLM 增强识别默认关闭，仅靠关键词匹配**
   - **描述**：虽然实现了 LLM 增强识别功能，但默认关闭，实际仅使用关键词匹配
   - **影响**：场景识别准确率有限，无法处理语义相近但关键词不同的输入
   - **原因推测**：可能是出于延迟、成本或稳定性考虑，也可能是功能尚未成熟

### 重要问题

3. **目录命名不规范**
   - **描述**：目录名为全小写 `m4-scene-engine`，而其他模块使用首字母大写（如 M1-agent-cluster、M2-skills-cluster）
   - **影响**：命名风格不一致，影响工程整体规范性
   - **建议**：统一为 `M4-scene-engine`

4. **版本号不一致**
   - **描述**：README 标注 v1.0.0，M8 接口标注 v1.2.0，版本不同步
   - **影响**：版本追溯困难，文档可信度降低

5. **技能系统与 M2 重叠**
   - **描述**：场景引擎内部也实现了一套技能组织逻辑，与 M2 技能集群的功能边界模糊
   - **影响**：职责重叠，维护成本增加，可能出现两套技能体系不一致的问题
   - **建议**：明确 M4 只负责场景级别的技能编排，具体技能执行完全委托 M2

6. **Metrics 指标薄弱**
   - **描述**：缺少场景切换频率、识别准确率、场景分布等核心业务指标
   - **影响**：无法量化场景引擎的效果，优化缺乏数据支撑

### 一般问题

7. **入口文件冗余**
   - **描述**：根目录的 server.py 仅做转发，实际入口在 src/main.py
   - **影响**：多一层跳转，增加理解成本
   - **说明**：也可能是为了兼容统一的启动约定

8. **上下文存储性能瓶颈**
   - **描述**：所有场景上下文存储在同一张 SQLite 表中，场景数量多后可能有性能问题
   - **影响**：高频上下文读写时可能出现延迟
   - **当前规模**：31 个文件、~8K 行，当前规模下问题不明显

## 七、优化建议

### 高优先级

1. **场景定义动态化**
   - **建议**：将场景定义从硬编码改为数据库存储，支持通过 API 动态创建、修改、删除场景
   - **方案**：
     1. 设计场景定义表结构（名称、描述、关键词、触发条件、关联技能、配置）
     2. 提供场景管理 CRUD API
     3. 支持场景配置热加载，无需重启服务
   - **预期收益**：场景管理灵活度大幅提升，运营可自助配置场景

2. **启用并优化 LLM 增强识别**
   - **建议**：默认启用 LLM 增强识别，或至少提供分级策略
   - **策略**：
     - 第一层：关键词快速匹配（低延迟、零成本）
     - 第二层：关键词置信度不足时，调用 LLM 增强识别（高准确率、有成本）
     - 可配置阈值，平衡准确率与成本
   - **预期收益**：场景识别准确率显著提升

3. **明确与 M2 的职责边界**
   - **建议**：M4 只负责场景识别和上下文管理，技能执行完全委托给 M2
   - **方案**：
     - M4 维护"场景-技能集"映射关系
     - 实际技能调用通过 M2 API 执行
     - 移除 M4 内部冗余的技能执行逻辑
   - **预期收益**：职责清晰，避免重复建设

### 中优先级

4. **统一目录命名规范**
   - **建议**：将 `m4-scene-engine` 重命名为 `M4-scene-engine`，与其他模块保持一致
   - **注意**：同步更新所有引用路径和文档
   - **预期收益**：工程规范度提升

5. **完善 Metrics 指标**
   - **建议**：增加场景引擎核心业务指标
   - **指标清单**：
     - 场景识别准确率（人工标注验证）
     - 场景分布（各场景活跃时长 / 调用次数）
     - 场景切换频率与平均停留时长
     - 识别延迟 P50 / P95
     - LLM 增强识别调用率与成本
   - **预期收益**：优化有数据支撑，可量化迭代效果

6. **统一版本号**
   - **建议**：README 与实际代码版本保持一致，建立单一版本号来源
   - **预期收益**：版本管理规范化

### 低优先级

7. **优化入口结构**
   - **建议**：评估是否需要保留根目录 server.py 转发层
   - **方案**：如确需统一启动约定，保留并在注释中说明；否则直接以 src/main.py 为入口

8. **上下文存储优化**
   - **建议**：当前 SQLite 单表方案可支撑中小规模，如场景数量大幅增长再考虑优化
   - **优化方向**：按场景分表、引入缓存层、迁移到 M5 记忆系统
