## 五、缺陷清单

### 5.1 P0 致命缺陷（2 个，必须修复）

#### FED-M1-P0-001：PII 脱敏功能不可达

- **等级**：P0 致命
- **模块**：federation/privacy_guard.py
- **问题**：PII 检测的 severity 默认设为 medium，导致 risk_level 直接升到 medium，走"拦截确认"分支，永远到不了 low 风险的"自动脱敏"分支。脱敏功能形同虚设。
- **根因**：三级处理策略（low→脱敏 / medium→拦截确认 / high→直接拦截）与 PII 全部标记为 medium 的设计矛盾
- **影响**：所有包含 PII 的内容都被直接拦截，用户无法获得"自动脱敏后发送"的体验
- **复现步骤**：
  1. 构造包含手机号的文本
  2. 调用 privacy_guard.scan() 扫描
  3. 观察：risk_level=medium，blocked=True，sanitized_content 未脱敏
- **预期**：轻度 PII（单个手机号/邮箱）应自动脱敏后放行
- **实际**：全部 PII 都被拦截
- **建议修复**：
  - 单条 PII（如仅1个邮箱）severity 设为 low → 自动脱敏
  - 多条 PII 叠加（3 个以上）才升到 medium → 拦截确认
  - 完善 _sanitize 方法，覆盖所有检测类型

#### FED-M1-P0-002：API Key 明文存储

- **等级**：P0 致命
- **模块**：federation/registry.py
- **问题**：API Key 以明文形式存储在内存字典 `self._api_keys` 中，`set_api_key / get_api_key` 均为明文读写，无加密/哈希/访问控制
- **影响**：
  - 进程内存 dump 可直接获取全部 API Key
  - 调试日志如果意外打印 dict 会泄露密钥
  - 不符合"API Key 加密存储"的安全基线要求
- **复现步骤**：
  1. 注册一个 Agent 并设置 api_key
  2. 调用 get_api_key() 返回明文
  3. 检查代码：无 encrypt/hash 相关逻辑
- **预期**：API Key 至少经过加密存储，读取时需鉴权
- **实际**：纯明文存储
- **建议修复**：
  - 使用 Fernet 或类似对称加密存储 API Key
  - 增加访问控制（只有特定权限才能读取明文 Key）
  - 日志中任何位置不打印 API Key

### 5.2 P1 严重缺陷（4 个，必须修复）

#### FED-M1-P1-001：PII 检测正则可轻易绕过

- **等级**：P1 严重
- **模块**：federation/privacy_guard.py
- **问题**：PII 正则检测对常见变形完全无效，包括：
  - 空格分隔：`138 1234 5678`
  - 横线分隔：`138-1234-5678`
  - 全角字符：`１３８１２３４５６７８`
  - [at]/[dot] 替换：`test[at]example[dot]com`
  - 中间插入零宽字符
- **影响**：攻击者可轻易绕过 PII 检测外发敏感数据
- **建议修复**：
  - 增加预处理步骤：去除空格/横线/特殊符号后再检测
  - 支持全角字符检测
  - 检测 [at]/[dot] 等常见绕过手法

#### FED-M1-P1-002：脱敏逻辑不完整（7 类检测无脱敏）

- **等级**：P1 严重
- **模块**：federation/privacy_guard.py _sanitize 方法
- **问题**：_sanitize 只处理了 email、phone_cn、custom_keyword 三类，其余 7 类检测无对应脱敏逻辑：
  - id_card_cn（身份证）
  - bank_card（银行卡）
  - api_key（密钥）
  - password（密码）
  - token（令牌）
  - private_key（私钥）
  - url_internal（内部域名）
- **影响**：检测到了也不知道怎么处理，只能全部拦截
- **建议修复**：为每类检测实现对应的脱敏策略

#### FED-M1-P1-003：接口无权限校验

- **等级**：P1 严重
- **模块**：api/server.py 联邦调度 API
- **问题**：以下接口无任何权限校验：
  - POST /federation/agents/register（注册 Agent）
  - POST /federation/cost/budget（修改预算）
  - DELETE /federation/agents/{id}（删除 Agent）
- **影响**：任意调用方都能注册 Agent、修改预算、删除配置
- **建议修复**：增加 API Key / Token 鉴权中间件

#### FED-M1-P1-004：缺少 GPL 协议风险提示

- **等级**：P1 严重
- **模块**：federation/registry.py
- **问题**：外部 Agent 注册时无协议/许可证字段，无法识别 GPL 等传染性协议的风险
- **影响**：接入 GPL 协议的外部服务可能带来开源协议风险
- **建议修复**：在 ExternalAgentProfile 中增加 license 字段，注册时提示协议风险

## 六、安全专项审计结论

### 6.1 整体安全评级：B-（待提升）

**优点：**
- 分层设计清晰：隐私层在调用层之前，形成有效防护
- API Key 与 Profile 分离存储，列表接口不泄露
- TOP_SECRET 级别双重拦截（调度 + 隐私），有效防止高涉密外发
- 审计日志机制存在，可追溯

**缺点：**
- 核心安全功能（PII 检测/脱敏）实现质量不高
- 密钥管理缺失，不符合安全基线
- 接口无鉴权，开放性风险高
- 渗透测试发现多种绕过手法

### 6.2 渗透测试结果

| 测试项 | 结果 | 绕过率 |
