## 一、模块整体定位与设计目标

### 1.1 内核定位

端云协同调度内核是云汐内核系统的**算力调度中枢与通信基础设施**，承担五大核心角色：

| 角色 | 职责描述 | 关键能力 |
|------|----------|----------|
| **算力路由中枢** | 根据任务特征、隐私策略、资源状态，将推理请求智能分配到端侧或云端执行 | 多维评分路由、意图亲和矩阵、实时显存感知 |
| **端云通信网关** | 管理与云端 LLM Provider 的全部网络通信，保障连接稳定性与安全性 | 连接池管理、三态熔断、指数退避重试、令牌桶限流 |
| **数据同步引擎** | 实现端侧与云端之间的上下文、配置、会话状态双向同步 | 增量 diff 同步、冲突自动解决、乐观锁版本控制 |
| **显存资源管家** | 实时监控 GPU 显存使用，通过三水位线策略保障 16GB 显存安全底线 | 实时 VRAM 监控、KV-Cache 分层管理、自适应潮汐调控 |
| **异常容错护盾** | 在 Provider 故障、网络中断、资源不足等异常场景下保障系统持续可用 | 自动降级、熔断隔离、优雅降级、异常自愈 |

该内核向上为板块1（多Agent集群）和板块2（Skill技能集群）提供统一的推理调度、数据同步和AI能力支撑，向下管理本地推理引擎（Ollama）和云端 Provider 连接。

### 1.2 六大设计目标

1. **透明算力路由**：上层 Agent 无需感知推理执行位置，内核根据任务特征自动决策端侧/云端/Hybrid 执行，路由决策延迟 < 100ms。
2. **16G 显存安全底线**：在端侧 16GB GPU 显存约束下，通过三水位线策略（安全 <70%、警戒 70%-85%、危险 >85%）和 KV-Cache 分层管理，杜绝显存 OOM，目标 24 小时零 OOM。
3. **端云上下文连贯**：用户在端侧和云端推理之间切换时，上下文信息无缝衔接，通过 ContextSyncController 实现增量 diff 同步，切换延迟 < 500ms。
4. **潮汐记忆深度联动**：与板块5潮汐记忆系统深度集成，记忆召回 < 200ms、记忆归档 < 500ms（异步），记忆覆盖率作为路由额外信号源，形成"越用越智能"正循环。
5. **异常自愈降级**：面对 Provider 故障、网络中断、显存不足等异常，系统能自动降级到备用方案（云端降端侧、端侧降云端），并在异常恢复后自动回切。
6. **全异步高性能**：全部 I/O 操作（网络、文件、数据库）使用 asyncio 异步执行，消除阻塞等待，目标端到端延迟端侧 < 2s，云端 < 5s。

### 1.3 端侧硬件约束

| 参数 | 约束值 | 设计影响 |
|------|--------|----------|
| GPU 显存 | 16GB（安全可用约 12GB，系统预留 4GB） | 必须严格管控 KV-Cache 和模型加载，单模型上限 8GB |
| 本地模型 | Ollama 7B Q4_K_M 量化（约 4.2GB VRAM） | 留 7.8GB 给 KV-Cache 和系统开销，context 上限 4096 tokens |
| 推理速度 | 48 tok/s @ RTX 4090 | 端侧推理适合短对话和快速响应场景 |
| 系统内存 | 32GB DDR | 支持将低频 KV-Cache 交换到 CPU 内存（温层），以及本地 SSD 冷层 |
| 网络环境 | 家庭/办公 WiFi，可能不稳定 | 云端调用必须有重试、熔断、降级策略 |

### 1.4 设计原则

| 原则 | 含义 | 实践体现 |
|------|------|----------|
| **端侧优先** | 能在端侧完成的任务绝不发到云端 | 路由评分中端侧权重偏高，短对话默认端侧执行 |
| **渐进增强** | 端侧能力不足时渐进引入云端能力 | Hybrid 模式：端侧初筛 + 云端精细推理 |
| **零感知切换** | 执行位置切换对上层 Agent 透明 | 统一的 LLMProvider 接口，Provider 切换延迟 < 500ms |
| **资源安全底线** | 显存 OOM、CPU 过载等绝对不允许 | 三水位线策略 + 熔断器 + 限流器 + 降级兜底 |
| **数据主权** | 用户隐私数据不离开端侧 | 隐私策略强制路由 + 审计日志 + 数据脱敏 |

## 二、分层子系统架构划分

端云协同调度内核采用**七层递进式架构**，自上而下从请求接入到资源管理，每层职责单一、接口清晰、松耦合高内聚。

```
┌─────────────────────────────────────────────────────────────┐
│  L7 接入层: AgentRequestPort + MessageBusAdapter            │
├─────────────────────────────────────────────────────────────┤
│  L6 路由层: RouteDecisionEngine + TaskComplexityAnalyzer    │
│              + PrivacyPolicyGuard + MemoryContextInjector    │
├─────────────────────────────────────────────────────────────┤
│  L5 调度层: RequestPriorityQueue(MLFQ) + BatchOptimizer     │
├─────────────────────────────────────────────────────────────┤
│  L4 执行层: LocalInferenceExecutor + CloudInferenceExecutor  │
│              + HybridAggregator                              │
├─────────────────────────────────────────────────────────────┤
│  L3 通信层: CloudGateway(aiohttp连接池) + CircuitBreaker     │
├─────────────────────────────────────────────────────────────┤
│  L2 同步层: ContextSyncController + CallLogWriter           │
│              + TideMemoryBridge                              │
├─────────────────────────────────────────────────────────────┤
│  L1 资源层: VRAMMonitor + CacheManager(LRU+TTL)             │
│              + LocalDataManager                              │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 L7 接入层（Access Layer）

#### AgentRequestPort

**职责**：作为内核对外暴露的唯一入口，接收来自板块1多Agent集群的推理请求，完成请求的标准化封装，生成唯一的 TaskTicket，交由内部流水线处理。

**核心接口**：

```python
class AgentRequestPort:
    async def submit_task(self, request: AgentRequest) -> TaskTicket:
        """接收Agent推理请求，生成TaskTicket，投递到MessageBus。
        
        Args:
            request: Agent请求对象，包含agent_id、messages、options等。
            
        Returns:
            TaskTicket: 任务票据，包含task_id、status、created_at等。
        """
        ...

    async def submit_stream(self, request: AgentRequest) -> AsyncIterator[StreamChunk]:
        """流式推理请求入口，返回异步迭代器。"""
        ...

    async def query_status(self, task_id: str) -> TaskStatus:
        """查询任务执行状态。"""
        ...

    async def cancel_task(self, task_id: str) -> bool:
        """取消排队中或执行中的任务。"""
        ...
```

**输入数据格式**：

```python
class AgentRequest(BaseModel):
    agent_id: str              # 发起请求的Agent标识
    session_id: str            # 会话ID
    messages: list[Message]   # 消息列表（system/user/assistant）
    options: RequestOptions    # 推理选项（temperature、max_tokens等）
    metadata: dict            # 扩展元数据（如优先级标签、来源等）
    stream: bool = False      # 是否流式响应
```

**输出数据格式**：

```python
class TaskTicket(BaseModel):
    task_id: str              # UUID v4
    agent_id: str
    status: TaskStatus        # pending / queued / executing / completed / failed / cancelled
    created_at: float        # time.time()
    priority: int             # 调度优先级
    route_decision: RouteDecision | None = None  # 路由决策结果（填充后）
```

**运行逻辑**：
1. 接收 AgentRequest，校验字段完整性。
2. 生成 UUID v4 作为 task_id。
3. 根据请求的 metadata 中的优先级标签，分配初始调度优先级。
4. 创建 TaskTicket，状态设为 `pending`。
5. 将 TaskTicket 发布到 MessageBus 的 `kernel.task.submit` topic。
6. 返回 TaskTicket 给调用方。

#### MessageBusAdapter

**职责**：适配板块1的 MessageBus 协议，将内核内部事件转换为 MessageBus 消息格式发布，同时订阅 MessageBus 上的外部事件。内核不直接依赖板块1的消息格式，通过 Adapter 解耦。

**核心接口**：

```python
class MessageBusAdapter:
    async def publish_event(self, topic: str, payload: dict) -> None:
        """将内核事件发布到MessageBus。"""
        ...

    async def subscribe_topic(self, topic: str, handler: Callable) -> str:
        """订阅MessageBus上的外部topic。返回subscription_id。"""
        ...

    async def unsubscribe_topic(self, subscription_id: str) -> None:
        """取消订阅。"""
        ...
```

**运行逻辑**：
1. 启动时自动订阅 `agent.task.*` 系列topic。
2. 收到消息后反序列化为内核内部对象，转发给 AgentRequestPort。
3. 内部各层完成处理后，通过 `publish_event` 发布结果事件（如 `inference.complete`、`inference.error`）。

### 2.2 L6 路由层（Routing Layer）

#### RouteDecisionEngine

**职责**：综合 TaskComplexityAnalyzer、PrivacyPolicyGuard、VRAMMonitor、MemoryContextInjector 的输出，做出最终的路由决策（端侧 / 云端 / Hybrid），并将决策结果注入 TaskTicket。

**核心接口**：

```python
class RouteDecisionEngine:
    async def decide(self, ticket: TaskTicket) -> RouteDecision:
        """综合多维度信号，做出路由决策。
        
        Args:
            ticket: 已注入记忆上下文的TaskTicket。
            
        Returns:
            RouteDecision: 路由决策，包含target、confidence、reason等。
        """
        ...

    async def re_evaluate(self, ticket: TaskTicket, decision: RouteDecision) -> RouteDecision:
        """对已有决策进行重评估（如一致性校验失败时触发）。"""
        ...
```

**输出数据格式**：

```python
class RouteDecision(BaseModel):
    target: Literal["local", "cloud", "hybrid"]  # 执行目标
    confidence: float                              # 决策置信度 0.0-1.0
    primary_provider: str                           # 主Provider名称
    fallback_provider: str | None                   # 备用Provider
    reason: str                                    # 决策理由（可观测性）
    complexity_score: float                        # 复杂度评分 0.0-1.0
    memory_coverage_score: float                    # 记忆覆盖评分 0.0-1.0
    privacy_flag: bool                             # 是否触发隐私路由
    vram_safe: bool                                # 显存是否安全
    agent_affinity: float                          # Agent意图亲和分 0.0-1.0
```

**运行逻辑**：
1. 调用 PrivacyPolicyGuard 检查隐私策略，若命中隐私规则，强制路由端侧，`privacy_flag=True`。
2. 调用 TaskComplexityAnalyzer 计算复杂度评分 `complexity_score`（6维度加权）。
3. 调用 MemoryContextInjector 获取记忆覆盖率 `memory_coverage_score`。
4. 查询 VRAMMonitor 当前显存状态 `vram_safe`。
5. 查询 Agent 亲和矩阵获取 `agent_affinity`。
6. 综合评分公式：
   ```
   route_score = 0.20 * complexity_score
               + 0.10 * memory_coverage_score
               + 0.20 * agent_affinity
               + 0.25 * (1.0 if vram_safe else 0.0)
               + 0.25 * (0.0 if privacy_flag else 1.0)
   ```
7. 若 `route_score > 0.6`，路由端侧；`0.3 < route_score <= 0.6`，路由 Hybrid；`route_score <= 0.3`，路由云端。

#### TaskComplexityAnalyzer

**核心接口**：

```python

        """分析任务复杂度。
        
        Returns:
            ComplexityReport: 包含各维度评分和加权总分。
        """
        ...
```

**输出数据格式**：

```python

    agent_type_score: float     # Agent类型评分 0.0-1.0
    code_need_score: float      # 代码需求评分 0.0-1.0
    ext_knowledge_score: float  # 外部知识需求评分 0.0-1.0
    multi_step_score: float     # 多步推理评分 0.0-1.0
```

#### PrivacyPolicyGuard

**职责**：检查请求中是否包含隐私敏感数据或涉及隐私类 Agent，命中隐私策略时强制路由端侧。

**核心接口**：

```python
class PrivacyPolicyGuard:
    def check(self, ticket: TaskTicket) -> PrivacyCheckResult:
        """隐私策略检查（纯CPU计算，无需异步）。
        
        Returns:
            PrivacyCheckResult: 包含is_private、matched_rules、forced_route。
        """
        ...

class PrivacyCheckResult(BaseModel):
    is_private: bool                  # 是否命中隐私规则
    matched_rules: list[str]         # 命中的规则ID列表
    forced_route: Literal["local"]    # 强制路由目标
    sensitivity_level: int            # 敏感度等级 1-5
```

**运行逻辑**：
1. 检查 `ticket.metadata.privacy_tags`，若包含隐私标签，`is_private=True`。
2. 检查 `agent_id` 是否在隐私 Agent 列表中（如 `agent_emotion_diary`、`agent_personal_finance`）。
3. 对 messages 内容进行关键词扫描（正则匹配身份证号、手机号、银行卡号等模式）。
4. 返回检查结果，`forced_route` 固定为 `"local"`。

**职责**：在路由决策前，从潮汐记忆系统召回相关记忆片段，注入到请求的 system prompt 中，并计算当前任务的记忆覆盖率。

```python
            MemoryInjectionResult: 包含注入的记忆片段列表和覆盖率。
        """
        ...
```

**输出数据格式**：


### 2.3 L5 调度层（Scheduling Layer）

#### RequestPriorityQueue

**职责**：基于 MLFQ（Multi-Level Feedback Queue，三级反馈队列）算法管理请求排队和调度，支持优先级动态调整和时间片轮转。

**核心接口**：

```python
class RequestPriorityQueue:
    async def enqueue(self, ticket: TaskTicket) -> None:
        """将任务加入队列，根据初始优先级放入对应层级。"""
        ...

    async def dequeue(self) -> TaskTicket | None:
        """从最高优先级非空队列中取出任务。"""
        ...

    async def promote(self, task_id: str, reason: str) -> None:
        """提升任务优先级（如等待时间过长时自动提升）。"""
        ...

    async def requeue(self, task_id: str, new_priority: int | None = None) -> None:
        """任务执行失败后重新排队（可能调整优先级）。"""
        ...

    def size(self) -> int:
        """返回当前队列中任务总数。"""
        ...
```

**三级反馈队列设计**：

| 队列层级 | 优先级范围 | 时间片 | 适用场景 | 降级规则 |
|----------|-----------|--------|----------|----------|
| Q0（高） | 80-100 | 无限制（立即执行） | 实时交互、用户直接对话 | 不降级 |
| Q1（中） | 40-79 | 5s | 常规推理、Agent内部调用 | 等待 >10s 降至 Q2 |
| Q2（低） | 0-39 | 3s | 后台任务、批量处理、记忆归档 | 等待 >30s 被丢弃 |

**运行逻辑**：
1. 新任务根据初始优先级进入对应队列。
2. 调度器始终从 Q0 开始检查，非空则取出执行；Q0 空则检查 Q1，以此类推。
3. Q1/Q2 的任务若在时间片内未完成，暂停并放回原队列尾部。
4. Q1 任务若累计等待 >10s，自动降至 Q2。
5. 队列满时（默认容量 100），拒绝新请求并返回 429。

#### BatchOptimizer

**职责**：对队列中的请求进行批量优化，通过长度桶表对齐（Bucket Binning）提升 KV-Cache 前缀共享率，减少显存碎片。

**核心接口**：

```python
class BatchOptimizer:
    async def optimize_batch(self, tickets: list[TaskTicket]) -> BatchPlan:
        """对一批任务进行批量优化编排。
        
        Returns:
            BatchPlan: 批量执行计划，包含分组、排序、KV-Cache共享策略。
        """
        ...

    def create_bucket(self, tickets: list[TaskTicket]) -> dict[int, list[TaskTicket]]:
        """按输入长度分桶，桶宽=256 tokens。"""
        ...
```

**运行逻辑**：
1. 将待执行任务按输入 token 长度分桶，桶宽 256 tokens（如 0-255、256-511、512-767...）。
2. 同桶内的任务共享 system prompt 前缀的 KV-Cache。
3. 按 system prompt 相似度（cosine similarity > 0.8）进一步分组。
4. 输出 BatchPlan，包含分组列表、每组共享前缀长度、执行顺序。

### 2.4 L4 执行层（Execution Layer）

#### LocalInferenceExecutor

**职责**：管理端侧 Ollama 推理引擎的生命周期和请求执行，处理本地推理的完整生命周期。

**核心接口**：

```python
class LocalInferenceExecutor:
    async def execute(self, ticket: TaskTicket) -> ChatResult:
        """执行端侧推理。"""
        ...

    async def execute_stream(self, ticket: TaskTicket) -> AsyncIterator[StreamChunk]:
        """流式端侧推理。"""
        ...

    async def warmup(self, model_name: str = "llama3:7b-q4_k_m") -> WarmupResult:
        """预热本地模型，预加载KV-Cache。"""
        ...

    async def health(self) -> ProviderHealth:
        """检查本地推理引擎健康状态。"""
        ...
```

**输入数据格式**：

```python
class ExecutionConfig(BaseModel):
    model: str = "llama3:7b-q4_k_m"
    temperature: float = 0.7
    max_tokens: int = 2048
    top_p: float = 0.9
    context_length: int = 4096
    keepalive: int = 300  # 5分钟
```

**输出数据格式**：

```python
class ChatResult(BaseModel):
    content: str                    # 推理输出文本
    usage: TokenUsage               # Token用量
    model: str                      # 使用的模型标识
    latency_ms: float               # 推理延迟
    provider: Literal["local", "cloud", "hybrid"]  # 执行Provider类型
    trace_id: str                   # 追踪ID
```

**运行逻辑**：
1. 从 TaskTicket 中提取 messages 和 options。
2. 检查 Ollama 服务是否可用（`health()` 检查）。
3. 通过 aiohttp 调用 Ollama 本地 API（`POST /api/chat`）。
4. 解析响应，构造 ChatResult。
5. 记录推理延迟和 token 用量到 CallLogWriter。

#### CloudInferenceExecutor

**职责**：通过 CloudGateway 调用云端 LLM Provider（OpenAI、Claude、DeepSeek），处理云端推理的完整生命周期。

**核心接口**：

```python
class CloudInferenceExecutor:
    async def execute(self, ticket: TaskTicket, provider: str) -> ChatResult:
        """执行云端推理。"""
        ...

    async def execute_stream(self, ticket: TaskTicket, provider: str) -> AsyncIterator[StreamChunk]:
        """流式云端推理。"""
        ...
```

**运行逻辑**：
1. 从 TaskTicket 获取目标 provider 名称。
2. 检查 CircuitBreaker 状态，若熔断则直接降级。
3. 通过 CloudGateway 发送 HTTP 请求到对应 Provider API。

6. 更新 CircuitBreaker 状态（成功/失败计数）。

**职责**：在 Hybrid 模式下协调端侧和云端的协同推理，实现两阶段推理（端侧初筛/草稿 + 云端精细化），以及推理一致性校验。

**职责**：实现三态熔断器（Closed / Open / Half-Open），保护系统免受级联故障。


**三态熔断器状态机**：

```
         成功            连续失败 >= threshold
  ┌──────────────┐                    ┌──────────────┐

  │  (正常通行)    │ ←───────────────── │  (熔断拒绝)   │
  └──────────────┘    resetTimeout     └──────────────┘

         │                          timeout到期
         │                                ↓
```

**参数配置**：

| 参数 | 值 | 说明 |
|------|----|------|
| volumeThreshold | 20 | 半开状态下需要累积的成功请求数 |
| errorThreshold | 50% | 失败率超过此阈值触发熔断 |
| resetTimeout | 10s | 熔断后等待时间，到期转为半开 |
| windowSize | 60s | 统计窗口时长 |

### 2.8 语义熵路由器 (SemanticEntropyRouter)

**层级**: L6 路由层（质量门控子组件）

**职责**: 对端侧本地模型输出进行多次采样（默认3次，temperature=0.8），通过语义聚类和熵值计算评估输出不确定性，作为路由质量门控的核心信号源。

**决策逻辑**:
- 低语义熵 (< 0.2): 模型确定性高 → quality_gate=True，直接交付本地结果
- 高语义熵 (> 0.4): 模型不确定性高 → quality_gate=False，建议升级云端
- 中间区域: 线性插值质量评分，>= 0.6 接受否则需进一步评估

**关键特性**:
- 纯 Python 余弦相似度计算（无 numpy 依赖），适配轻量化环境
- Union-Find 聚类算法，O(N²) 两两比较
- LRU 缓存（256条目），通过 embed 相似度 > 0.95 匹配相似 prompt
- 500ms 超时保障，采样超时自动降级为 accept_local

**与 RouteDecisionEngine 的集成**: 语义熵结果通过 `SemanticEntropyResult.quality_gate` 注入路由决策的 Phase 2 质量门控字段，形成"潮汐记忆覆盖度 + 历史路由准确率 EMA + 语义熵"三维度质量评估体系。

### 2.9 vLLM Provider (VLLMProvider)

**层级**: L4 执行层（本地高性能推理后端）

**职责**: 替代 Ollama 作为本地高性能推理后端，通过 vLLM 的 OpenAI 兼容 API 实现更高吞吐量推理。

**核心优势对比 Ollama**:

| 维度 | Ollama | vLLM (VLLMProvider) |
|------|--------|---------------------|
| KV-Cache 管理 | 无自动前缀缓存 | PagedAttention + 自动 Prefix Caching |
| 吞吐量（7B Q4） | 48 tok/s | 19x @16并发（continuous batching） |
| TTFT | ~800ms | ~300ms（prefix cache 命中时） |
| 显存利用率 | keep_alive 粗粒度 | gpu_memory_utilization 精细控制 |
| KV-Cache 监控 | 无 | /metrics Prometheus 接口 |

**Prefix Caching 实现**: 基于 session_id 追踪系统提示前缀，计算 SHA-256 hash 作为 cache key，通过 `extra_body` 传递给 vLLM，实现跨请求的 KV-Cache 复用。

**性能监控**: TTFT（首 Token 时间）、TPOT（每 Token 平均时间）、KV-Cache 利用率（从 /metrics 解析），全部注入到 ChatResult.metadata 供下游分析。

### 2.10 分布式链路追踪 (TraceContext)

**层级**: L7 接入层（全链路追踪基础设施）

**职责**: 为每个 Agent 推理请求创建唯一的 TraceContext，贯穿 enqueue → route → execute → aggregate → complete 全生命周期。

**核心设计**:
- 基于 `contextvars.ContextVar` 实现异步安全传播
- 6个管线阶段枚举 (PipelineStage): ENQUEUE / ROUTE / LOCAL_EXECUTE / CLOUD_EXECUTE / AGGREGATE / COMPLETE
- 每个阶段独立计时（record_stage / finish_stage）
- 嵌套 span 支持（`async with trace.span("operation_name")`）
- 自动绑定 structlog contextvars，所有日志自动携带 trace_id / span_id / agent_name

**数据流集成**: TaskOrchestrator.run() 在执行链路中自动创建 TraceContext，并将追踪信息注入到最终 ChatResult.metadata["trace"] 中，实现端到端可观测性。

### 2.11 令牌桶限流器 (TokenBucketRateLimiter)

**层级**: L3 通信层（请求速率控制）

**职责**: 在网关层实现经典令牌桶算法，为端云协同调度内核提供请求速率控制。

**双重限流机制**:
- **全局桶**: 限制整体请求速率（默认 50 tokens, 10 tokens/s）
- **per-agent 桶**: 每个 Agent 独立的速率限制（容量为全局的 20%）
- 双重检查: `acquire_for_agent()` 需同时通过全局桶和 agent 桶

**运维管理**: start/stop 生命周期管理、get_stats() 统计快照、reset_agent_bucket() 按 agent 重置、Prometheus 指标集成。

### 2.12 端云混合执行器 (CloudHybridExecutor)

**层级**: L4 执行层（端云并行推理聚合执行器）

**职责**: 当路由决策为 hybrid 时，CloudHybridExecutor 通过 `asyncio.gather()` 同时启动本地和云端推理，然后通过 HybridAggregator 聚合两端结果。

**四象限容错策略**:
| 本地 | 云端 | 处理策略 |
|------|------|----------|
| 成功 | 成功 | HybridAggregator 一致性校验 + 加权选优 |
| 成功 | 失败 | 降级使用本地结果 (hybrid/local only) |
| 失败 | 成功 | 降级使用云端结果 (hybrid/cloud only) |
| 失败 | 失败 | 抛出 InferenceError（含两端失败详情） |

**流式支持**: 同时启动两个流式推理，通过 HybridAggregator.aggregate_stream() 交替合并 StreamChunk。

## 三、端云分工边界与路由判定规则

### 3.1 端侧/云端/Hybrid 分工矩阵

根据任务类型、Agent 特征和资源约束，定义端侧、云端、Hybrid 三种执行模式的分工边界：

| 执行模式 | 适用场景 | 典型 Agent 类型 | 优势 | 劣势 |
|----------|---------|----------------|------|------|
| **端侧（Local）** | 短对话、情绪陪伴、轻量问答、隐私敏感、快速响应 | agent_emotion（情绪陪伴）、agent_chat（闲聊）、agent_reminder（提醒）、agent_diary（日记） | 低延迟（<2s）、隐私安全、离线可用、零成本 | 能力有限（7B模型）、长上下文受限（4096 tokens） |
| **云端（Cloud）** | 长文生成、代码开发、复杂推理、知识密集、多步规划 | agent_dev（开发助手）、agent_writer（写作）、agent_research（研究）、agent_planner（规划） | 强大能力（GPT-4o/Claude-3）、长上下文（128K+）、多模态 | 依赖网络、延迟较高（2-5s）、有成本、隐私风险 |
| **Hybrid** | 需要快速初响应 + 精细结果的场景、端云一致性校验 | agent_code_review（代码审查）、agent_analysis（数据分析）、agent_translation（翻译） | 兼顾速度和质量、本地校验云端结果 | 两阶段延迟叠加、显存占用更高 |

**Agent 类型与执行模式默认映射**：

| Agent 类型 | 默认模式 | 亲和分（端侧） | 亲和分（云端） | 说明 |
|------------|---------|---------------|---------------|------|
| agent_emotion | 端侧 | 0.95 | 0.05 | 情绪陪伴高度隐私敏感，端侧优先 |

### 3.3 任务复杂度评分模型

采用 6 维度加权评分模型，每个维度独立评分后按权重加权求和，输出 0.0-1.0 的标准化复杂度分数。

| 维度 | 权重 | 评分规则 | 评分范围 |
|------|------|---------|---------|
| **input_token** | 0.20 | `min(token_count / 4096, 1.0)`，输入 token 越多越复杂 | 0.0-1.0 |
| **agent_type** | 0.20 | 根据亲和矩阵反推，云端亲和度越高越复杂 | 0.0-1.0 |
| **code_need** | 0.15 | 消息中包含代码块/代码关键词（function、class、import等）→ 1.0；否则 → 0.0 | 0.0 / 1.0 |
| **ext_knowledge** | 0.15 | 需要外部知识（事实查询、数据分析、网络搜索）→ 1.0；否则 → 0.0 | 0.0 / 1.0 |
| **multi_step** | 0.15 | 消息中包含多步指令（"先...再...然后..."、"依次"、"分步骤"等模式）→ 1.0；否则 → 0.0 | 0.0 / 1.0 |
| **output_length** | 0.15 | 预期输出长度：短回答(< 200 tok)→0.2，中等(200-1000 tok)→0.5，长文(> 1000 tok)→1.0 | 0.2/0.5/1.0 |

**综合评分公式**：

```
complexity_score = 0.20 * input_token_score
                + 0.20 * agent_type_score
                + 0.15 * code_need_score
                + 0.15 * ext_knowledge_score
                + 0.15 * multi_step_score
                + 0.15 * output_length_score
```

## 四、完整端云数据流链路

### 4.1 九步全链路详述

#### Step 1: 任务发起（Agent -> MessageBus -> 内核）

**流程**：上层 Agent（如 agent_emotion）构造 AgentRequest 对象，通过板块1的 MessageBus 发布到 `agent.task.submit` topic。MessageBusAdapter 监听到消息后，反序列化为内部 AgentRequest 对象，转发给 AgentRequestPort。

**关键数据**：

```python
# Agent 发布到 MessageBus 的消息格式
{
    "topic": "agent.task.submit",
    "payload": {
        "agent_id": "agent_emotion",
        "session_id": "sess_abc123",
        "messages": [
            {"role": "system", "content": "你是一个温暖的情绪陪伴助手。"},
            {"role": "user", "content": "今天感觉有点焦虑..."}
        ],
        "options": {"temperature": 0.7, "max_tokens": 512},
        "metadata": {"priority": "high", "source": "interactive"},
        "stream": false
    },
    "timestamp": 1719500000.0,
    "trace_id": "trace_xyz789"
}
```

**性能目标**：从 Agent 发布到内核接收 < 5ms（进程内 MessageBus）。

---

#### Step 2: 请求接入（MessageBusAdapter -> AgentRequestPort -> TaskTicket）

**流程**：
1. AgentRequestPort 接收 AgentRequest，校验必填字段（agent_id、messages 非空）。
2. 生成 `task_id = uuid4()`。
3. 根据 `metadata.priority` 映射为调度优先级（high→80、medium→50、low→20）。
4. 创建 TaskTicket，状态设为 `pending`。
5. 将 TaskTicket 存入内存中的任务注册表（`TaskRegistry`），方便后续状态查询。
6. 发布 `kernel.task.created` 事件到 MessageBus。
7. 将 TaskTicket 投递到路由层入口。

**关键数据**：

```python
TaskTicket(
    task_id="a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    agent_id="agent_emotion",
    session_id="sess_abc123",
    status="pending",
    created_at=1719500000.1,
    priority=80,
    route_decision=None
)
```

**性能目标**：< 2ms。

---

#### Step 3: 记忆注入（TideMemoryBridge -> recall -> 注入 system prompt）

**流程**：
1. 路由层在决策前，先调用 MemoryContextInjector。
2. MemoryContextInjector 从 TaskTicket 中提取最后一条 user 消息作为 query。
3. 调用 `TideMemoryBridge.recall(query, agent_id, top_k=5)`，从板块5潮汐记忆系统召回相关记忆。
4. 计算记忆覆盖率 `coverage_score`。
5. 将召回的记忆片段格式化后注入到 messages 列表的 system prompt 中（追加在原有 system prompt 之后）。

**注入示例**：

```python
# 原始 system prompt
"You are a warm emotional companion assistant."

# 注入记忆后
"""You are a warm emotional companion assistant.

[Memory Context - recalled 3 fragments]
1. (2026-06-20) 用户提到最近工作压力大，容易焦虑。
2. (2026-06-22) 用户喜欢听轻音乐来缓解情绪。
3. (2026-06-25) 用户本周睡眠质量下降。
[End Memory Context]
"""
```

**性能目标**：recall < 200ms（P99），注入 < 5ms。

---

#### Step 4: 路由决策（PrivacyGuard -> ComplexityAnalyzer -> VRAMMonitor -> RouteDecision）

**流程**：
1. **PrivacyPolicyGuard.check()**：隐私检查（< 1ms，纯 CPU 计算）。
2. 若未命中隐私规则，**TaskComplexityAnalyzer.analyze()**：六维度复杂度分析。
3. **VRAMMonitor.get_usage()**：查询当前显存水位线。
4. **RouteDecisionEngine.decide()**：综合评分，输出 RouteDecision。
5. 将 RouteDecision 注入到 TaskTicket 的 `route_decision` 字段。
6. 记录决策日志到 CallLogWriter。
7. 发布 `kernel.route.decided` 事件。

**决策输出示例**：

```python
RouteDecision(
    target="local",
    confidence=0.92,
    primary_provider="ollama",
    fallback_provider="deepseek",
    reason="隐私安全(无敏感数据)+低复杂度(0.25)+高端侧亲和(0.95)+显存安全(safe)",
    complexity_score=0.25,
    memory_coverage_score=0.65,
    privacy_flag=False,
    vram_safe=True,
    agent_affinity=0.95
)
```

**性能目标**：路由决策 < 100ms。

---

#### Step 5: 调度排队（RequestPriorityQueue -> BatchOptimizer -> KV-Cache 前缀共享）

**流程**：
1. 将 TaskTicket（已填充路由决策）提交到 RequestPriorityQueue。
2. MLFQ 调度器根据优先级放入对应队列层级（Q0/Q1/Q2）。
3. 调度器从最高优先级非空队列取出任务。
4. BatchOptimizer 对取出的任务进行批量优化：
   - 按 input token 长度分桶（桶宽 256）。
   - 计算同桶任务间的 system prompt 相似度。
   - 识别可共享的 KV-Cache 前缀。
5. 生成 BatchPlan，指导执行层的批量执行。

**性能目标**：调度延迟 < 10ms。

---

#### Step 6: 推理执行（端侧 Ollama / 云端 API / Hybrid 两阶段）

根据 RouteDecision 的 `target` 字段，分三种执行路径：

**路径 A：端侧执行（target=local）**

```python
# LocalInferenceExecutor.execute()
1. 检查 Ollama 健康状态
2. VRAMMonitor.reserve(estimated_vram_mb)  # 预留显存
3. POST http://localhost:11434/api/chat
   {
       "model": "llama3:7b-q4_k_m",
       "messages": [...],  # 含注入记忆的 messages
       "stream": false,
       "options": {"temperature": 0.7, "num_predict": 512}
   }
4. 解析响应 → ChatResult
5. VRAMMonitor.release(estimated_vram_mb)
6. 返回 ChatResult
```

**路径 B：云端执行（target=cloud）**

```python
# CloudInferenceExecutor.execute()
1. 检查 CircuitBreaker 状态
2. 通过 CloudGateway 发送请求
3. POST https://api.openai.com/v1/chat/completions
   Headers: Authorization: Bearer sk-xxx
   Body: {"model": "gpt-4o", "messages": [...], ...}
4. 解析 SSE/JSON 响应 → ChatResult
5. 更新 CircuitBreaker 状态（成功/失败计数）
6. 返回 ChatResult
```

**路径 C：Hybrid 两阶段（target=hybrid）**

```python
# HybridAggregator.execute_two_stage()
# 阶段一：端侧快速生成草稿
local_draft = await LocalInferenceExecutor.execute(ticket)

# 阶段二：云端基于草稿精细化
cloud_request = modify_messages(ticket, hint=local_draft.content)
cloud_result = await CloudInferenceExecutor.execute(cloud_request)

# 可选：一致性校验
consistency = await HybridAggregator.validate_consistency(local_draft, cloud_result)
if consistency.similarity_score < 0.6:
    logger.warning("Hybrid一致性低", similarity=consistency.similarity_score)
```

**性能目标**：端侧首 token < 500ms，云端首 token < 2s。

---

#### Step 7: 结果回传（ChatResult -> MessageBus -> inference.complete 事件）

**流程**：
1. 执行层返回 ChatResult。
2. 将 ChatResult 封装为标准化响应消息。
3. 通过 MessageBusAdapter 发布 `inference.complete` 事件。
4. 同时更新 TaskTicket 状态为 `completed`。
5. 调用 CallLogWriter 记录调用日志（异步，不阻塞）。

**事件消息格式**：

```python
{
    "topic": "inference.complete",
    "payload": {
        "task_id": "a1b2c3d4-...",
        "agent_id": "agent_emotion",
        "content": "我理解你今天的感受，焦虑是很正常的情绪...",
        "usage": {"prompt_tokens": 256, "completion_tokens": 128, "total_tokens": 384},
        "model": "llama3:7b-q4_k_m",
        "latency_ms": 1250.5,
        "provider": "local",
        "trace_id": "trace_xyz789"
    }
}
```

---

#### Step 8: 上下文同步（ContextSyncController -> 增量 diff -> SyncClient.push）

**流程**：
1. 推理完成后，ContextSyncController 检测到会话上下文变更（新增了 assistant 回复）。
2. 生成 ContextDelta，记录变更内容。
3. 通过 SyncClient.push() 将增量变更推送到云端。
4. 更新本地版本号。
5. 若推送失败，加入重试队列（最多 3 次指数退避）。

**性能目标**：同步操作异步执行，不阻塞主流程。

---

#### Step 9: 记忆归档（TideMemoryBridge -> archive -> MemoryFragment）

**流程**：
1. 推理完成后，TideMemoryBridge 将本次对话的关键信息提取为 MemoryFragment。
2. 异步调用 `TideMemoryBridge.archive(fragment)`，归档到板块5。
3. 归档操作不阻塞主流程，失败时记录日志但不影响结果回传。
4. 归档完成后更新记忆覆盖率缓存。

**数据格式**：

```python
MemoryFragment(
    fragment_id="frag_xxx",
    agent_id="agent_emotion",
    session_id="sess_abc123",
    content="用户今天感到焦虑，可能和工作压力有关",
    timestamp=1719500001.0,
    tags=["emotion", "anxiety", "work_pressure"],
    importance=0.7,
    embedding=[0.12, -0.34, ...]  # 向量嵌入
)
```

**性能目标**：归档 < 500ms（异步）。

## 六、五大原创创新机制

### 6.1 潮汐感知路由（Tide-Aware Routing）

**核心差异**：传统路由系统仅基于任务特征（复杂度、token数）和资源状态（显存、CPU）做决策，而潮汐感知路由额外引入**记忆覆盖率**作为路由信号源，使路由决策能感知用户的"记忆积累程度"。

**设计思路**：
- 每次推理前，`TideMemoryBridge.get_coverage(agent_id)` 返回当前 Agent 的记忆覆盖率。
- 记忆覆盖率高的 Agent（如经常使用的情绪陪伴助手），说明端侧积累了丰富的个性化记忆，端侧推理质量更高。
- 将 `memory_coverage_score` 以 10% 权重纳入路由评分公式。
- 形成正循环：**使用越多 → 记忆越多 → 覆盖率越高 → 越倾向端侧 → 端侧推理质量越高 → 用户体验越好 → 使用越多**。

**技术优势**：
1. **个性化路由**：不同 Agent、不同用户有不同的路由策略，而非一刀切。
2. **动态适应**：随着用户使用，路由策略自动优化，无需手动调参。
3. **资源节省**：高覆盖率的任务留在端侧，节省云端 API 成本。

**实现要点**：

```python
# RouteDecisionEngine 中的潮汐感知评分
async def _compute_tide_score(self, agent_id: str) -> float:
    coverage = await self.tide_memory_bridge.get_coverage(agent_id)
    # 覆盖率越高，端侧路由倾向越强
    return coverage  # 0.0-1.0

# 综合评分中增加记忆覆盖率维度
route_score = (0.20 * complexity_score
             + 0.10 * memory_coverage_score    # 潮汐感知
             + 0.20 * agent_affinity
             + 0.25 * (1.0 if vram_safe else 0.0)
             + 0.25 * (0.0 if privacy_flag else 1.0))
```

### 6.2 Agent 意图梯度路由（Agent-Intent Gradient Routing）

**核心差异**：传统路由按"任务类型"（简单/复杂）粗粒度分类，而 Agent 意图梯度路由按"Agent 身份"预配置路由亲和矩阵，利用不同 Agent 的固有特征（隐私敏感度、推理复杂度倾向）实现细粒度路由，且预配置避免了实时计算的额外开销。

**设计思路**：
- 预配置 Agent-Route 亲和矩阵（见第三章 3.4 节），每个 Agent 有三个亲和分（端侧/云端/Hybrid）。
- 亲和矩阵在系统初始化时从配置文件 `~/.yunxi/config/route_affinity.yaml` 加载。
- 路由决策时直接查表获取 `agent_affinity`，无需实时分析 Agent 类型特征。
- 亲和矩阵支持运行时动态更新（通过 `config_change` 事件）。

**技术优势**：
1. **零计算开销**：查表操作 O(1)，不增加路由延迟。
2. **可解释性强**：每个 Agent 的路由倾向清晰可追溯。
3. **灵活可调**：运维人员可通过修改配置调整亲和分，无需改代码。

**实现要点**：

```python
# 亲和矩阵配置 ~/.yunxi/config/route_affinity.yaml
affinity_matrix:
  agent_emotion:
    local: 0.95
    cloud: 0.05
    hybrid: 0.00
  agent_dev:
    local: 0.10
    cloud: 0.70
    hybrid: 0.20
  # ... 其他Agent

# RouteDecisionEngine 中查表
def _get_agent_affinity(self, agent_id: str) -> float:
    return self.affinity_matrix.get(agent_id, {}).get("local", 0.5)
```

### 6.3 潮汐预热缓存（Tide Warm-up Cache）

**核心差异**：传统缓存是被动式的（请求来了才查缓存），而潮汐预热缓存是主动式的，基于历史调用日志预测使用高峰期，提前预热记忆片段和本地模型 KV-Cache，消除冷启动延迟。

**设计思路**：
1. CallLogWriter 持续记录每次推理调用的详细日志。
2. 后台分析任务每周汇总调用统计数据，识别使用高峰时段（如每日 20:00-22:00 情绪陪伴使用高峰）。
3. 在高峰期开始前 5 分钟，触发预热任务：
   - 预热高频 Agent 的记忆片段到 CacheManager 热层。
   - 预热本地 Ollama 模型的 KV-Cache（预加载常见 system prompt）。
   - 预建立与云端 Provider 的 HTTP 连接（连接池预热）。
4. 首次请求到来时，缓存已热，首延迟 < 500ms。

**技术优势**：
1. **消除冷启动**：高峰期首请求延迟从秒级降到毫秒级。
2. **智能预测**：基于真实使用模式，而非固定时间表。
3. **资源高效**：仅在高峰前预热，低谷期自动回收。

**实现要点**：

```python
class TideWarmupScheduler:
    async def predict_peak(self) -> list[PeakPrediction]:
        """基于调用日志预测未来高峰期。"""
        stats = await self.call_log_writer.query_stats(time_range=(now - 7d, now))
        # 按小时聚合调用次数，识别高峰时段
        hourly_counts = aggregate_by_hour(stats)
        peaks = detect_peaks(hourly_counts, threshold_percentile=80)
        return peaks

    async def warmup(self, peak: PeakPrediction) -> WarmupResult:
        """执行预热。"""
        # 1. 获取高峰期高频 Agent
        top_agents = get_top_agents(peak, limit=3)
        # 2. 预热记忆片段
        for agent_id in top_agents:
            fragments = await self.tide_memory_bridge.recall(
                query="", agent_id=agent_id, top_k=10
            )
            for frag in fragments:
                await self.cache_manager.set(frag.fragment_id, frag, hot=True)
        # 3. 预热 Ollama KV-Cache
        await self.local_executor.warmup()
        # 4. 预建立云端连接
        await self.cloud_gateway.warmup_connections()
```

### 6.4 端云推理一致性校验（Edge-Cloud Consistency Validation）

**核心差异**：传统 Hybrid 模式仅将端侧结果作为中间产物，而本机制将端侧 7B 模型作为"裁判"，对云端大模型的输出进行质量评估，形成**端侧监督云端**的反向校验模式。

**设计思路**：
1. Hybrid 模式下，端侧和云端分别独立推理同一请求。
2. 端侧 7B 模型不直接对比两段文本（超出 7B 能力），而是使用语义相似度（embedding cosine similarity）作为量化指标。
3. 计算端侧结果与云端结果的语义相似度 `similarity_score`。
4. 若 `similarity_score < 0.6`，标记为高不确定性结果，记录告警日志。
5. 若连续 5 次相似度 < 0.6，触发路由重评估（可能是 Agent 亲和矩阵配置不合理）。

**技术优势**：
1. **质量保障**：自动检测端云推理不一致，避免错误结果。
2. **路由自优化**：连续不一致触发重评估，驱动路由策略持续优化。
3. **低成本**：端侧 7B 推理几乎零成本，校验开销极低。

**实现要点**：

```python
class HybridAggregator:
    async def validate_consistency(self, local_result: ChatResult, 
                                    cloud_result: ChatResult) -> ConsistencyReport:
        # 使用端侧 embedding 模型计算语义相似度
        local_emb = await self.local_executor.embed([local_result.content])
        cloud_emb = await self.local_executor.embed([cloud_result.content])
        similarity = cosine_similarity(local_emb, cloud_emb)
        
        # 记录一致性历史
        self.consistency_history.append(similarity)
        
        # 连续5次不一致 → 触发重评估
        if len(self.consistency_history) >= 5:
            recent_5 = self.consistency_history[-5:]
            if all(s < 0.6 for s in recent_5):
                await self.route_engine.re_evaluate(...)
                self.consistency_history.clear()
        
        return ConsistencyReport(
            similarity_score=similarity,
            is_consistent=similarity >= 0.6,
            uncertainty_level="low" if similarity >= 0.8 
                           else "medium" if similarity >= 0.6 
                           else "high"
        )
```

### 6.5 自适应显存潮汐调控（Adaptive VRAM Tide Control）

**核心差异**：传统显存管理采用固定阈值（达到 X% 就触发），而潮汐调控模仿自然界的潮汐现象，根据显存使用率在"涨潮/平潮/退潮"三个阶段间平滑过渡，每个阶段有不同的调度策略。

**设计思路**：
- **涨潮阶段**（VRAM > 60%）：显存持续增长，需要主动防御。限制并发推理数 ≤ 2，缩短 KV-Cache 保留时间（5min → 2min），暂停后台任务（记忆压缩、缓存清理）。
- **平潮阶段**（VRAM 40%-60%）：显存稳定在中等水平。正常运行，所有优先级任务可执行，KV-Cache 保留时间 5min。
- **退潮阶段**（VRAM < 40%）：显存充裕。全优先级放开，延长 KV-Cache 保留时间（5min → 10min），启动后台任务预热缓存。

**技术优势**：
1. **平滑过渡**：避免阈值触发的突变（瞬间拒绝所有请求），提供渐进式调控。
2. **资源利用率最大化**：退潮阶段充分利用闲置显存，提升整体性能。
3. **预防性保护**：涨潮阶段提前防御，而非等到危险水位才紧急处理。

**实现要点**：

```python
class VRAMMonitor:
    @property
    def tide_phase(self) -> Literal["rising", "calm", "ebbing"]:
        ratio = self.current_usage.usage_ratio
        if ratio > 0.60:
            return "rising"    # 涨潮
        elif ratio >= 0.40:
            return "calm"      # 平潮
        else:
            return "ebbing"    # 退潮

class TideController:
    async def apply_tide_policy(self) -> None:
        phase = self.vram_monitor.tide_phase
        
        if phase == "rising":
            # 涨潮：限制并发 + 缩短KV保留 + 暂停后台
            self.request_queue.max_concurrency = 2
            self.cache_manager.set_ttl(hot=120)  # 2min
            self.background_scheduler.pause()
            
        elif phase == "calm":
            # 平潮：正常运行
            self.request_queue.max_concurrency = 5
            self.cache_manager.set_ttl(hot=300)  # 5min
            self.background_scheduler.resume()
            
        elif phase == "ebbing":
            # 退潮：全优先级 + 延长KV + 启动预热
            self.request_queue.max_concurrency = 10
            self.cache_manager.set_ttl(hot=600)  # 10min
            self.background_scheduler.warmup()
```

### 7.3 动态令牌桶限流

保护云端 API 和本地推理引擎免受过载冲击。

**参数配置**：

```python
class TokenBucketRateLimiter:
    def __init__(self):
        self.rate = 10.0        # 令牌补充速率：10 tokens/秒
        self.capacity = 50       # 桶容量：最大50个令牌
        self.tokens = 50.0       # 当前令牌数（初始满桶）
        self.last_refill = time.time()
    
    async def acquire(self, cost: float = 1.0) -> bool:
        """消耗令牌，基于 input_tokens 动态计算消耗量。
        
        Args:
            cost: 本次请求消耗的令牌数，默认1.0。
                  基于请求的 input_tokens 动态计算：
                  cost = max(1, ceil(input_tokens / 100))
        
        Returns:
            True 获取成功，False 桶中令牌不足。
        """
        self._refill()
        if self.tokens >= cost:
            self.tokens -= cost
            return True
        return False
    
    def _refill(self) -> None:
        now = time.time()
        elapsed = now - self.last_refill
        self.tokens = min(self.capacity, self.tokens + elapsed * self.rate)
        self.last_refill = now
```

### 7.7 指数退避重试策略

| 错误类型 | 是否重试 | 初始延迟 | 最大延迟 | 抖动策略 | 最大重试次数 |
|---------|---------|---------|---------|---------|------------|
| 429 (Rate Limited) | 是 | 1s | 60s | 全抖动（Full Jitter） | 5 |
| 5xx (Server Error) | 是 | 1s | 30s | 全抖动（Full Jitter） | 3 |
| 503 (Service Unavailable) | 是 | 2s | 30s | 全抖动 | 4 |
| 401/403 (Auth Error) | 否 | - | - | - | 0 |
| 400 (Bad Request) | 否 | - | - | - | 0 |
| 网络超时 | 是 | 1s | 10s | 全抖动 | 3 |

**全抖动退避公式**：

```python
import random

def calculate_backoff(retry_count: int, base_delay: float = 1.0, 
                      max_delay: float = 30.0) -> float:
    """全抖动指数退避。
    
    delay = random_between(0, min(base_delay * 2^retry_count, max_delay))
    """
    exponential_delay = base_delay * (2 ** retry_count)
    capped_delay = min(exponential_delay, max_delay)
    jittered_delay = random.uniform(0, capped_delay)
    return jittered_delay
```

### 11.1 语义熵路由实现方案

**Confidence-Driven LLM Router 核心机制**

语义熵路由（Semantic Entropy Routing）通过量化 LLM 输出不确定性来驱动路由决策，是 HybridFlow 路由引擎的 Phase 2 核心能力。其核心流程为：对同一输入进行多次独立采样，通过语义聚类将语义等价（含义相同但措辞不同）的响应归为一组，然后基于聚类分布计算信息熵。

**核心流程（3步）：**

1. **多次采样（N=3次）**：以温度 T=0.8 对同一 prompt 进行 N 次独立采样，获得候选响应集合 {y_1, y_2, ..., y_N}
2. **语义聚类**：使用蕴含检测模型（DeBERTa-v3-large-mnli）对所有响应对进行两两蕴含判断，将互相蕴含的响应归入同一语义类
3. **熵计算**：基于语义类的概率分布计算 Shannon 熵 SE(x) = -Σ p_i * log(p_i)

**关键参数配置：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 采样次数 N | 3 | 端侧资源受限，默认3次采样平衡延迟与准确性 |
| 采样温度 T | 0.8 | 适度温度以产生多样性，避免过低导致响应趋同 |
| 熵阈值 τ_low | 0.2 | 低于此值为高置信度，直接使用本地结果 |
| 熵阈值 τ_high | 0.4 | 高于此值为低置信度，路由云端处理 |
| 蕴含模型 | DeBERTa-v3-large-mnli | 微调于 MNLI 数据集，用于判断两段文本是否语义等价 |

**端侧延迟预估（7B 模型）：**

| 硬件 | 单次采样延迟 | 3次采样总延迟（vLLM continuous batching） |
|------|-------------|------------------------------------------|
| RTX 4060 (8GB) | ~0.5-1.0s | ~0.5-1.0s（并行生成，非串行3x） |

> **关键优化**：使用 vLLM 的 continuous batching 机制，3次采样请求可以合并为一个 batch 并行生成，实际延迟接近单次采样而非3倍。

**路由决策规则：**

| 条件 | 判定 | 动作 |
|------|------|------|
| SE(x) < τ_low (0.2) | 高置信度 | 直接使用本地模型输出，无需云端介入 |
| SE(x) > τ_high (0.4) | 低置信度 | 路由到云端大模型处理 |
| τ_low ≤ SE(x) ≤ τ_high | 中间带 | 执行 best-of-N 选择：从 N 个候选中选最优响应（可配合 reward model 打分） |

**Phase 2 实现规划：**

在 `RouteDecisionEngine` 中实现 `_calculate_semantic_entropy()` 方法，与现有 `_compute_confidence()` 方法协同工作。语义熵作为置信度计算的新维度（Phase 2 可设定 30% 权重），Phase 1 仍以规则基置信度为主。

### 11.2 vLLM 替代 Ollama 方案评估

**性能对比（实测数据）：**

| 指标 | vLLM（16并发） | Ollama（16并发） | 差异倍数 |
|------|---------------|----------------|---------|
| 吞吐量 | 793 tok/s | 41 tok/s | **19x** |
| 首 Token 延迟（TTFT） | 65ms | 210ms | **3x** |
| 最大并发 | 100+ | ~4（有效） | 25x |
| GPU 利用率 | ~95% | ~30% | 3x |

> **结论**：Ollama 适合开发调试（零配置启动），vLLM 适合生产部署（高吞吐、低延迟、高并发）。

**API 兼容性评估：**

vLLM 原生提供 OpenAI 兼容 API（`/v1/chat/completions`、`/v1/models`、`/v1/completions`），仅需将 `base_url` 从 `http://localhost:11434`（Ollama）改为 `http://localhost:8000`（vLLM），请求/响应格式完全一致，无需修改业务逻辑代码。

**前缀缓存（Prefix Caching）：**

vLLM 通过 `--enable-prefix-caching` 启动参数支持自动前缀缓存：

- **缓存机制**：对输入 prompt 的 hash（sha256）进行匹配，自动复用已计算的 KV Cache
- **cache_salt 隔离**：不同会话或 Agent 的缓存通过 cache_salt 参数隔离，避免跨会话污染
- **适用场景**：多轮对话中 system prompt + 历史上下文可被高效缓存，减少重复计算

**迁移建议：**

| 阶段 | 推荐后端 | 理由 |
|------|---------|------|
| 开发/调试 | Ollama | 零配置、快速启动、`ollama pull` 一键拉模型 |
| 生产部署 | vLLM | 高吞吐、低延迟、原生前缀缓存、continuous batching |

实现策略：将 `ollama_provider.py` 扩展为同时支持 Ollama 和 vLLM 后端，通过配置项 `llm_backend_type` 切换，保持上层接口不变。

**LMCache 三层 KV-Cache 分层架构：**

```
┌─────────────────────────────────────────────────┐
│  L1: GPU VRAM（热层）                            │
│  最频繁访问的 KV Cache，纳秒级访问                  │
│  容量：受显存水位线约束                            │
├─────────────────────────────────────────────────┤
│  L2: CPU DRAM（温层）                             │
│  中频访问的 KV Cache，微秒级访问                    │
│  容量：系统内存 32GB 中分配专用区域                  │
├─────────────────────────────────────────────────┤
│  L3: Redis / NVMe SSD（冷层）                     │
│  低频但需持久化的 KV Cache，毫秒级访问              │
│  Redis 用于跨端云传输；NVMe SSD 用于本地持久化      │
└─────────────────────────────────────────────────┘
```

三层之间支持自动晋升（晋升到更热层）和降级（降级到更冷层），与 vLLM 前缀缓存协同工作。

### 11.3 MCP 协议端云协同集成方案

```
端侧 Agent

    │                │     - 本地文件操作工具
    │                │     - 本地 LLM 推理工具
    │                │     - 隐私数据处理工具
    │                │
    │                └── Cloud MCP Server (remote)
    │                      - 云端搜索工具
    │                      - 云端大模型工具
    │                      - 云端知识库工具
```

**Sampling with Tools（工具采样）：**

MCP 协议的 Sampling 扩展允许 Server 端反向请求 Client 端的 LLM 能力：

- **场景**：云端 MCP Server 需要处理包含隐私数据的请求时，通过 `sampling/createMessage` 请求端侧 LLM 在本地完成推理，隐私数据不出端
- **流程**：云端 Server 发送 sampling 请求 → 端侧 Client 接收 → 端侧 LLM 本地推理 → 结果返回云端 Server
- **价值**：实现"数据不动模型动"的隐私保护范式

**Agent2Agent 通信模式：**

| 角色 | MCP 角色 | 职责 |
|------|---------|------|
| 编排 Agent | MCP Host | 发起工具调用、管理对话流程、协调专家 Agent |
| 专家 Agent | MCP Tool Provider | 提供专业能力（代码审查、文档分析、数据分析等） |

双向 MCP 交互意味着 Agent 既可以作为 Host 调用其他 Agent 的能力，也可以作为 Tool Provider 被其他 Agent 调用。

**StreamableHTTP 传输层升级：**

从 SSE（Server-Sent Events）迁移到 StreamableHTTP：

| 特性 | SSE | StreamableHTTP |
|------|-----|---------------|
| 方向性 | 单向（服务端→客户端） | 双向（支持客户端发送） |
| 会话恢复 | 不支持 | 支持（通过 session_id） |
| 传输效率 | 文本流 | 二进制流（更高效） |
| 连接管理 | 长连接，断线需重建 | 自动重连 + 会话恢复 |

> StreamableHTTP 支持网络中断后的会话恢复，对于家庭/办公 WiFi 不稳定环境尤为重要。

### 11.4 "潮汐KV-Cache"创新概念

**核心理念：**

潮汐KV-Cache（Tidal KV-Cache）借鉴潮汐的自然节律，将KV Cache设计为在端云之间周期性流动的动态资源。传统路由方案仅做"请求级"端云分发，而潮汐KV-Cache在"时间+空间"双维度上实现端云资源的协同优化。

```
     高峰期（涨潮）                低谷期（退潮）
  ┌──────────────┐              ┌──────────────┐
  │  云端热点 Cache │  ──推送──→  │  端侧 Cache 扩充 │
  │  高频访问 prompt │            │  加速本地推理    │
  └──────────────┘              └──────────────┘
  
  ┌──────────────┐              ┌──────────────┐
  │  端侧闲置 Cache │  ──回传──→  │  云端 Cache 回收  │
  │  低频访问 prompt │            │  释放端侧显存    │
  └──────────────┘              └──────────────┘
```

**技术基础（三层支撑）：**

1. **LMCache 分层存储**：提供 KV Cache 的持久化和跨层传输能力（GPU VRAM ↔ CPU DRAM ↔ Redis/NVMe SSD）
2. **vLLM 前缀缓存**：通过 sha256 hash 匹配实现 KV Cache 的精确复用和快速加载
3. **Redis L2 高速传输**：Redis 作为端云之间的 KV Cache 传输通道，支持毫秒级跨设备缓存迁移

**与 HybridFlow 的协同：**

语义熵路由预判下一阶段的路由需求：
- 当 SE(x) 趋向 τ_high 时，提前从云端预取对应 prompt 的 KV Cache 到端侧
- 当 SE(x) 趋向 τ_low 时，标记端侧 Cache 为可回收，退潮期回传云端
- 形成"预测→预取→推理→回收"的闭环

**竞赛差异化卖点：**

| 维度 | 传统方案 | 潮汐KV-Cache方案 |
|------|---------|-----------------|
| 优化维度 | 仅空间（端 or 云） | 时间 + 空间双维度 |
| 路由粒度 | 请求级 | Cache 级（更细粒度） |
| 缓存管理 | 静态分配 | 动态潮汐流动 |
| 预测能力 | 无 | 语义熵预判 + 提前预取 |
| 资源利用 | 各自为政 | 端云一体化潮汐调控 |

> 潮汐KV-Cache不是简单的路由选择，而是将端云视为统一的"推理计算池"，通过Cache的动态流动实现全局最优。
