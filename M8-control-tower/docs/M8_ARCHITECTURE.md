## 1. M8 的定位

### 1.1 核心定位

**M8 = 控制塔 / 调度中心 / 管理后台**

M8 是云汐系统的"大脑指挥中心"，负责统一管控所有模块，但**不直接承载业务功能**。

类比：
- M8 是**机场塔台**，负责调度、监控、管理
- M1~M7、M9~M12 是**飞机**，负责具体的业务飞行

### 1.2 三大核心职责

| 职责 | 说明 | 举例 |
|------|------|------|
| **管控中枢** | 统一管理所有模块的生命周期、状态、配置 | 模块启动/停止、健康检查、配置下发 |
| **流量枢纽** | 统一入口，请求路由与代理转发 | M4 业务模式代理、M5 成长系统代理 |
| **运维后台** | 系统监控、告警、审计、备份 | 资源监控、日志查询、数据备份 |

### 1.3 非核心职责（应逐步迁移）

以下功能如果在 M8 中存在，属于历史遗留，应逐步迁移到对应专业模块：

| 功能领域 | 应迁移到 | 说明 |
|----------|----------|------|
| 业务模式（成长/复盘/学习等） | M4 场景引擎 | 业务模式是 M4 的核心职责 |
| 记忆系统 | M5 潮汐记忆 | 记忆存储与检索是 M5 的核心 |
| 硬件设备管理 | M6 硬件外设 | 硬件接入与驱动是 M6 的核心 |
| 工作流编排 | M7 积木平台 | 流程编排是 M7 的核心 |
| 算力调度 | M10 系统卫士（共享算力中台） | 算力调度可保留在 M8 或下沉 |

### 2.3 "M8 做什么 / 不做什么" 原则

**M8 应该做：**
- 统一认证与权限管理
- 模块注册与发现
- 模块健康检查与状态监控
- 模块启动/停止/重启操作
- 请求路由与代理转发
- 系统级监控（CPU/内存/磁盘/网络）
- 集中告警与通知
- 审计日志收集与查询
- 数据备份与恢复
- 全局配置管理

**M8 不应该做：**
- 具体的业务模式逻辑（成长/复盘/学习等）
- 具体的记忆存储与检索
- 具体的硬件驱动与设备控制
- 具体的工作流执行引擎
- 具体的 Agent 对话逻辑
- 任何需要深度领域知识的业务功能

## 3. 代理模式设计原则

### 3.1 三种代理模式

M8 对业务模块的代理支持三种模式：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **on** | 强制代理，失败返回 503 | 业务已完全迁移到目标模块，M8 本地无实现 |
| **fallback** | 优先代理，失败回退到本地实现 | 迁移过渡期，M8 本地还有简化实现 |
| **off** | 关闭代理，使用本地实现 | 目标模块不可用或调试期 |

### 3.2 代理健康检查机制

```
调用前快速检查（5秒缓存）
    │
    ├─ 健康 → 正常转发
    │
    └─ 不健康 → 根据模式决定：
            ├─ on 模式 → 返回 503 + 友好提示
            ├─ fallback 模式 → 走本地降级逻辑
            └─ off 模式 → 走本地实现
```

**健康检查策略：**
1. 周期性后台检查（默认 30 秒间隔）
2. 请求前快速检查（带缓存，不增加延迟）
3. 连续失败触发熔断器
4. 熔断器打开后，定时尝试半开恢复

### 3.3 降级策略分级

| 降级级别 | 说明 | 用户体验 |
|----------|------|----------|
| **L0 正常** | 代理成功，返回完整功能 | 无感 |
| **L1 缓存降级** | 使用缓存数据返回 | 数据可能稍有延迟 |
| **L2 简化降级** | 返回本地简化实现 | 功能受限，但可用 |
| **L3 提示降级** | 返回友好提示，告知服务暂不可用 | 功能不可用，但有提示 |
| **L4 错误降级** | 返回标准错误响应 | 功能不可用 |

### 3.4 超时控制

- 健康检查超时：**3 秒**（快速失败）
- 普通代理请求超时：**10 秒**（可配置）
- 文件上传/大请求超时：**60 秒**（特殊接口）
- 超时后立即触发降级，不等待

## 4. 服务层架构规范

### 4.1 分层架构

```
┌─────────────────────────────────┐
│         Router 层                │  参数校验 → 调用 service → 返回响应
│  (routers/*.py)                  │  不包含业务逻辑，不直接操作数据库
├─────────────────────────────────┤
│         Service 层               │  业务逻辑封装
│  (services/*.py)                 │  可被多个 router 复用
├─────────────────────────────────┤
│         Schema 层                │  Pydantic 模型定义
│  (schemas/*.py)                  │  请求/响应数据结构
├─────────────────────────────────┤
│         Model / DAO 层           │  数据模型与数据访问
│  (models.py / data layer)        │  数据库/文件/缓存操作
├─────────────────────────────────┤
│         Shared 层                │  跨模块共享能力
│  (shared/*)                      │  统一认证、日志、配置等
└─────────────────────────────────┘
```

### 4.2 Router 层规范

**Router 只做三件事：**
1. **参数校验**：通过 Pydantic Schema 定义请求/响应格式
2. **调用 Service**：将业务逻辑委托给 Service 层
3. **构造响应**：将 Service 返回的数据包装成标准响应格式

**反模式（禁止）：**
- 在 Router 中直接写 SQL
- 在 Router 中写复杂的业务判断
- 在 Router 中内联 Pydantic 模型
- 在 Router 中直接读写文件

### 4.3 Service 层规范

**Service 层职责：**
1. 封装业务逻辑
2. 管理数据持久化
3. 处理事务一致性
4. 调用外部依赖（其他模块、第三方服务）
5. 提供可复用的业务方法

**Service 命名规范：**
- 文件名：`{domain}_service.py`（如 `auth_service.py`）
- 主类名：`{Domain}Service`（如 `AuthService`）
- 单例获取：`get_{domain}_service()`（如 `get_auth_service()`）

### 4.4 Schema 层规范

**按领域划分 Schema 文件：**
- `schemas/common.py` - 通用模型（响应、分页、排序）
- `schemas/auth.py` - 认证相关
- `schemas/user.py` - 用户管理相关
- `schemas/module.py` - 模块管理相关
- `schemas/compute.py` - 算力调度相关
- ...

**模型命名规范：**
- 请求体：`{Action}Request`（如 `LoginRequest`）
- 响应体：`{Action}Response`（如 `LoginResponse`）
- 列表项：`{Entity}ListItem`（如 `UserListItem`）
- 列表响应：`{Entity}ListResponse`

## 6. 关键设计决策记录

### ADR-001: M8 不承载业务逻辑

**决策：** M8 只做管控，不做业务。所有业务模式逻辑归属 M4 场景引擎。

**理由：**
1. 单一职责原则：M8 聚焦管控，M4 聚焦业务
2. 可扩展性：新业务模式直接在 M4 中开发，无需改动 M8
3. 稳定性：M8 作为管控中心需要高稳定性，业务迭代不应影响管控

### ADR-002: 代理模式采用中间件 + Service 双层

**决策：** M4 代理同时支持中间件层和 Service 层两种方式。

**理由：**
1. 中间件层（m4_proxy_middleware）：透明代理，对 router 零侵入
2. Service 层（proxy_fallback_service）：业务可控的降级策略
3. 两种方式互补，适用于不同场景

### ADR-003: Service 层使用单例模式

**决策：** 每个 Service 类提供 `get_xxx_service()` 函数获取单例。

**理由：**
1. 避免重复初始化
2. 便于共享状态（如缓存、连接池）
3. 与现有代码风格一致（monitor_service 已使用此模式）

### ADR-004: 用户数据暂用 JSON 文件存储

**决策：** M8 用户数据继续使用 ~/.yunxi/users.json 存储，暂不迁移到数据库。

**理由：**
1. M8 用户量小（管理员级别），JSON 足够
2. 迁移到数据库需要额外的 schema 设计和迁移工作
3. UserService 已封装数据访问层，后续迁移只需修改内部实现
