## 一、背景

根据 M5 深度调查报告和优化升级方案，当前 M5 模块存在以下 P0 级问题：

1. **L1/L2 层代码重复严重**：两层 SQLite 操作代码重复率 >80%，维护成本高
2. **M8 接口两套实现**：server.py 与 m8_interface.py 各实现一套，存在不一致风险
3. **缺少批量操作 API**：无批量写入/删除，数据迁移效率低
4. **缺少分页查询 API**：只有 top_k 截断，无法分页浏览
5. **巩固调度器未实现**：配置了 cron 但无实际调度器
6. **版本号硬编码**："2.4.0-REV2" 多处硬编码

本次 Phase 1 优化聚焦**架构对齐与基础补强**，解决上述 6 个 P0 问题。

## 二、改动内容

### 2.1 抽取 BaseSQLLayer 基类

**新增文件**：`src/tide_memory/layers/base.py`

将 L1 浅水层和 L2 深水层共有的 SQLite 操作代码抽取到 `BaseSQLLayer` 基类中。

**共有方法统一到基类：**
- `_ensure_db()` - 数据库初始化与表结构创建
- `add(item)` - 添加记忆
- `get(memory_id)` - 获取记忆（自动更新访问计数）
- `_touch(memory_id)` - 更新访问时间和计数
- `search(query, domain, top_k)` - 关键词搜索（可钩子化定制）
- `count()` - 计数
- `items()` - 返回所有记忆项
- `remove(memory_id)` - 删除
- `_row_to_item(row)` - 行转 MemoryItem
- `batch_add(items)` - 批量添加（新增）
- `batch_remove(memory_ids)` - 批量删除（新增）
- `list_items(...)` - 分页查询（新增）

**子类化设计：**
- 子类通过 `_layer_enum` 属性指定 MemoryLayer 枚举值
- 子类通过 `_get_extra_indexes()` 返回额外索引 SQL
- 子类通过 `_get_search_columns()` / `_build_search_result()` / `_sort_search_results()` 钩子定制搜索行为

**代码精简效果：**

| 模块 | 重构前 | 重构后 | 减少行数 | 精简率 |
|------|--------|--------|---------|--------|
| L1 ShallowLayer | 189 行 | 27 行 | 162 行 | 85.7% |
| L2 DeepLayer | 212 行 | 97 行 | 115 行 | 54.2% |
| BaseSQLLayer（新增） | - | 约 280 行 | - | - |
| **合计** | **401 行** | **约 404 行** | **持平** | **重复消除** |

> 注：总代码量持平是因为新增了批量操作和分页方法。核心 CRUD 代码重复率从 80%+ 降到 0。

**修改文件：**
- `src/tide_memory/layers/base.py`（新增）
- `src/tide_memory/layers/l1_shallow.py`（重写，继承基类）
- `src/tide_memory/layers/l2_deep.py`（重写，继承基类，保留 compress()）
- `src/tide_memory/layers/__init__.py`（导出 BaseSQLLayer）

### 2.2 统一 M8 接口实现

**改动前问题：**
- `server.py` 中直接实现了 `/m8/health`、`/m8/metrics`、`/m8/config` 等路由
- `api/m8_interface.py` 中的 `M8Interface` 类也有一套实现
- 两套实现逻辑重复，可能不一致

**改动方案：**
- 完善 `M8Interface` 类，补充 `m8_metrics()` 和 `m8_config()` 方法
- `server.py` 中所有 `/m8/*` 路由改为委托 `M8Interface` 调用
- 删除 `server.py` 中的 `_get_m5_real_metrics()` 等重复逻辑

**效果：**
- 业务逻辑全部收敛到 `M8Interface` 类
- `server.py` 仅负责 HTTP 层适配
- 消除约 100 行重复代码
- 确保 M8 接口响应格式统一

### 2.3 批量操作 API

**新增端点：**

| 方法 | 路径 | 功能 |
|------|------|------|
| POST | `/api/v1/memory/batch_archive` | 批量写入记忆 |
| DELETE | `/api/v1/memory/batch_delete` | 批量删除记忆 |

**batch_archive 请求体：**
```json
{
  "items": [
    { "content": "...", "tags": ["..."], ... },
    ...
  ]
}
```

**batch_archive 响应：**
```json
{
  "success_count": 95,
  "failed": ["id1", "id2"],
  "total": 100
}
```

**实现层级：**
- 数据层：`BaseSQLLayer.batch_add()` / `batch_remove()`
- 业务层：`RecallEngine.batch_archive()` / `batch_delete()`（跨层处理）
- Skill 接口层：`TideSkillInterface.batch_archive()` / `batch_delete()`（权限+审计）
- API 层：`MemoryAPIRouter.batch_archive()` / `batch_delete()`
- 入口层：`server.py` FastAPI 端点

---

### 2.4 分页查询 API

**新增端点：**

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/api/v1/memory/list` | 游标分页查询 |

**请求参数：**
| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| page_size | int | 20 | 每页条数 |
| cursor | string | null | 游标，用于获取下一页 |
| domain | string | null | 按域过滤（private/shared/core） |
| sort_by | string | created_at | 排序字段（created_at/quality_score/access_count） |
| order | string | desc | 排序方向（desc/asc） |

**响应格式：**
```json
{
  "items": [...],
  "next_cursor": "eyJjcmVhdGVkX2F0IjoiMjAyNi0wNy0xMiJ9",
  "has_more": true,
  "total": 1523
}
```

**实现要点：**
- 使用游标分页（基于排序字段值），而非 offset，性能更好
- 游标使用 base64 编码的 JSON 对象
- 跨 L1+L2 层合并后统一排序
- 支持 3 种排序字段和正反序

### 2.5 定期巩固调度器

**新增文件：** `src/tide_memory/sleep/scheduler.py`

**核心类：** `ConsolidationScheduler`

**功能：**
- 按配置的 cron 表达式定时执行记忆巩固
- 双后端实现：优先 APScheduler，回退到 threading + 轮询
- 守护线程运行，不阻塞主进程
- 支持启动/停止/状态查询

**配置项：**
| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| memory.consolidation_schedule | "0 3 * * *" | cron 表达式，默认每天凌晨3点 |
| memory.consolidation_mode | "full" | 巩固模式（fast/full/deep） |
| memory.consolidation_enabled | true | 是否启用自动巩固 |

**全局管理函数：**
- `start_scheduler(config, consolidation_engine)` - 启动调度器
- `stop_scheduler()` - 停止调度器
- `get_scheduler()` - 获取调度器实例

**集成位置：** `src/main.py` 的 `create_app()` 函数中，应用启动时自动启动调度器。

---

### 2.6 版本号统一

**改动前问题：**
- "2.4.0-REV2" 在 server.py、routes.py、m8_interface.py 等多处硬编码
- 版本更新需要修改多处，容易遗漏

**改动方案：**
统一从 `shared.version.SYSTEM_VERSION` 动态导入，优先级：
1. `shared.version.SYSTEM_VERSION`（系统统一版本）
2. `version.txt` 文件（模块本地版本）
3. 内置默认值 `"2.4.0-REV2"`（兜底）

**修改文件：**
- `src/tide_memory/__init__.py` - `__version__` 动态加载
- `src/tide_memory/api/routes.py` - health_check 使用动态版本
- `src/tide_memory/api/m8_interface.py` - 所有版本字段动态化
- `server.py` - FastAPI 元数据和启动 banner
- `src/main.py` - SYSTEM_VERSION 动态化

## 三、影响范围

### 3.1 正面影响

| 维度 | 改进前 | 改进后 |
|------|--------|--------|
| 代码重复率（L1/L2） | >80% | ~0% |
| M8 接口一致性 | 两套实现，风险高 | 统一一套，可靠 |
| 批量写入能力 | 无（逐条API调用） | 支持，效率提升 10x+ |
| 分页查询能力 | 无（只有 top_k） | 支持游标分页 |
| 自动巩固 | 配置了但未实现 | 完整实现，定时执行 |
| 版本号维护 | 5+ 处硬编码 | 1 处动态导入 |

### 3.2 兼容性

- **完全向后兼容**：所有现有 API 签名不变
- **纯新增能力**：批量 API、分页 API、调度器都是新增功能
- **内部重构**：BaseSQLLayer 抽取不影响外部调用
- **数据格式不变**：数据库表结构、响应格式完全一致

### 3.3 风险评估

| 风险点 | 等级 | 应对措施 |
|--------|------|---------|
| 基类抽取引入回归 bug | 低 | 运行全部测试，20/20 冒烟测试通过 |
| M8 接口格式变化 | 低 | 对比重构前后响应格式，完全一致 |
| 调度器资源占用 | 低 | 默认凌晨3点执行，不影响白天使用 |
| 版本号动态加载失败 | 低 | 三级降级兜底，确保始终有版本号 |

## 六、后续计划

Phase 1 完成后，进入 Phase 2 核心能力深化：

1. **可选加密原文存储** - 解决向量检索质量问题
2. **FAISS 升级为 HNSW 索引** - 检索性能从 O(n) → O(log n)
3. **SQLite WAL 模式 + 性能优化** - 并发读写性能提升
4. **L0-L1 KV 缓存联动** - 沉降与预加载机制
5. **潮汐四相循环控制器（基础版）** - 状态机驱动生命周期
