## 一、背景

根据系统体检报告，云汐系统缺少统一的代码规范和 lint 工具链，导致：

- 各模块代码风格不一致，阅读和维护成本高
- 缺少自动化代码质量检查，潜在问题难以及时发现
- 新成员加入没有明确的代码规范参考
- 提交质量参差不齐，代码 review 效率低

---

## 二、改动内容

### 2.1 新增配置文件

| 文件 | 说明 |
|------|------|
| `pyproject.toml` | 项目配置，包含 ruff + mypy 完整配置 |
| `.pre-commit-config.yaml` | Git pre-commit 钩子配置 |
| `requirements-dev.txt` | 开发依赖列表 |
| `docs/code-style-guide.md` | 中文代码风格指南文档 |

### 2.2 ruff 配置

启用 10 个规则集：

| 规则集 | 前缀 | 说明 |
|--------|------|------|
| pycodestyle | E, W | PEP8 风格检查 |
| pyflakes | F | 未使用变量/导入等错误 |
| isort | I | 导入排序 |
| pep8-naming | N | 命名规范 |
| pydocstyle | D | docstring 规范 |
| pyupgrade | UP | Python 版本升级建议 |
| flake8-bugbear | B | 常见 bug 模式 |
| flake8-simplify | SIM | 代码简化建议 |
| flake8-print | T20 | 禁止 print 语句 |

配置要点：
- 行长度：120
- 目标 Python 版本：py310
- 排除目录：`__pycache__`, `.git`, `venv`, `build`, `dist`, `data`, `logs`
- 测试文件、脚本文件、`__init__.py` 有单独的规则豁免

### 2.3 pre-commit 钩子配置

| 钩子 | 用途 |
|------|------|
| trailing-whitespace | 清理行尾空格 |
| end-of-file-fixer | 确保文件以换行结尾 |
| check-yaml | 检查 YAML 格式 |
| check-json | 检查 JSON 格式 |
| check-toml | 检查 TOML 格式 |
| check-merge-conflict | 检查合并冲突标记 |
| check-added-large-files | 检查大文件 |
| pretty-format-json | 格式化 JSON 文件 |
| ruff | Python lint + 自动修复 |
| ruff format | Python 代码格式化 |
| codespell | 拼写检查 |

### 2.4 代码风格指南文档

`docs/code-style-guide.md` 涵盖：

- **Python 代码规范**：命名、缩进、行长度、导入顺序、注释、类型提示
- **项目目录结构规范**：core/models/services/api/utils 分层推荐
- **RESTful API 设计规范**：统一响应格式、错误码规范
- **Git 提交规范**：Conventional Commits 风格（中文描述）
- **工具使用说明**：ruff、pre-commit 的安装和使用方法

## 三、使用方法

### 3.1 安装开发依赖

```powershell
cd c:\云汐\工作台\yunxi-project
pip install -r requirements-dev.txt
```

### 3.2 安装 pre-commit 钩子

```powershell
pre-commit install
```

安装后，每次 `git commit` 时会自动运行检查。

### 3.3 手动运行检查

```powershell
# 运行 ruff 检查
ruff check .

# 自动修复可修复的问题
ruff check --fix .

# 代码格式化
ruff format .

# 运行所有 pre-commit 检查
pre-commit run --all-files
```

## 四、影响范围

### 4.1 正面影响
- 统一代码风格，提升可读性和可维护性
- 提前发现潜在 bug 和代码异味
- 降低 code review 成本
- 新成员有明确的规范参考

### 4.2 兼容性
- **向后兼容**：不修改现有代码，只新增配置和规范
- **渐进式采用**：pre-commit 可选安装，不强制
- **规则务实**：避免过于严格的规则，确保可落地
