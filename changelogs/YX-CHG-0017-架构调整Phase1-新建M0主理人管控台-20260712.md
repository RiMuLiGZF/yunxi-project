## 一、背景

### 1.1 问题诊断

经过对 M8 控制塔的深度调查，发现以下核心问题：

| 问题 | 具体表现 |
|------|---------|
| **M8 职责严重过载** | 4.7万行代码、77张表、39个路由，同时承担管控/业务/算力/网关4种不同性质的职责 |
| **真正的"控制塔"功能仅占 17%** | 业务功能代码占 44%，算力中台占 21%，网关代理占 12%，自进化/巡检占 9% |
| **M4 名不副实** | 叫"场景引擎"但只有 ~8000行，核心是关键词匹配器+状态机，8大业务模式100%在 M8 里 |
| **缺少主理人视角** | 没有一个"上帝视角"的专属管控平台，主理人只能用技术管理台（M8）凑合 |

### 1.2 新架构设计

**三层架构：M0 管人 + M4 管事 + M8 管技术**

| 模块 | 定位 | 核心职责 | 类比 |
|------|------|---------|------|
| **M0 主理人管控台** | 主理人专属平台 | 全局仪表盘、主理人工具、配置中心、权限/审计、升级/回滚、紧急操作 | 舰长室 / 总控室 |
| **M4 业务场景引擎** | 业务核心层 | 8大业务模式 + 场景识别/切换引擎 + 主聊天/语音/手表 | 业务大厅 / 前台 |
| **M8 技术控制塔** | 技术基础设施 | 认证/权限、模块注册、监控告警、部署、API网关、算力调度、自进化 | 机房 / 后台 |

#### M0 特色设计

- **M8 客户端 + Fallback**：调用 M8 接口获取数据，M8 不可用时自动返回 mock 数据
- **Owner 最高权限**：独立角色体系，与 M8 共享 JWT，M0 有额外权限校验
- **深色科技风 UI**：与 M8 管理台风格一致，但视觉上更高端、更"舰长室"
- **全响应式布局**：左侧菜单 + 顶部导航 + 主内容区的经典三栏布局

#### 核心设计

**BaseMode 基类接口**：
```python
属性：mode_id / mode_name / mode_description / icon / category / priority / is_enabled
方法：
  async on_enter(context) -> dict   # 进入模式
  async on_leave(context) -> dict   # 离开模式
  async handle_message(message, context) -> dict  # 处理消息
  async get_config() -> dict        # 获取配置
  get_info() -> dict                # 获取基本信息
```

**ModeRegistry 功能**：
- 单例模式，全局唯一注册表
- register / unregister / get / list_all / list_enabled / get_by_category / get_default
- 启动时自动注册所有 8 大业务模式

**新增 API 端点（7个）**：
- `GET /api/v1/modes` - 列出所有模式（支持分类/启用状态筛选）
- `GET /api/v1/modes/{mode_id}` - 模式详情
- `POST /api/v1/modes/{mode_id}/enter` - 进入模式
- `POST /api/v1/modes/{mode_id}/leave` - 离开模式
- `GET /api/v1/modes/default/info` - 默认模式信息
- `GET /api/v1/modes/categories/list` - 分类列表
- `GET /web/` - 前端入口页

## 四、对现有系统的影响

### 4.1 零破坏性

- **M8 完全没动**：所有现有功能、API、数据保持不变
- **M4 增量扩展**：原有场景引擎功能 100% 保留，仅新增模式层
- **shared 向后兼容**：所有新增字段都有默认值，不影响现有调用

### 4.2 系统稳定性

- 新增模块独立运行，不影响现有模块
- M0 有完整的 fallback 机制，M8 不可用时也能展示 mock 数据
- M4 的模式层是纯新增，不修改原有逻辑
