## 一、背景

Phase 1 完成了架构对齐和基础补强（BaseSQLLayer、M8统一、批量/分页API、调度器、版本统一）。

Phase 2 聚焦**核心能力深化**，解决 M5 深度调查报告中指出的关键问题：

1. **"不存原文"与向量检索的根本矛盾** → 可选加密原文存储
2. **FAISS 暴力搜索性能瓶颈** → 升级 HNSW 索引
3. **SQLite 并发性能差** → WAL 模式 + 长连接 + 缓存优化
4. **L0/L1 各自独立，无智能缓存** → KV 缓存联动
5. **无生命周期状态管理** → 潮汐四相循环控制器

## 二、改动内容

### 2.1 可选加密原文存储（P1）

**解决的问题**：设计上不存储记忆原文，导致向量检索只能用标签+元数据做"代理文本"，检索质量低下。

**实现方案**：

| 维度 | 说明 |
|------|------|
| **配置项** | `store_original: bool = False`（默认关闭，隐私优先）<br>`original_encryption: bool = True`（默认加密） |
| **数据模型** | MemoryItem 新增 `original_content: Optional[str]` 字段 |
| **存储层** | 数据库表新增 `original_encrypted TEXT` 列，自动迁移 |
| **加密方案** | AES-256-GCM，复用 L3 层加密基础设施 |
| **向量检索** | 有原文用原文生成 embedding，无原文用标签+元数据代理文本 |
| **API** | 归档时可选 `store_original` 参数覆盖默认值 |
| **安全** | 原文读取有审计日志，受密级和权限控制 |

**修改文件**：
- `core/config.py` - 新增配置项
- `core/models.py` - 新增 original_content 字段
- `layers/base.py` - 加列迁移 + 加密写入/解密读取
- `recall/recall_engine.py` - 支持 store_original 参数 + 向量索引用原文
- `audit/audit_logger.py` - 新增原文读写审计类型

**设计原则**：默认关闭，隐私优先；开启后自动加密，密钥统一管理。

---

### 2.2 FAISS 升级为 HNSW 索引（P1）

**解决的问题**：IndexFlatIP 暴力搜索 O(n) 复杂度，10万+向量时检索延迟显著。

**实现方案**：

| 维度 | 说明 |
|------|------|
| **索引类型** | IndexHNSWFlat，O(log n) 复杂度 |
| **配置项** | `vector_index_type = "HNSW"`<br>`hnsw_m = 32`（连接数）<br>`hnsw_ef_construction = 200`（构建精度）<br>`hnsw_ef_search = 128`（搜索精度） |
| **向后兼容** | 已有 Flat 索引保持不变，不强制迁移 |
| **自动降级** | HNSW 不可用 → Flat → numpy，三级降级 |
| **迁移工具** | `migrate_to_hnsw()` 函数，支持手动迁移 |
| **动态调参** | `set_ef_search()` 方法，运行时调整搜索精度 |

**性能提升预期**：
- 1万向量：Flat ~5ms → HNSW ~0.5ms（10x）
- 10万向量：Flat ~50ms → HNSW ~1ms（50x）
- 100万向量：Flat ~500ms → HNSW ~3ms（150x+）

**修改文件**：
- `recall/vector_search.py` - 全面改造索引系统

---

### 2.3 SQLite WAL 模式 + 性能优化（P2）

**解决的问题**：每次操作新建/关闭连接，DELETE 模式并发差，缓存小。

**实现方案**：

| 优化项 | 说明 | 预期收益 |
|--------|------|---------|
| **WAL 模式** | `PRAGMA journal_mode=WAL` | 并发读写性能 5-10x |
| **同步级别** | `synchronous=NORMAL` | 写入性能 2-3x（WAL下仍安全） |
| **缓存大小** | `cache_size=-20000`（20MB） | 热点查询性能提升 |
| **内存临时表** | `temp_store=MEMORY` | 排序/分组性能提升 |
| **MMAP** | `mmap_size=256MB` | 大数据库读取性能提升 |
| **长连接** | `_conn` + `threading.Lock()` | 减少连接创建开销 |
| **索引优化** | 新增 idx_last_accessed / idx_quality | 查询排序性能提升 |

**长连接方案**：
- 使用 `sqlite3.connect(..., check_same_thread=False)` + 线程锁
- Python GIL + 单应用场景下足够安全
- `close()` 方法 + `__del__` 析构确保资源释放

**修改文件**：
- `layers/base.py` - BaseSQLLayer 全面改造

---

### 2.4 L0-L1 KV 缓存联动（P2）

**解决的问题**：L0 沙滩层和 L1 浅水层各自独立，没有智能的沉降和预加载机制。

**新增模块**：`layers/cache_coordinator.py` - CacheCoordinator 类

**核心机制**：

| 机制 | 说明 | 默认值 |
|------|------|--------|
| **L0→L1 沉降** | 访问次数 ≥ 阈值的记忆自动从 L0 沉降到 L1 | `settle_threshold_access = 2` |
| **L1→L0 预加载** | recall 结果 top_k 的前 N 个自动预加载到 L0 | `preload_top_n = 3` |
| **热门预加载** | 按时间窗口+访问次数批量预加载热门记忆 | `hot_window_hours = 1` |

**缓存统计**：
- L0 命中率
- 沉降次数 / 跳过次数
- 预加载次数 / 命中次数
- 可通过 metrics 接口查看

**集成方式**：
- RecallEngine 中自动集成
- 检索结果自动触发预加载
- get() 命中 L0 时检查沉降条件

**修改文件**：
- `layers/cache_coordinator.py`（新增）
- `recall/recall_engine.py`（集成）

---

### 2.5 潮汐四相循环控制器（基础版）（P2）

**解决的问题**：记忆操作直接调用接口，没有生命周期状态管理。

**新增模块**：`core/tide_phase.py` - TidePhaseController 类

**四相定义与策略**：

| 相位 | 时间段 | 缓存 | 检索 | 巩固 | 写入 |
|------|--------|------|------|------|------|
| **Flood 潮起** | 09:00-12:00 | 全开+扩容 | 超时短 | 禁用 | 小批量异步 |
| **Rising 涨潮** | 14:00-18:00 | 正常 | 正常 | 禁用 | 大批量异步+同步 |
| **Slack 平潮** | 其余时段 | 正常 | 正常 | quick模式 | 中批量同步 |
| **Ebb 潮落** | 22:00-06:00 | 关闭 | 可慢 | full+蒸馏+遗忘 | 大批量同步 |

**功能特性**：
- 基于时间的自动相位切换（Phase 2 基础版）
- 手动切换 API
- 回调注册机制（相位变化时触发）
- 下一相位预测 `get_next_phase()`
- 完整统计信息

**集成位置**：
- `main.py` - 初始化并加入 app 上下文
- `api/routes.py` - 新增相位查询和切换 API

**修改文件**：
- `core/tide_phase.py`（新增）
- `main.py`（初始化）
- `api/routes.py`（API 端点）

## 三、影响范围

### 3.1 性能提升汇总

| 优化项 | 预期提升 | 适用场景 |
|--------|---------|---------|
| HNSW 向量索引 | 50-150x 检索速度 | 10万+ 向量规模 |
| SQLite WAL 模式 | 5-10x 并发读写 | 高并发场景 |
| 长连接 + 缓存 | 2-3x 操作速度 | 频繁查询 |
| L0-L1 缓存联动 | 20-30% L0 命中率提升 | 热点记忆访问 |

### 3.2 功能增强汇总

| 功能 | 状态 | 说明 |
|------|------|------|
| 原文加密存储 | 新增 | 默认关闭，可选开启 |
| HNSW 向量索引 | 新增 | 新建默认 HNSW，旧索引保持 |
| 缓存协调器 | 新增 | L0/L1 智能联动 |
| 四相控制器 | 新增 | 时间驱动的生命周期管理 |

### 3.3 兼容性

- **完全向后兼容**：所有新功能默认关闭或自动降级
