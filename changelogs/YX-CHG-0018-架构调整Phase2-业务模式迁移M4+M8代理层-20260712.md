## 一、变更背景

### 1.1 问题分析

M8 管控塔历史上承担了过多的业务职责（4.7万行代码、77张表、39条路由），导致：
- 模块边界模糊，技术管控与业务逻辑混杂
- 维护成本高，迭代困难
- 不符合"高内聚、低耦合"的架构原则

### 1.2 架构目标

三权分立架构：
- **M0 主理人管控台** — 最高权限，主理人专属管理平台（Phase1 已完成）
- **M4 场景引擎** — 业务核心，承载所有用户-facing 的业务模式
- **M8 管控塔** — 技术基础设施，聚焦算力调度、监控、安全等技术能力

## 二、变更内容

### 2.1 M4 场景引擎扩展（核心）

#### 2.1.1 模式体系建设

| 分类 | 模式名称 | mode_id | 优先级 | 数据表 |
|------|---------|---------|--------|--------|
| 成长 | 成长中心 | `growth` | 1 | 复用 M5 潮汐记忆 |
| 工作 | 工作开发 | `work_dev` | 2 | 7张表 |
| 复盘 | 复盘总结 | `review` | 3 | 5张表 |
| 学业 | 学业规划 | `study_plan` | 4 | 7张表 |
| 生活 | 生活管理 | `life_management` | 5 | 10张表 |
| 社交 | 人际关系 | `social_relation` | 6 | 5张表 |
| 情绪 | 情绪陪伴 | `emotion_comfort` | 7 | 7张表 |
| 形象 | 形象工坊 | `appearance` | 8 | 5张表 |

#### 2.1.2 关键组件

- **`BaseMode` 抽象基类** — 定义标准模式接口（7个属性 + 4个核心方法）
- **`ModeRegistry` 模式注册表** — 单例模式，统一管理所有业务模式
- **自动注册机制** — 模块导入时自动注册全部 8 大模式，幂等设计
- **分层架构** — 每个模式独立实现 mode / models / repository / service / router 五层

#### 2.1.3 通用能力迁移

| 能力 | 路由前缀 | 核心功能 |
|------|---------|---------|
| 聊天服务 | `/api/v1/chat` | 对话管理、历史记录、模式切换 |
| 语音服务 | `/api/v1/voice` | 语音合成、语音识别、配置管理 |
| 手表交互 | `/api/v1/watch` | 设备管理、健康数据、通知推送 |

#### 2.1.4 数据层建设

- 新增 **56 张数据表**（含 8 大业务模式 + 通用能力 + 场景引擎基础表）
- SQLAlchemy ORM 统一管理
- SQLite 持久化存储
- 支持自动建表与数据库初始化

### 2.2 M8 管控塔调整

#### 2.2.1 M4 代理网关（新增）

- **文件**：`backend/routers/m4_gateway.py`
- **路由前缀**：`/api/m4-gateway`
- **代理模式数**：11 个（8 业务模式 + 聊天 + 语音 + 手表）
- **核心特性**：
  - 通用通配代理 `{mode_key}/{path:path}`，一个接口覆盖所有模式
  - 鉴权透传（`X-M8-User-Id` 请求头）
  - 错误分级处理（502/503/504）
  - 响应标识头 `X-M4-Proxied: true`
  - 环境变量驱动配置（`M4_BASE_URL` / `M4_TIMEOUT` / `M4_PROXY_ENABLED`）

#### 2.2.2 向后兼容策略

- **保留原有路由**：M8 所有业务路由本地实现保持不变
- **新增代理层**：通过独立前缀 `/api/m4-gateway` 提供 M4 代理访问
- **渐进式迁移**：可逐步切换流量，不影响现有业务

#### 2.2.3 模式管理路由接入

- `modes.py` 路由全部代理到 M4 模式管理接口
- M4 不可用时回退到本地模拟数据
- 保持 API 兼容性

### 2.3 M4 成长中心 M5 对接

- **文件**：`m4-scene-engine/src/modes/growth/m5_client.py`
- **模式**：成长中心作为 M5 潮汐记忆系统的业务外壳
- **能力对接**：成就、天赋、日历、编年史、回声、季节 6 大游戏化模块
- **降级策略**：M5 不可用时自动回退到 mock 数据，保证可用性

## 三、技术亮点

1. **标准模式接口** — `BaseMode` 统一所有业务模式的开发规范
2. **注册表模式** — `ModeRegistry` 实现模式的动态注册与发现
3. **分层架构** — mode / models / repository / service / router 清晰分层
4. **渐进式迁移** — M8 保留本地实现 + 新增代理层，平滑过渡
5. **降级容错** — M5 客户端支持 fallback 机制，保证服务可用
6. **向后兼容** — 所有原有 API 保持不变，不产生破坏性变更
