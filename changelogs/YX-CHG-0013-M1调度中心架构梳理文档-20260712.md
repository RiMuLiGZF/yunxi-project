## 一、背景

根据系统体检报告，M1 调度中心是云汐系统最复杂的模块之一，存在以下问题：

- 7 个版本的编排器（v2~v9）散落在根目录，新人难以理解
- 缺少架构文档，演进脉络不清晰
- 不知道当前主用版本，不知道哪些可以清理
- orchestrator/ 新目录定位不明确

本次不做大规模重构（风险太高），而是先通过文档化的方式梳理清楚，为后续整理奠定基础。

## 三、核心发现

### 3.1 当前主版本
**OrchestratorV9** 是当前生产环境的统一入口。

构建方式：从 V2 开始，通过继承逐层包装，最终构建出 V9 实例。
```
V2 → V3 → V4 → V5 → V7 → V8 → V9
```

### 3.2 各版本核心特性

| 版本 | 核心特性 | 状态 |
|------|---------|------|
| V2 | Workflow + Guardrails + Tracing + AgentCard | 历史基线 |
| V3 | 记忆 + 反思 + 自适应路由 + 反馈闭环 | 历史 |
| V4 | 事件溯源 + 流式响应 + LLM + 熔断 + 持久化 | 历史 |
| V5 | 配置中心 + 向量记忆 + 插件 + MCP | 历史 |
| V6 | —（跳过，无独立文件） | 已废弃/合并 |
| V7 | 集成引擎 + 预算管理 + 任务耐久性 | 历史 |
| V8 | A2A + Checkpointer + Swarm + RBAC | 历史 |
| V9 | 语义意图V3 + GroupChat + OTLP + Ledger | **在用** |
| V10+ | OrchestratorAgent（DAG + 子Agent模式） | 演进方向 |

### 3.3 V6 跳过原因
V7 文件注释写着「在 V6 基础上」，但实际继承自 V5，目录中无 V6 文件。推断 V6 的设计在开发中被废弃或直接合并入 V7，具体原因待确认。

### 3.4 orchestrator/ 新目录定位
V10+ 架构方向，将编排器本身实现为一个子 Agent（`OrchestratorAgent`），采用 DAG 编排模式，与系统中其他子 Agent 平级协作。目前未在 `app_bootstrap.py` 中构建，属于实验/演进方向。
