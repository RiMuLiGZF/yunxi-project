## 一、背景

根据系统体检报告，M11 MCP 总线作为全系统服务治理的核心，虽然基础架构完整，但还缺少一些关键的运维能力：

- 缺少服务器健康状态自动巡检
- 缺少统一的监控统计 API
- 服务状态变更缺乏可视化手段
- 运维排查效率有待提升

本次增强聚焦在**健康检查自动巡检**和**监控统计 API**两个方向，提升 M11 的可运维性。

---

## 二、改动内容

### 2.1 健康检查服务 - `services/health_checker.py`（新建）

核心类 `McpHealthChecker` 提供自动巡检能力：

| 方法 | 说明 |
|------|------|
| `start()` / `stop()` | 启动/停止后台巡检线程，失败自动降级为手动模式 |
| `check_server(server_id)` | 异步检查单个服务器健康状态 |
| `check_all()` | 并发检查所有服务器，返回统计汇总 |
| `get_server_health(server_id)` | 获取服务器最新健康状态缓存 |
| `get_all_health()` | 获取全部服务器健康状态列表 |
| `get_health_summary()` | 获取健康状态汇总统计 |

**实现要点：**
- 使用 `httpx.AsyncClient` 异步 HTTP 客户端
- 健康检查端点优先使用 `health_check_url`（GET），无配置则尝试 `endpoint`（MCP JSON-RPC 探测）
- 记录响应时间、最后检查时间、连续失败次数
- 状态分级：`online`（正常）→ `degraded`（1~2次失败）→ `offline`（连续3次失败）
- 后台线程 + asyncio 事件循环，默认 60 秒巡检一次，超时 5 秒
- 提供单例实例 `mcp_health_checker`

### 2.2 监控统计 API - `routers/monitor.py`（新建）

提供 7 个 REST 接口，prefix 为 `/api/v1/monitor`：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/overview` | 总览统计（总调用数、成功率、平均响应时间、服务器数、工具数、健康状态） |
| GET | `/server-stats` | 各服务器调用统计（支持 limit 参数） |
| GET | `/tool-stats` | 各工具调用统计 Top N（支持 limit 参数） |
| GET | `/recent-calls` | 最近调用记录（内存缓冲区，支持 limit） |
| GET | `/health-status` | 所有服务器健康状态（含汇总和详情） |
| POST | `/health-check/{server_id}` | 手动触发单服务器健康检查 |
| POST | `/health-check-all` | 手动触发全部服务器健康检查 |

### 2.3 应用生命周期集成 - `main.py`（更新）

- **启动阶段**：初始化 `mcp_health_checker` 并启动后台巡检线程
- **关闭阶段**：优雅停止健康检查巡检，清理资源
- **路由注册**：挂载 `monitor_router.router` 到应用
- **根路径信息**：新增 5 个 monitor 相关端点信息

## 三、影响范围

### 3.1 新增能力
- 服务器健康状态自动巡检，异常自动降级标记
- 统一的监控统计 API，便于前端集成运维面板
- 手动健康检查触发，便于故障排查
- 健康状态分级（online/degraded/offline），更精细的状态感知

### 3.2 兼容性
- **完全向后兼容**：不修改现有接口，只新增能力
- **优雅降级**：健康检查线程启动失败不影响主服务
- **零侵入**：不影响现有注册、路由、调用流程

## 四、验证方法

### 4.1 接口验证

```bash
# 总览统计
curl http://localhost:8700/api/v1/monitor/overview

# 服务器统计
curl http://localhost:8700/api/v1/monitor/server-stats

# 工具统计
curl http://localhost:8700/api/v1/monitor/tool-stats

# 最近调用
curl http://localhost:8700/api/v1/monitor/recent-calls

# 健康状态
curl http://localhost:8700/api/v1/monitor/health-status

# 手动触发全量检查
curl -X POST http://localhost:8700/api/v1/monitor/health-check-all
```

### 4.2 OpenAPI 文档验证
访问 `http://localhost:8700/docs`，确认 Monitor 分组下的 7 个接口正常显示。
