## 一、背景

根据系统体检报告，云汐系统各模块版本号管理混乱，存在以下问题：

- 多个模块硬编码版本号 `"0.4.0"`，注释写着"与 shared/version.py 保持同步"但实际未同步
- `shared/version.py` 定义的系统版本为 `v1.0.0`，但各模块实际使用的版本号不一致
- 版本号格式不统一（有的带 `v` 前缀，有的不带）
- 无法通过修改一处实现全系统版本号更新

### 2.1 统一版本号来源

```python
# shared/version.py - 唯一版本来源
SYSTEM_VERSION = "v1.0.0"
BUILD_DATE = "2026-07-10"
VERSION_CODE = 100
```

### 2.3 实现方式

每个模块新增 `_load_system_version()` 函数，采用以下策略：

1. **自动定位项目根目录**：从当前文件路径向上遍历，查找包含 `shared/version.py` 的目录
2. **动态路径注入**：将项目根目录加入 `sys.path`，确保导入路径可用
3. **容错降级**：导入失败时回退到默认值 `"v1.0.0"`，保证模块能正常启动
4. **格式统一**：统一使用带 `v` 前缀的版本号格式（如 `v1.0.0`）

**示例代码：**
```python
def _load_system_version() -> str:
    """从 shared.version 导入系统版本号，失败时返回默认值。"""
    try:
        import sys
        from pathlib import Path
        _project_root = Path(__file__).parent.parent.parent
        if str(_project_root) not in sys.path:
            sys.path.insert(0, str(_project_root))
        from shared.version import SYSTEM_VERSION as _ver
        return _ver
    except Exception:
        return "v1.0.0"

SYSTEM_VERSION = _load_system_version()
```

## 三、影响范围

### 3.1 版本号统一效果
- 全系统版本号统一为 `v1.0.0`
- 修改 `shared/version.py` 即可更新全系统版本
- 各模块健康检查、配置查询、根路径信息等接口返回的版本号保持一致

### 3.2 兼容性
- **向后兼容**：各模块独立运行时，即使找不到 shared 模块也会回退到默认版本号
- **不影响功能**：版本号仅用于展示和标识，不影响业务逻辑
- **格式统一**：统一为 `vX.Y.Z` 格式，便于解析和比较

## 四、验证方法

### 4.1 各模块版本号验证
```python
# 验证各模块加载的版本号是否为 v1.0.0
from shared.version import SYSTEM_VERSION
print(SYSTEM_VERSION)  # 应为 v1.0.0
```

### 4.2 HTTP 接口验证
```bash

curl http://localhost:8002/m8/health  # M2
curl http://localhost:8004/m8/health  # M4
curl http://localhost:8005/m8/health  # M5
curl http://localhost:8008/m8/health  # M8
curl http://localhost:8010/m8/health  # M10
# 所有返回的 data.version 或 data.module_version 应为 v1.0.0
```

# 预期：模块正常启动，版本号为 v1.0.0（默认值）
