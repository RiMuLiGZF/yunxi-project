# 云汐系统灾难恢复文档

> 第四阶段 - 生产就绪：容灾与恢复验证

---

## 1. 概述

本文档定义了云汐系统的灾难恢复策略、备份方案和操作流程，确保在各种灾难场景下能够快速、安全地恢复系统服务。

### 1.1 适用范围

本文件适用于云汐系统所有模块的数据备份与灾难恢复，包括：

- **M8 控制塔** - 核心调度中心
- **M5 潮汐记忆** - 核心数据存储
- **M12 安全盾** - 安全保障
- **M4 场景引擎** - 业务核心
- **M9 开发工坊** - 生产力工具
- **M6 硬件外设** - 外围设备
- **M10 系统卫士** - 系统监控

## 2. 灾难恢复策略

### 2.1 RPO/RTO 目标

| 模块 | 优先级 | RPO（数据丢失容忍） | RTO（恢复时间目标） | 说明 |
|------|--------|---------------------|---------------------|------|
| M8 控制塔 | P0 | 24 小时 | 1 小时 | 调度中心，必须最先恢复 |
| M12 安全盾 | P1 | 24 小时 | 2 小时 | 安全保障，需快速恢复 |
| M5 潮汐记忆 | P1 | 6 小时 | 4 小时 | 核心数据，恢复优先级高 |
| M4 场景引擎 | P2 | 24 小时 | 4 小时 | 业务核心功能 |
| M9 开发工坊 | P2 | 24 小时 | 8 小时 | 开发生产工具 |
| M6 硬件外设 | P3 | 48 小时 | 24 小时 | 外围设备，可延后恢复 |
| M10 系统卫士 | P3 | 24 小时 | 12 小时 | 系统监控 |

**RPO (Recovery Point Objective)**：可容忍的最大数据丢失时间窗口
**RTO (Recovery Time Objective)**：从灾难发生到服务恢复的最长时间

### 2.2 恢复优先级

灾难发生时，按以下优先级顺序恢复：

1. **P0 - 核心调度层**：M8 控制塔
2. **P1 - 核心数据与安全层**：M12 安全盾、M5 潮汐记忆
3. **P2 - 业务功能层**：M4 场景引擎、M9 开发工坊
4. **P3 - 外围与监控层**：M6 硬件外设、M10 系统卫士

### 2.3 恢复原则

- **数据安全第一**：所有恢复操作前必须创建安全网备份
- **先验证后恢复**：恢复前验证备份文件完整性
- **先核心后外围**：按优先级顺序恢复
- **可回滚**：所有恢复操作必须支持回滚
- **可验证**：恢复后必须进行数据完整性校验

## 3. 备份策略详解

### 3.1 备份类型

#### 3.1.1 全量备份 (Full Backup)

- **定义**：备份所有数据库文件的完整副本
- **频率**：每日一次（凌晨 2:00 - 5:00）
- **保留周期**：30 天
- **适用场景**：每日基线备份、灾难恢复基础

#### 3.1.2 增量备份 (Incremental Backup)

- **定义**：备份自上一次备份（全量或增量）以来的变化
- **频率**：每 6 小时一次（核心数据模块）
- **保留周期**：7 天
- **适用场景**：M5 潮汐记忆等数据变化频繁的模块

#### 3.1.3 差异备份 (Differential Backup)

- **定义**：备份自上次全量备份以来的所有变化
- **频率**：每日中午 12:00
- **保留周期**：7 天
- **适用场景**：减少恢复时间，平衡备份开销

### 3.2 备份时间表

| 模块 | 全量备份时间 | 增量备份频率 | 压缩 | 加密 |
|------|-------------|-------------|------|------|
| M8 控制塔 | 每日 02:00 | 不启用 | gzip | 可选 |
| M5 潮汐记忆 | 每日 03:30 | 每 6 小时 | gzip | 可选 |
| M4 场景引擎 | 每日 03:00 | 不启用 | gzip | 可选 |
| M9 开发工坊 | 每日 03:00 | 不启用 | gzip | 可选 |
| M6 硬件外设 | 每日 04:00 | 不启用 | gzip | 可选 |
| M10 系统卫士 | 每日 04:30 | 不启用 | gzip | 可选 |
| M12 安全盾 | 每日 05:00 | 不启用 | gzip | 启用 |

### 3.3 备份存储

#### 3.3.1 存储位置

- **本地存储**：`backups/module_backups/{module_id}/`
- **备份文件命名**：`{module_id}_{backup_type}_{timestamp}/`
- **元数据文件**：`backup_manifest.json`、`*.meta.json`

#### 3.3.2 压缩

- **算法**：gzip (level 6)
- **压缩率**：约 40%-70%（取决于数据类型）
- **文件后缀**：`.db.gz`

#### 3.3.3 加密

- **算法**：AES-256-GCM
- **密钥长度**：256 位（32 字节）
- **文件格式**：`[12字节 nonce][密文][16字节 tag]`
- **文件后缀**：`.db.enc` 或 `.db.gz.enc`

#### 3.3.4 校验和

- **算法**：SHA-256
- **存储位置**：元数据文件 `.meta.json`
- **验证时机**：备份后立即验证、恢复前验证、定期验证

### 3.4 保留策略

| 策略类型 | 说明 | 默认值 |
|----------|------|--------|
| 按数量保留 | 保留最近 N 个备份 | 30 个 |
| 按时间保留 | 保留最近 N 天的备份 | 30 天 |
| 按空间保留 | 最大占用空间 | 10 GB |
| 混合策略 | 同时应用数量和时间策略 | 启用 |

各模块保留策略：

| 模块 | 策略 | 最大数量 | 最大天数 | 最大空间 |
|------|------|----------|----------|----------|
| M8 控制塔 | hybrid | 50 | 60 天 | 5 GB |
| M5 潮汐记忆 | hybrid | 30 | 45 天 | 10 GB |
| M12 安全盾 | hybrid | 30 | 90 天 | 2 GB |
| M4 场景引擎 | hybrid | 30 | 30 天 | 5 GB |
| M9 开发工坊 | hybrid | 30 | 30 天 | 5 GB |
| M6 硬件外设 | count | 20 | 20 天 | - |
| M10 系统卫士 | hybrid | 30 | 30 天 | 3 GB |

## 4. 常见灾难场景及处理步骤

### 4.1 场景一：单模块数据库损坏

**现象**：某个模块的数据库文件损坏或丢失

**触发条件**：
- 数据库完整性检查失败
- 应用报错 "database disk image is malformed
- 数据库文件意外删除

**处理步骤**：

1. **确认故障
```powershell
# 检测模块健康状态
.\scripts\disaster-recovery.ps1 -Detect
```

2. **确认可用备份**
```powershell
# 查看该模块的备份列表
python shared/data/data_layer/backup.py list --module m9
```

3. **执行恢复**
```powershell
# 恢复指定模块（自动启用安全网）
.\scripts\disaster-recovery.ps1 -Module m9
```

4. **验证恢复结果**
```powershell
# 验证备份完整性
.\scripts\verify-backup.ps1 -Module m9 -TestRestore
```

5. **验证服务正常**
```powershell
# 健康检查
.\scripts\health-check.ps1
```

**预期 RTO**：15-30 分钟

### 4.2 场景二：多模块同时故障

**现象**：多个模块数据库同时损坏（如磁盘故障、误操作）

**处理步骤**：

1. **全面检测故障**
```powershell
.\scripts\disaster-recovery.ps1 -Detect
```

2. **按优先级恢复**
```powershell
# 恢复多个模块（自动按优先级排序）
.\scripts\disaster-recovery.ps1 -Modules m5,m9
```

3. **验证各模块恢复状态**
```powershell
.\scripts\verify-backup.ps1 -All -TestRestore
```

4. **验证依赖关系**
   - M8 恢复后验证调度功能正常
   - M5 恢复后验证数据访问正常
   - 业务模块恢复后验证端到端功能

**预期 RTO**：1-4 小时（取决于模块数量和数据量）

### 4.3 场景三：配置文件丢失

**现象**：系统配置文件损坏或丢失

**处理步骤**：

1. **确认配置文件丢失情况
2. **从配置备份恢复**
3. **验证配置完整性**
4. **重启相关服务**

**预期 RTO**：15-30 分钟

### 4.4 场景四：整个系统故障

**现象**：整个系统无法正常运行（如服务器宕机）

**处理步骤**：

1. **评估灾难范围**
2. **按优先级顺序恢复**（P0 → P1 → P2 → P3）
```powershell
# 全系统恢复
.\scripts\disaster-recovery.ps1 -All
```

3. **逐模块验证**
4. **端到端测试**

**预期 RTO**：4-24 小时

## 5. 恢复演练计划

### 5.1 演练频率

| 演练类型 | 频率 | 参与方 |
|----------|------|--------|
| 单模块恢复演练 | 每月一次 | 模块负责人 |
| 多模块恢复演练 | 每季度一次 | 全体技术团队 |
| 全系统灾难恢复演练 | 每半年一次 | 全体技术团队 + 管理层 |
| 备份验证 | 每周一次 | 运维团队 |

### 5.2 演练内容

#### 月度演练（单模块）

- 模拟单个模块数据库损坏
- 使用备份进行恢复
- 验证恢复后数据完整性
- 记录恢复时间和问题

#### 季度演练（多模块）

- 模拟多个模块同时故障
- 验证恢复优先级和依赖关系
- 测试批量恢复流程
- 评估 RTO 达标情况

#### 半年演练（全系统）

- 模拟全系统故障
- 完整执行灾难恢复流程
- 验证所有模块恢复顺序
- 端到端功能验证
- 更新灾难恢复文档

### 5.3 演练记录

每次演练后需记录：

- 演练日期和参与人员
- 演练场景和范围
- 恢复时间（实际 RTO）
- 数据丢失量（实际 RPO）
- 遇到的问题和解决方案
- 改进建议

### 5.4 演练验收标准

- 所有模块恢复成功率 100%
- 数据完整性校验通过率 100%
- RTO 达标率 ≥ 95%
- 恢复流程顺畅，无阻塞性问题
- 文档完整且可操作

## 6. 恢复验证标准

### 6.1 备份文件验证标准

| 检查项 | 标准 | 说明 |
|--------|------|------|
| 文件存在性 | 通过 | 备份文件必须存在 |
| 文件大小 | > 0 | 文件大小必须大于 0 |
| SHA-256 校验和 | 匹配 | 与元数据中的校验和一致 |
| gzip 有效性 | 通过 | 压缩文件可正常解压 |
| 加密文件结构 | 通过 | 加密文件结构完整 |
| SQLite 完整性 | ok | PRAGMA integrity_check = ok |
| SQLite 快速检查 | ok | PRAGMA quick_check = ok |
| 表数量 | > 0 | 数据库中包含表 |

### 6.2 恢复操作验证标准

| 检查项 | 标准 |
|--------|------|
| 恢复成功率 | 100% |
| 安全网创建 | 成功（恢复前自动创建） |
| 回滚机制 | 可用（恢复失败自动回滚） |
| 数据完整性 | 通过（恢复后 integrity_check = ok） |
| 表结构完整 | 表数量与备份时一致 |
| 数据一致性 | 关键数据校验通过 |

### 6.3 服务恢复验证标准

- 服务进程正常运行
- 健康检查接口返回正常
- 核心功能可用
- 无明显错误日志

## 8. 联系人与升级流程

### 8.1 联系人

| 角色 | 职责 | 响应时间 |
|------|------|----------|
| 运维负责人 | 备份系统运维、告警处理 | 工作时间 30 分钟 |
| 模块负责人 | 模块级故障恢复 | 工作时间 1 小时 |
| 技术负责人 | 重大灾难决策 | 24 小时 |

### 8.2 升级流程

```
备份失败
  ↓
一级（15分钟未恢复 → 通知模块负责人
  ↓
二级（1小时未恢复）→ 通知技术负责人
  ↓
三级（4小时未恢复）→ 通知管理层
```

### 8.3 严重程度定义

| 级别 | 定义 | 示例 |
|------|------|------|
| P1 轻微 | 单个非核心模块故障，不影响主流程 | M6 硬件外设备份失败 |
| P2 一般 | 单个核心模块故障，部分功能受影响 | M9 开发工坊数据库损坏 |
| P3 严重 | 多个核心模块故障，主要功能不可用 | M5 + M8 同时故障 |
| P4 紧急 | 全系统故障，业务中断 | 服务器宕机、数据中心故障 |

## 9. 工具与脚本

### 9.1 备份 CLI 工具

**位置**：`shared/data/data_layer/backup.py

**常用命令**：

```bash
# 备份所有模块
python shared/data/data_layer/backup.py backup --all

# 备份指定模块
python shared/data/data_layer/backup.py backup --module m9

# 增量备份
python shared/data/data_layer/backup.py backup --module m5 --type incremental

# 恢复备份
python shared/data/data_layer/backup.py restore --module m9 --backup-dir /path/to/backup

# 列出备份
python shared/data/data_layer/backup.py list --module m9

# 校验备份
python shared/data/data_layer/backup.py verify --backup-path /path/to/backup.db

# 清理旧备份
python shared/data/data_layer/backup.py clean --max-age 30

# 查看备份状态
python shared/data/data_layer/backup.py status
```

### 9.2 备份验证脚本

**位置**：`scripts/verify-backup.ps1`

**常用命令**：

```powershell
# 验证指定模块
.\scripts\verify-backup.ps1 -Module m9

# 验证所有模块
.\scripts\verify-backup.ps1 -All

# 验证并测试恢复
.\scripts\verify-backup.ps1 -All -TestRestore

# 自动验证模式（用于定时任务）
.\scripts\verify-backup.ps1 -All -AutoVerify -ReportPath C:\reports\verify.json
```

### 9.3 灾难恢复脚本

**位置**：`scripts/disaster-recovery.ps1

**常用命令**：

```powershell
# 检测故障模块
.\scripts\disaster-recovery.ps1 -Detect

# 恢复指定模块
.\scripts\disaster-recovery.ps1 -Module m9

# 恢复多个模块
.\scripts\disaster-recovery.ps1 -Modules m5,m9

# 全系统恢复
.\scripts\disaster-recovery.ps1 -All

# 演练场景
.\scripts\disaster-recovery.ps1 -Scenario single_module_corruption

# 预览恢复计划
.\scripts\disaster-recovery.ps1 -Module m9 -DryRun
```
