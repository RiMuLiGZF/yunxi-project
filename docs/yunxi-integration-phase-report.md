## 二、当前系统架构

### 2.1 模块分层结构

| 层级 | 模块 | 端口 | 核心职责 |
|------|------|------|---------|
| 管控层 | **M8 控制塔** | 8000 | 模块注册/状态监控/代理转发/用户管理/权限控制 |
| 业务层 | **M4 场景引擎** | 8004 | 六大业务场景切换/上下文管理/场景识别/配置管理 |
| 业务层 | **M7 积木平台** | 3001 | 工作流编排/DAG引擎/模板管理/积木组件库 |
| 能力层 | **M1 Agent集群** | 8001 | 多Agent联邦调度/大模型路由/成本控制 |
| 能力层 | **M2 技能集群** | 8002 | 20+技能服务/技能发现/权限控制/流水线 |
| 能力层 | **M3 端云协同** | 8003 | 边缘计算/云端调度/数据同步/离线代理 |
| 能力层 | **M5 潮汐记忆** | 8005 | 长期记忆存储/语义检索/三级域权限/密级标记 |
| 能力层 | **M6 硬件外设** | 8006 | 6类设备管理/传感器数据采集/实时推送 |

### 2.2 关键架构约束

- **M8 不承载业务逻辑** — 仅做管控、代理、观测，业务能力全部下沉到各业务模块
- **M4 完全独立** — 有自己的数据库、API、鉴权机制，不依赖 M8 即可运行
- **统一鉴权协议** — 所有模块使用 X-M8-Token 头部进行模块间通信
- **全局配置中心** — `config/yunxi.env` 为唯一 Token 配置源，模块配置可覆盖
- **故障隔离** — 任一模块故障不影响其他模块，M8 代理自动降级

## 三、本次整合重点工作

### 3.1 Token 统一治理

**问题：** 各模块使用各自的默认 admin token，导致 M8 代理转发时鉴权失败。

**解决方案：** 建立全局配置文件 `config/yunxi.env`，所有模块启动时先加载全局配置，再加载自身配置。

已修复模块：M4 / M5 / M6 / M7 / M8（M1/M2/M3 原本已支持）

### 3.2 M8 启动脚本修复

**问题：** M8 使用 uvicorn 直接启动时存在相对 import 路径问题，无法正常加载 backend 模块。

**解决方案：** 新增 `M8-control-tower/server.py`，配置正确的 sys.path 后再导入 app。

### 3.3 M8 代理端点完善
