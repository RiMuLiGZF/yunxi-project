# 云汐系统多 Agent 架构说明：M1 接入时机与场景指南

> 本文档说明云汐系统中 M1（多 Agent 调度中心）在什么阶段、什么场景下需要被接入和使用，帮助开发者和架构师理解系统各模块的分工与协作关系。

---

## 一、整体架构总览

云汐系统是一套 **8 模块协同的智能系统**，各模块分工明确：

```
                    ┌─────────────────────┐
                    │   M8 管理工作台      │  ← 管控中心、统一入口
                    └─────────┬───────────┘
                              │
                    ┌─────────▼───────────┐
                    │   M1 多Agent调度中心  │  ← 大脑、推理调度、唯一Agent层
                    └─────────┬───────────┘
                              │
        ┌───────────┬─────────┼─────────┬───────────┐
        ▼           ▼         ▼         ▼           ▼
    ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
    │ M2   │   │ M3   │   │ M4   │   │ M5   │   │ M6   │
    │技能集 │   │端云协 │   │场景引 │   │潮汐记 │   │硬件外 │
    │  群  │   │  同  │   │  擎  │   │  忆  │   │  设  │
    └──────┘   └──────┘   └──────┘   └──────┘   └──────┘
                              │
                        ┌─────▼──────┐
                        │    M7      │  ← 积木编排平台
                        │  积木平台   │    （可视化工作流）
                        └────────────┘
```

**核心原则**：
- **M1 是唯一的 Agent 层** — 所有 AI 推理、Agent 调度都在 M1 完成
- **其他模块是能力模块** — 被 M1 调用，增强 M1 的能力边界
- **M8 是管控层** — 不做业务推理，负责管理、监控、调度

---

## 二、M1 中的 8 个子 Agent

M1（多 Agent 调度中心）内置了 8 个子 Agent，组成联邦调度系统：

| 子 Agent | 职责 | 触发场景 |
|----------|------|----------|
| **Master 调度 Agent** | 总控调度、任务分发、负载均衡 | 所有请求的入口，决定路由方向 |
| **Orchestrator 编排 Agent** | 多步任务编排、流程控制 | 需要多步推理、多轮交互的复杂任务 |
| **Intent 意图识别 Agent** | 用户意图分类、任务理解 | 所有用户输入的第一步，理解用户想做什么 |
| **Router 路由 Agent** | 模型/Agent 路由选择、成本优化 | 需要选择最优模型或 Agent 处理时 |
| **Budget 成本管控 Agent** | Token 成本监控、预算管理 | 所有调用，实时计算成本和用量 |
| **Federation 联邦对比 Agent** | 多模型对比、择优输出 | 需要高质量输出、多模型对比验证时 |
| **Guardrails 安全审核 Agent** | 内容安全审核、合规检查 | 输入输出的安全过滤 |
| **Memory 记忆 Agent** | 对话记忆管理、上下文管理 | 需要历史上下文、长期记忆时 |

## 三、各阶段 M1 接入时机

### 阶段 1：系统启动初始化

**是否需要接入 M1**：✅ 是

**接入内容**：
- M1 自身启动，加载配置，初始化 8 个子 Agent
- 注册联邦调度框架，建立消息总线
- 加载 Agent 注册表（内置 + 外部）

**说明**：M1 是系统的大脑，必须最先启动，其他模块才能通过它进行 AI 推理。

---

### 阶段 2：简单对话问答

**是否需要接入 M1**：⚠️ 不一定

**场景分析**：

| 场景 | 是否用 M1 | 原因 |
|------|-----------|------|
| 单轮简单问答（如"今天天气怎么样"） | ❌ 不需要 | 直接调用大模型 API 即可，M1 调度反而增加延迟 |
| 多轮对话（需要上下文记忆） | ✅ 需要 | M1 的 Memory Agent 管理对话上下文 |
| 需要意图识别的问答 | ✅ 需要 | M1 的 Intent Agent 理解意图后路由到正确处理逻辑 |

**建议**：
- 简单单轮问答 → M8 直接调用大模型（当前已实现的 fallback 模式）
- 复杂对话、需要上下文 → 走 M1 调度

---

### 阶段 3：复杂任务处理

**是否需要接入 M1**：✅ **必须接入**

**典型场景**：
- 多步骤推理任务（如"分析这个问题，给出三个方案，对比优缺点"）
- 需要工具调用的任务（如"查询数据库并生成报告"）
- 需要多模型协作的任务（如"先用代码模型写代码，再用通用模型写文档"）
- 需要成本优化的任务（如"用最便宜的模型完成这个简单任务"）

**M1 在此阶段发挥的作用**：
1. **任务拆解** — Orchestrator Agent 将复杂任务拆成多个子任务
2. **智能路由** — Router Agent 为每个子任务选择最合适的模型/Agent
3. **成本管控** — Budget Agent 全程监控 Token 消耗
4. **结果整合** — Federation Agent 整合多个子任务的输出
5. **安全审核** — Guardrails Agent 对输入输出做安全检查

---

### 阶段 4：调用其他模块能力

**是否需要接入 M1**：✅ **必须接入**

**M2 技能集群**：
- 当任务需要调用具体技能（如"帮我写一份周报"、"翻译这段英文"）
- M1 识别意图 → 查找匹配的技能 → 调用 M2 技能执行 → 返回结果

**M3 端云协同**：
- 当任务需要本地模型处理（如隐私数据不能上云）
- M1 的 InferenceRouter 判断 → 云端/本地路由决策 → M3 执行本地推理

**M4 场景引擎**：
- 当用户进入特定场景模式（如编程模式、心流模式、设计模式）
- M1 根据场景模式调整 Agent 行为、提示词、模型选择

**M5 潮汐记忆**：
- 当任务需要长期记忆（如"上次我们讨论的那个方案怎么样了"）
- M1 的 Memory Agent → 调用 M5 潮汐记忆系统 → 检索相关记忆 → 注入上下文

**M6 硬件外设**：
- 当任务需要硬件数据（如"我现在的心率是多少"、"分析今天的运动数据"）
- M1 识别硬件相关意图 → 调用 M6 获取数据 → 基于数据生成回复

**M7 积木平台**：
- 当任务需要执行预定义工作流（如"按照入职流程给新员工发欢迎邮件"）
- M1 识别工作流意图 → 调用 M7 执行工作流 → 返回执行结果

---

### 阶段 5：多用户并发场景

**是否需要接入 M1**：✅ **必须接入**

**M1 在此阶段发挥的作用**：
1. **负载均衡** — Master Agent 将请求分发到不同的模型/Agent 实例
2. **队列管理** — TaskDispatcher 管理任务队列，防止系统过载
3. **并发控制** — 限制最大并发数，保护后端服务
4. **优先级调度** — 高优先级任务插队处理

---

### 阶段 6：系统运维与监控

**是否需要接入 M1**：⚠️ 间接接入

**说明**：
- M8 管理台负责系统监控和运维
- M1 的运行状态、性能指标通过 M8 展示
- 但 M1 自身的调度决策不需要 M8 干预
- M8 可以配置 M1 的参数（如模型选择策略、预算阈值）

---

## 四、各模块与 M1 的协作关系总结

| 模块 | 是否需要 M1 | 协作方式 | 触发时机 |
|------|------------|----------|----------|
| **M8 管理台** | ✅ 是 | M8 提交任务给 M1，M1 处理后返回结果 | 用户在管理台提交任务 |
| **M2 技能集群** | ✅ 是 | M1 调用 M2 的技能执行具体任务 | 意图识别匹配到具体技能时 |
| **M3 端云协同** | ✅ 是 | M1 的推理路由决策后，调用 M3 执行本地推理 | 需要本地模型处理隐私数据时 |
| **M4 场景引擎** | ✅ 是 | M4 切换场景时，通知 M1 调整 Agent 配置 | 用户切换场景模式时 |
| **M5 潮汐记忆** | ✅ 是 | M1 的 Memory Agent 读写 M5 的记忆存储 | 需要长期记忆、上下文检索时 |
| **M6 硬件外设** | ✅ 是 | M1 调用 M6 获取硬件数据，基于数据生成回复 | 需要传感器/穿戴设备数据时 |
| **M7 积木平台** | ✅ 是 | M1 调用 M7 执行工作流，获取执行结果 | 需要执行预定义工作流时 |

## 五、当前系统的调用链路

### 当前实现（MVP 阶段）

```
用户请求
    │
    ▼
M8 管理台
    │
    ├─→ 尝试调用 M1
    │      │
    │      ├─ M1 能处理（匹配到 Agent/技能）→ M1 处理 → 返回结果
    │      │
    │      └─ M1 返回 fallback（无匹配）→ M8 直接调用大模型（fallback）
    │
    └─ M1 不可用 → M8 直接调用大模型
```

### 未来目标（完整形态）

```
用户请求
    │
    ▼
M8 管理台
    │
    ▼
M1 调度中心
    ├─ 意图识别 → 理解用户想做什么
    ├─ 记忆检索 → 加载相关上下文
    ├─ 任务拆解 → 复杂任务拆分成子任务
    ├─ 路由选择 → 选择最优模型/Agent/模块
    │
    ├─→ M2 技能集群（调用具体技能）
    ├─→ M3 端云协同（本地模型推理）
    ├─→ M5 潮汐记忆（读写记忆）
    ├─→ M6 硬件外设（获取硬件数据）
    ├─→ M7 积木平台（执行工作流）
    │
    ├─ 结果整合 → 合并多个模块的输出
    ├─ 安全审核 → 检查输出合规性
    └─ 成本统计 → 记录本次调用成本
    │
    ▼
返回最终结果
```

## 六、接入 M1 的判断标准

当你不确定是否需要接入 M1 时，问自己以下问题：

| 问题 | 回答"是"则需要 M1 |
|------|-------------------|
| 任务需要多步推理或多轮对话吗？ | ✅ 需要 |
| 需要调用其他模块的能力吗？ | ✅ 需要 |
| 需要管理对话上下文或长期记忆吗？ | ✅ 需要 |
| 需要控制成本或选择最优模型吗？ | ✅ 需要 |
| 需要内容安全审核吗？ | ✅ 需要 |
| 是简单的单轮问答，直接问大模型就行？ | ❌ 不需要，直接调 API 更高效 |

## 七、下一步接入建议

### 短期（MVP 优化）

1. ✅ **M8 直接调用大模型**（已完成）— 保证基础对话能力
2. 🔄 **M1 注册基础 Agent** — 在 M1 中注册通用对话 Agent，让 M1 能处理简单对话而不走 fallback
3. 🔄 **M1 ↔ M5 记忆对接** — 让 M1 能读写 M5 的潮汐记忆

### 中期（能力扩展）

1. **M1 ↔ M2 技能对接** — M1 能发现和调用 M2 的技能
2. **M1 ↔ M7 工作流对接** — M1 能触发 M7 的工作流执行
3. **M1 ↔ M4 场景对接** — 场景切换时调整 M1 的 Agent 配置

### 长期（完整形态）

1. **M1 ↔ M3 端云协同** — 本地/云端智能路由
2. **M1 ↔ M6 硬件对接** — 硬件数据驱动的 AI 决策
3. **联邦调度优化** — 多模型对比、自动择优、成本最优

---

## 八、总结

**M1 是系统的大脑，不是简单的大模型 API 包装层。**

- **简单问答** → 直接调用大模型，快、省
- **复杂任务** → 必须走 M1，用多 Agent 协作处理
- **调用其他模块** → 必须走 M1，M1 是统一调度入口
- **多用户并发** → 必须走 M1，M1 负责负载均衡和队列管理

**一句话**：只要任务超出了"一问一答"的范畴，就应该接入 M1。
