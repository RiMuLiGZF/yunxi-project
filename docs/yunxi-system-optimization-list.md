# 云汐系统优化升级清单

**日期**：2026-07-08
**版本**：v1.1.0
**说明**：本清单列出所有需要优化和升级的项目，按优先级排序，由人工选择执行顺序。

---

## 一、🔴 P0 级（严重问题，优先修复）

### 1. M9 缺少 M8 标准对接接口
- **问题**：M9 编程开发模块没有实现 /m8/health, /m8/metrics, /m8/config 三个标准接口，无法被 M8 管理台统一纳管
- **影响**：M9 无法在 M8 部署中心显示和管理，状态无法监控
- **建议方案**：参考 M10 的实现，添加 M8 标准接口和 Token 鉴权中间件
- **工作量**：约 4-6 小时
- **依赖**：无

### 2. M9 未注册到 ModuleRegistry
- **问题**：shared/module_client.py 的 ModuleRegistry 中没有 M9 的注册信息
- **影响**：M8 管理台的模块列表中看不到 M9
- **建议方案**：在 _register_default_modules() 中添加 M9 配置
- **工作量**：约 0.5 小时
- **依赖**：需先完成 M8 标准接口

### 3. 全局配置缺少 M9 配置项
- **问题**：config/yunxi.env 中没有 M9 的端口、host、token、base_url 配置
- **影响**：M9 配置不统一，模块间调用可能失败
- **建议方案**：在 yunxi.env 中添加 M9 完整配置项
- **工作量**：约 0.5 小时
- **依赖**：无

### 4. M8 端口配置不一致
- **问题**：config.py 默认 port=8080，server.py 读取 M8_PORT 环境变量默认 8000，yunxi.env 中配置的是 M8_BACKEND_PORT=8000，变量名不匹配
- **影响**：启动日志显示 "started on 0.0.0.0:8080" 但实际绑定 8000，造成困惑
- **建议方案**：统一配置变量名为 M8_PORT 或 M8_BACKEND_PORT，保持一致
- **工作量**：约 1 小时
- **依赖**：无

## 二、🟡 P1 级（重要问题，近期修复）

- **建议方案**：每个模块至少补充 5-10 个核心功能测试
- **工作量**：每个模块约 2-3 天
- **依赖**：无

- **依赖**：无

- **依赖**：无

### 8. M4 缺少 README 和架构文档

- **工作量**：约 6-8 小时
- **依赖**：无

### 10. M10 缺少 README 和架构文档
- **问题**：M10 代码完整但没有文档
- **影响**：新成员上手困难

### 11. M8 /health 端点重复定义

## 三、🟢 P2 级（改进建议，有空再做）

### 13. 各模块统一 README 文档
- **问题**：M2/M6/M7/M8/M9/M10 都缺少 README
- **影响**：项目整体可维护性差
- **建议方案**：为每个模块补充 README.md，包含模块介绍、快速开始、配置说明、API 概览
- **工作量**：每个模块约 2-4 小时
- **依赖**：无

### 14. M8 测试文件标准化
- **问题**：9个测试/验证脚本散落在 backend/ 和根目录，未统一到 tests/，部分是验证脚本而非 pytest 测试
- **影响**：测试难以管理和自动化执行
- **建议方案**：将所有测试迁移到 tests/ 目录，改用 pytest 标准格式
- **工作量**：约 2-3 天
- **依赖**：无

### 15. M1 API 入口统一
- **问题**：M1 的 API 分散在 api/server.py, api_server.py, api/m8_interface.py 三个文件中
- **影响**：路由管理混乱，可能有重复
- **建议方案**：统一到一个入口文件，按功能分路由模块
- **工作量**：约 1-2 天
- **依赖**：无

### 16. M1 清理旧版本编排器
- **问题**：orchestrator_v2.py ~ orchestrator_v9.py 多个版本并存
- **影响**：代码冗余，维护成本高
- **建议方案**：保留最新版本，删除旧版本（或移到 archive/ 目录）
- **工作量**：约 4-6 小时
- **依赖**：无

### 17. M2 启动脚本命名统一

- **影响**：不统一，自动化脚本需要特殊处理
- **建议方案**：重命名为 server.py 或提供别名
- **工作量**：约 0.5 小时
- **依赖**：无

### 18. M3 API 版本策略明确
- **问题**：同时存在 /api/v1/* 和 /api/v3/* 两套路径
- **影响**：接口使用混乱，不知道该用哪个

- **工作量**：约 2-3 小时
- **依赖**：无

### 19. M4/M5 /m8/metrics 接入真实数据
- **问题**：M4 和 M5 的 /m8/metrics 返回硬编码假数据
- **影响**：M8 监控中心看到的指标不准确
- **建议方案**：接入 psutil 等库，采集真实系统指标

- **依赖**：无

- **影响**：扩展性差，并发能力有限，不支持复杂查询
- **建议方案**：改用 SQLite 或其他数据库存储

- **依赖**：无

- **影响**：不确定哪个是在用的，可能数据不一致
- **建议方案**：统一到一个位置，清理冗余文件
- **工作量**：约 1-2 小时
- **依赖**：无

### 22. 前端增加 M6/M9/M10 页面
- **问题**：frontend/ 中只有 M7 和 M8 的前端页面，M6/M9/M10 没有
- **影响**：这些模块没有可视化管理界面
- **建议方案**：为 M6/M9/M10 增加管理前端页面
- **工作量**：每个模块约 3-5 天
- **依赖**：无

### 23. 前端清理备份文件
- **问题**：frontend/m8/ 下有 agents.html.bak.v2 和 dashboard.html.bak 等备份文件
- **影响**：代码库不整洁，可能引起混淆
- **建议方案**：删除或纳入版本管理策略

- **依赖**：无

### 24. shared/ 增加单元测试
- **问题**：公共组件是整个系统的基础，但没有单元测试
- **影响**：公共组件的 bug 会影响所有模块
- **建议方案**：为 config/logger/module_client/process_manager 增加测试

### 25. shared/ 增加统一异常定义
- **问题**：缺少 errors.py 或 exceptions.py，各模块自己定义异常
- **影响**：错误处理不统一，模块间异常传递混乱
