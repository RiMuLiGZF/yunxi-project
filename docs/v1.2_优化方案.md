## 一、核验结论总览

基于对调查报告的逐一核验，8 个核心问题的确认结果如下：

| # | 问题 | 核验结果 | 严重程度 | 优先级 |
|---|------|---------|---------|--------|
| 1 | M12 M8 标准接口接入存疑 | 部分存在（M12 实现了三接口，M8 运维调用路径不统一） | 中 | P1 |

## 二、P0 级问题修复方案

### P0-1：M10 Token 空值绕过 M8 接口鉴权

**问题描述**：`M10-system-guard/server.py` 中 `_verify_m8_token()` 函数在 `M10_ADMIN_TOKEN` 环境变量未配置时返回 `True`（直接放行），而业务接口的 `verify_token()` 行为相反（空配置时拒绝）。安全默认值搞反。

**影响范围**：M10 的 `/m8/health`、`/m8/metrics`、`/m8/config` 三个标准接口。

**修复方案**：

1. **统一安全默认值**：修改 `_verify_m8_token()`，token 未配置时返回 `False`（拒绝），与业务接口鉴权逻辑保持一致
2. **开发环境兼容**：新增 `M10_DEV_MODE` 环境变量，仅在显式开启开发模式时允许空 token 访问
3. **启动告警**：token 未配置时在启动日志中输出 WARNING 级别告警
4. **补充测试**：增加 token 空值、token 错误、token 正确三种场景的测试用例

**修改文件**：
- `M10-system-guard/server.py` — 修复 `_verify_m8_token()` 逻辑
- `M10-system-guard/tests/test_m8_auth.py` — 新增鉴权测试

**验收标准**：
- 未配置 token 时，M8 三接口返回 401
- 配置错误 token 时，返回 401
- 配置正确 token 时，正常访问
- `M10_DEV_MODE=true` 时，空 token 可访问（仅开发环境）
- 启动日志有明确告警信息

**预估工作量**：0.5 人天

### P0-2：M4 用户 ID 全面硬编码为 "default"

**问题描述**：M4 场景引擎中用户 ID 从 schema → router → service 各层均硬编码为 "default"，多处标注 TODO 说明为临时方案。单用户场景下可工作，但完全没有多用户隔离能力。

**影响范围**：M4 全模块，涉及 10+ 路由文件、全部 service 层、schema 层。

**修复方案**：

**第一阶段：建立用户上下文传递机制（基础设施）**

1. **认证中间件**：新增 `auth_middleware.py`，从请求头 `X-User-ID` 或 JWT Token 中提取用户 ID，存入 `contextvars.ContextVar`
2. **用户上下文工具**：`src/common/user_context.py`，提供 `get_current_user_id()` 函数，从 ContextVar 读取
3. **Schema 改造**：移除 `user_id: str = "default"` 的默认值，改为可选字段，未传时从上下文获取
4. **Service 层改造**：方法签名中 `user_id: str = "default"` 改为 `user_id: str | None = None`，内部调用 `get_current_user_id()` 兜底

**第二阶段：路由层改造**

5. **路由参数清理**：移除所有 `Query("default")`、`Header("default")` 的用户 ID 默认值
6. **场景智能模块**：修复 `scene_intelligence.py` 中 5 处直接硬编码，改为从上下文获取
7. **各 mode 路由**：逐一改造 8 大场景模式的路由文件

**第三阶段：向后兼容与测试**

8. **默认用户兼容**：如果请求中完全没有用户标识，自动分配 "default" 作为兼容默认值（保证现有调用不中断）
9. **补充测试**：增加多用户场景隔离测试，验证不同用户数据互不干扰

**修改文件**：
- `src/common/user_context.py` — 新增，用户上下文管理
- `src/middlewares/auth_middleware.py` — 新增，认证中间件
- `src/schemas/common.py` — 修改，移除默认值
- `src/routers/*.py` — 10+ 个路由文件，逐一改造
- `src/modes/*/router.py` — 8 大场景路由改造
- `src/services/*.py` — service 层改造
- `tests/test_multi_user.py` — 新增多用户隔离测试

**验收标准**：
- 请求头携带 `X-User-ID: user_a` 时，所有操作使用 user_a 的数据
- 请求头携带 `X-User-ID: user_b` 时，所有操作使用 user_b 的数据
- 两个用户的数据完全隔离（上下文、场景状态、记忆等）
- 不带用户 ID 的请求自动降级为 "default" 用户，保持向后兼容
- 核心路由测试用例 ≥ 20 个

**预估工作量**：3-4 人天

## 三、P1 级质量债务方案

### P1-1：版本号统一治理

**问题描述**：各模块版本号不统一，M2（1.2.0 vs 3.10.2）、M11（0.1.0 / 0.5.0 / 1.2.0 三个版本）、M5（内置 0.5.2）、M8（内置 1.0.0）等多处不一致。

**修复方案**：

1. **确立权威来源**：以 `shared.core.version.SYSTEM_VERSION` 为系统版本唯一真值源
2. **双轨清晰化**：
   - `__version__` = 系统版本（对外暴露，如 "1.2.0"）
   - `INTERNAL_VERSION` = 模块内部迭代版本（模块内部使用，如 M2 的 "3.10.2"）
3. **逐模块清理**：
   - M2：`__version__` 保持 1.2.0，`__version_info__` 重命名为 `INTERNAL_VERSION_INFO`
   - M3：统一各处版本号为 1.2.0，内部迭代版本单独存放
   - M5：内置版本从 0.5.2 升级到 1.2.0，优先从 shared 读取
   - M8：`backend/__init__.py` 和 `config.py` 中的版本号统一为 1.2.0
   - M11：README 更新到 1.2.0，代码中 0.1.0 更新为 1.2.0
   - M12：`config.py` 中的 version 从 1.0.0 更新为 1.2.0
4. **版本检查脚本**：新增 `scripts/version_check.py`，自动扫描所有模块的版本号并报告不一致
5. **CI 集成**：在测试流程中加入版本一致性检查

**修改文件**：
- `M2-skills-cluster/skill_cluster/version.py` — 重命名变量
- `M5-tide-memory/src/tide_memory/core/version.py` — 更新默认值
- `M8-control-tower/backend/__init__.py` — 更新版本
- `M8-control-tower/backend/config.py` — 更新版本
- `M11-mcp-bus/README.md` — 更新版本
- `M11-mcp-bus/src/main.py` — 更新版本
- `M11-mcp-bus/src/models.py` — 更新版本
- `M11-mcp-bus/src/routers/health.py` — 更新版本
- `M12-security-shield/backend/config.py` — 更新版本
- `scripts/version_check.py` — 新增检查脚本

**验收标准**：
- 运行 `python scripts/version_check.py` 返回 0 告警
- 所有模块对外暴露的 `__version__` 均为 "1.2.0"
- README 版本与代码版本一致
- 模块内部版本号通过 `INTERNAL_VERSION` 变量独立管理

**预估工作量**：1 人天

### P1-2：异常处理精细化

**问题描述**：仅 M5、M8、shared 三个模块就有 1157 处 `except Exception`，大量宽泛的异常捕获会掩盖真实问题，不利于调试和错误定位。

**修复方案（分阶段推进，先试点）**：

**第一阶段：M8 试点（except Exception 最多的模块，497 处）**

1. **分类梳理**：将 M8 中的 `except Exception` 分为三类：
   - **A 类：预期内异常**（文件不存在、网络超时、配置缺失等）→ 替换为具体异常类型，保留 `logger.info` 或 `logger.debug`
   - **B 类：容错兜底**（外层大 try 包裹整段逻辑）→ 补充 `logger.exception` 记录完整堆栈，增加错误计数指标
   - **C 类：空 except**（只捕获不处理）→ 至少补充 `logger.debug` 记录，避免静默失败
2. **M8 重点文件**：`routers/modules.py`（62 处）、`services/health_service.py`（19 处）优先改造

**第二阶段：建立规范**

3. **异常类型扩展**：在 `shared.core.errors` 中补充常用业务异常类型（配置错误、依赖不可用、超时、数据不存在等）
4. **异常处理规范文档**：在 `docs/` 下新增异常处理指南，明确：
   - 什么时候可以用 `except Exception`（仅最外层容错）
   - 预期内异常必须捕获具体类型
   - 所有异常捕获必须有日志记录
   - 错误返回必须带错误码和 trace_id

**第三阶段：M5 跟进**

5. **M5 改造**：215 处 `except Exception`，同样按 A/B/C 三类处理

**验收标准**：
- M8 的 `except Exception` 数量减少 30%（从 497 → ≤ 350）
- M5 的 `except Exception` 数量减少 30%（从 215 → ≤ 150）
- 剩余的 `except Exception` 全部有对应的 `logger.exception` 或 `logger.error` 记录
- shared.core.errors 中新增 ≥ 10 种标准业务异常类型

**预估工作量**：2-3 人天（试点 + 规范 + M5）

### P1-3：M8 运维监控调用路径统一

**问题描述**：M12 已规范实现了 M8 三接口，但 M8 的 `ops_status_aggregator.py` 在调用模块健康检查时使用通用 `/health` 路径，而不是 M8 标准的 `/m8/health`。M8 自身调用路径不统一。

**修复方案**：

1. **统一调用路径**：修改 `ops_status_aggregator.py` 中的健康检查调用，全部使用 `/m8/health` 标准路径
2. **兼容降级**：如果 `/m8/health` 返回 404，降级尝试 `/health`（兼容尚未完全迁移的模块）
3. **调用规范确认**：检查 M8 中其他调用模块的地方（如 `startup_check_agent.py` 已使用标准路径），确保全部统一
4. **M7 metrics 对齐**：M7 的 `/m8/metrics` 数据与 `/api/v1/admin/metrics` 不一致问题，补齐 M8 标准接口的指标字段（CPU、内存、平均响应时间、错误率）

**修改文件**：
- `M8-control-tower/backend/services/ops_status_aggregator.py` — 统一调用路径
- `M7-workflow-engine/backend/m8_api.py` — 补齐 metrics 字段
- `M7-workflow-engine/tests/test_m8_metrics.py` — 新增指标对齐测试

**验收标准**：
- M8 运维仪表盘能正常显示所有模块的健康状态
- M7 的 `/m8/metrics` 返回的指标字段与 `/api/v1/admin/metrics` 一致
- 调用失败时有明确的降级逻辑和日志记录

**预估工作量**：0.5 人天

## 四、P2 级体验优化方案

### P2-1：M8 已迁移路由清理

**问题描述**：M8 中有约 15 个路由文件处于"代理占位"状态，业务已迁移到对应模块但本地保留回退代码。

**修复方案**：

1. **评估稳定性**：确认已迁移模块（M4/M5/M6/M7/M9 等）运行稳定，无频繁回退需求
2. **清理代理路由**：移除稳定迁移超过 1 个月的路由文件，仅保留 1-2 个作为代理模式示例
3. **归档保留**：清理的文件移动到 `_archive/m8_migrated/` 目录
4. **文档更新**：在架构文档中记录迁移历史和回滚路径

**预估工作量**：1 人天

---

### P2-2：M6 硬件从模拟走向半真实

**问题描述**：M6 硬件外设全为模拟器，设备驱动文件仅 500-900 字节，服务层单薄。

**修复方案**：

1. **硬件驱动抽象层**：建立 `BaseDeviceDriver` 抽象基类，定义标准接口（连接/断开/读取数据/发送指令）
2. **模拟驱动重构**：现有模拟器改造为 `SimulatedWatchDriver`，继承抽象基类
3. **真实驱动实现（选一种）**：优先实现 BLE 智能手表驱动（如小米手环/华为手表的 BLE 协议），或串口设备驱动
4. **设备数据真实化**：从纯随机生成改为"基线 + 随机波动 + 真实生理节律"模式
5. **连接状态管理**：新增在线/离线/重连状态机

**预估工作量**：2-3 人天

---

### P2-3：shared 库瘦身第一步

**问题描述**：shared 库近 10 万行代码，business 层包含大量应拆分到各模块的业务逻辑，根目录与三层结构有大量重复文件。

**修复方案**：

1. **审计分类**：审计 `shared/business/` 下所有文件，标记为"通用能力"和"特定模块业务"两类
2. **回迁试点**：选取 2-3 个明显属于特定模块的文件迁回对应模块（如模块自注册 → M8，语音合成工具 → M1/M4）
3. **根目录冗余清理**：清理根目录中与 core/data/business 三层重复的文件，通过 re-export 保持向后兼容
4. **准入标准文档**：新增 `shared/README.md`，明确什么能进 shared（跨模块使用的通用能力），什么应该在模块内

**预估工作量**：2 人天

---

### P2-4：M2 缓存命中率与技能加载优化

**问题描述**：M2 缓存命中率指标始终为 0，部分技能未在启动时自动加载。

**修复方案**：

1. **缓存命中率排查**：检查缓存 key 生成逻辑、缓存读写时机，修复命中率为 0 的问题
2. **启动自动加载**：将内置技能改为启动时自动注册到技能注册中心，而非首次调用时才加载
3. **预热机制**：新增技能预热接口，支持提前加载高频使用的技能

**预估工作量**：0.5 人天

## 七、风险与注意事项
