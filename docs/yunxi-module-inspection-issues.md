## 二、P0 级严重问题（必须修复）

### 1. M4 测试与实现严重不匹配
- **模块**：M4 场景引擎
- **问题**：测试文件引用了大量不存在的方法（memory_decay、_apply_memory_decay、_get_dynamic_threshold 等），测试完全无法通过
- **影响**：测试形同虚设，无法验证功能正确性

### 2. M5 get_memory / delete_memory 是占位符
- **模块**：M5 潮汐记忆
- **问题**：`GET /api/v1/memory/{id}` 和 `DELETE /api/v1/memory/{id}` 返回固定假数据，不操作存储层
- **影响**：核心 CRUD 功能缺失，无法单条查询和删除记忆

### 3. M6/M7 目录命名混乱
- **模块**：M6、M7
- **问题**：各存在两个目录（空壳+真代码），容易混淆
  - M6: `M6-wearable-hardware/`（空壳） vs `M6-hardware-peripheral/`（真代码）
  - M7: `M7-block-orchestration/`（空壳） vs `M7-workflow-builder/`（真代码）
- **影响**：shared/module_client 和 process_manager 中的目录配置可能指向错误

### 4. shared/process_manager.py 缺少 M9
- **模块**：shared
- **问题**：进程管理器中硬编码的模块列表缺少 M9，无法通过进程管理启动 M9
- **影响**：M9 无法被 M8 统一启动和管理

## 三、P1 级重要问题（建议修复）

### 共性问题

#### 5. M8 metrics 普遍硬编码
- **影响模块**：M2、M3、M4、M5、M6、M8
- **问题**：各模块的 `/m8/metrics` 接口中，CPU、内存、请求数等指标大量硬编码为 0
- **影响**：M8 监控中心无法获取真实的模块运行指标

#### 6. M8 config 普遍硬编码
- **影响模块**：M2、M3、M4、M5
- **问题**：`/m8/config` 接口返回硬编码的配置信息，未接入实际的 ConfigManager
- **影响**：无法通过 M8 查看和管理各模块配置

#### 7. 缺少 README 文档
- **影响模块**：M1、M2、M3、M6、M7、M9、shared
- **问题**：7个模块没有 README.md，新手上手困难

#### 8. 缺少 .env.example 配置模板
- **影响模块**：M6、M7、M8、M9
- **问题**：没有配置示例文件，环境变量配置不透明

### 单模块问题

#### 9. M1 目录结构混乱
- 根目录平铺 80+ 个 py 文件，与子目录存在重复文件
- orchestrator_v2~v8 共7个历史版本文件未清理

#### 10. M2 M8 接口与启动脚本耦合
- 三个 `/m8/*` 接口直接写在 start_server.py 中，未封装到独立模块

#### 11. M3 /m8/* 接口未复用 m8_api 实现
- m8_api/ 目录下有完整的 health_endpoints.py、config_endpoints.py
- 但 `/m8/health`、`/m8/metrics`、`/m8/config` 三个标准路径返回硬编码简化数据

#### 12. M4 缺统一配置管理模块
- 没有 config.py，环境变量散落在各文件中
- database.py 定义了 5 张 ORM 表，但业务服务用内存/JSON 存储，未使用数据库

#### 13. M6 传感器数据无持久化
- 传感器历史数据仅内存存储，服务重启后丢失
- 完全没有测试文件
- 完全没有文档

#### 14. M7 存储层是 JSON 文件
- 工作流数据存在 ~/.yunxi/m7_workflows.json
- 无数据库，并发安全性差
- 缺少 config.py 配置管理类

#### 15. M9 代码执行无沙箱隔离
- sandbox_enabled 配置名不副实，直接在主机执行代码
- API 设计不规范，POST 接口用 query 参数而非 body
- 测试覆盖率极低（<15%）

#### 16. M10 README 与实际代码结构不符
- README 描述的是 src/main.py 分层结构
- 实际是扁平的 m10_system_guard/ 目录

#### 17. shared 零测试覆盖
- 作为公共基础组件，没有任何单元测试
- module_client.py 中异步方法用同步 time.sleep，会阻塞事件循环

#### 18. 前端设计系统不统一
- M8/M4/M7/Xian 四大板块各有独立 CSS 变量和色值
- 大量页面为静态原型，JS 业务逻辑不完整
- 亮色版和暗色版为独立 HTML 文件，维护成本翻倍

## 四、P2 级改进建议（16项）

1. M1 多入口文件职责不清（server.py、api_server.py、api/server.py）
2. M2 有 .bak 备份文件应清理（mcp_bridge.py.bak）
3. M3 server.py 过大（961行），应拆分
4. M3 API 路径版本混乱（/api/v1/、/api/v3/、/m8/ 三套前缀）
5. M4 错误处理薄弱，缺少统一错误码体系
6. M5 根目录缺 README（在 docs/ 下，不易发现）
7. M7 定时触发未实际调度
8. M7 条件分支未实现
9. M8 schemas 目录几乎为空
10. M8 错误处理风格不统一
11. M9 缺少 CORS 中间件
12. M9 VSCode 实例无持久化
13. shared logger 功能单薄（缺文件输出、结构化JSON、轮转）
14. 前端 JS 内嵌在 HTML 中，缺少模块化
15. 前端无构建工具，CDN 依赖外部服务
16. 前端无路由系统，页面跳转靠纯 HTML 链接
