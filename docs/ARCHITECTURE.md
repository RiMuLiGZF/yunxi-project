## 1. 系统定位

云汐系统是一套以大语言模型为核心、模块化微服务架构的 **AI 原生个人操作系统**。采用「潮汐记忆 + 多Agent协作 + 场景化服务」三层智能架构，覆盖工作、学习、生活、休闲等全场景智能交互。

### 1.1 设计目标

| 目标 | 说明 |
|------|------|
| **本地优先** | 核心数据和计算在本地完成，保护隐私 |
| **模块化** | 每个模块独立部署、独立演进，松耦合 |
| **可扩展** | 通过 MCP 协议和技能市场轻松扩展能力 |
| **高可用** | 模块故障不影响全局，支持降级运行 |
| **自进化** | 具备自我监控、自我优化、自我修复能力 |

### 1.2 适用场景

- 个人 AI 助手与知识管理
- 开发者效率工具与代码辅助
- 多设备协同的智能工作流
- 本地化的隐私敏感数据处理
- 可定制的 AI 原生应用开发平台

### 2.3 模块依赖关系

```
                    M0 主理人管控台
                        │ 纳管调用
                        ▼
                 M8 管理控制塔
            ┌──────────┼──────────┐
            │          │          │
     ┌──────▼───┐ ┌───▼────┐ ┌──▼─────┐
     │ M1 Agent │ │ M4 场景 │ │ M7 工作流│
     │  集群    │ │  引擎   │ │  编排   │
     └──────┬───┘ └───┬────┘ └──┬─────┘
            │         │         │
     ┌──────▼───┐ ┌───▼────┐ ┌──▼─────┐
     │ M2 技能  │ │ M5 记忆 │ │ M9 开发 │
     │  集群    │ │        │ │  工坊   │
     └──────────┘ └────────┘ └────────┘
            │         │         │
     ┌──────▼─────────▼─────────▼──────┐
     │   基础设施层：M3 / M10 / M11     │
     └───────────────┬─────────────────┘
                     │
            ┌────────▼────────┐
            │  M12 安全盾 (横切) │
            └─────────────────┘
                     │
            ┌────────▼────────┐
            │  M6 硬件外设 (横切)│
            └─────────────────┘
```

**核心依赖规则**：

1. **上层依赖下层**：管控层 → 业务层 → 智能层 → 基础设施层
2. **横切贯穿全栈**：M12 安全盾、M6 硬件外设以横切方式提供能力
3. **M8 是核心枢纽**：所有业务模块通过 M8 统一纳管和调度
4. **M11 是能力总线**：跨模块能力调用通过 MCP 总线完成
5. **避免循环依赖**：严格遵循分层依赖方向

## 3. 模块详细说明

### 3.1 M0 主理人管控台

**定位**：系统最高权限管控节点，仅对主理人（Owner）开放。

**核心能力**：
- 全局仪表盘：所有模块状态、系统资源、告警一览
- 模块管理：启停、配置、健康检查、指标查看
- 全局配置中心：系统级配置管理
- 审计日志：全系统操作审计
- 紧急操作中心：一键停机、紧急制动
- 主理人专属工具

**技术选型**：FastAPI + 纯 HTML/CSS/JS 前端（深色科技感主题）

**设计要点**：
- 复用 M8 JWT 认证体系，增加 Owner 角色校验
- M8 未启动时提供 fallback mock 数据，保证界面可用
- 所有操作均记录审计日志

---

### 3.2 M1 多 Agent 集群调度

**定位**：智能核心层的调度中心，负责 Agent 的联邦调度和任务路由。

**核心能力**：
- 8 个子 Agent 管理（代码、写作、分析、学习等）
- 16+ 外部 Agent Adapter 接入
- 联邦调度策略（成本最优、质量最优、混合模式）
- 智能路由与故障转移
- 成本控制与预算管理
- 隐私保护与数据脱敏
- Agent 生命周期管理

**技术选型**：FastAPI + 联邦调度引擎 + 适配器模式

**设计要点**：
- 支持多种调度策略：轮询、优先级、成本、质量
- 断路器模式：故障 Agent 自动熔断
- 渐进式降级：核心能力优先保障
- 实时流式输出（SSE）

---

### 3.3 M2 技能集群

**定位**：能力扩展层，提供可插拔的技能注册与执行。

**核心能力**：
- 18+ 内置技能（翻译、搜索、计算、总结等）
- 技能注册中心（注册/发现/路由/执行）
- MCP 协议接入（MCP 服务作为技能）
- A2A 协议接入（Agent-to-Agent）
- 技能市场（打包、发布、搜索、安装、评分）
- 技能版本管理

**技术选型**：FastAPI + 插件式架构 + MCP SDK

**设计要点**：
- 技能沙箱执行，隔离安全风险
- 技能间数据传递标准化
- 支持技能组合（技能链）
- 技能执行可观测（日志、指标）

---

### 3.4 M3 端云协同内核

**定位**：基础设施层，实现本地与云端的混合计算调度。

**核心能力**：
- 端云混合调度（任务路由决策）
- 离线降级模式（无网络也能用）
- 数据同步与冲突解决
- 隐私分级（本地/云端数据分类）
- 算力弹性扩展
- 网络状态监测与切换

**技术选型**：FastAPI + 状态机 + 同步引擎

**设计要点**：
- 本地优先原则：隐私敏感数据不上云
- 渐进式同步：增量同步，冲突可解
- 断网自动降级，联网自动恢复
- 数据端到端加密传输

---

### 3.5 M4 业务场景引擎

**定位**：业务调度层，根据用户状态智能切换场景模式。

**核心能力**：
- 8 大场景智能识别（工作/学习/生活/休闲等）
- 场景暖切换（上下文平滑过渡）
- 场景规则引擎
- 多模态感知（时间、位置、活动、语音）
- 场景配置与自定义
- 场景间上下文传递

**技术选型**：FastAPI + 规则引擎 + 状态机

**设计要点**：
- 场景切换无感知，上下文自动迁移
- 支持用户手动切换和自动识别
- 每个场景有独立的 Agent 组合和技能配置

---

### 3.6 M5 潮汐分层记忆

**定位**：智能核心层的数据基座，模拟人类记忆机制。

**核心能力**：
- 四层记忆模型（沙滩记忆/短期记忆/长期记忆/深度记忆）
- 记忆巩固机制（睡眠巩固、重复强化）
- 情绪推断与记忆关联
- 向量语义检索（FAISS）
- 记忆权限与三级域权限
- 四级密级标记规范
- 跨设备记忆同步

**技术选型**：FastAPI + SQLite + FAISS + 向量嵌入

**设计要点**：
- 记忆自动流转：沙滩 → 浅水 → 深水 → 深海
- 睡眠巩固：闲置期间自动整理记忆
- 隐私核心：完全本地存储，不上云
- 语义检索：支持自然语言查询记忆

> 🔒 **M5 为核心私有模块**，仅保留在本地设备。

---

### 3.7 M7 积木编排平台

**定位**：业务调度层，提供可视化的工作流编排能力。

**核心能力**：
- 可视化 DAG 工作流编辑器
- 节点类型丰富（AI 调用、技能、条件、循环、并行）
- 模板市场（分享、下载、评分）
- 工作流版本管理
- 执行历史与回放
- 自定义积木开发

**技术选型**：FastAPI + Vue 3 可视化编辑器 + DAG 执行引擎

**设计要点**：
- 拖拽式设计，低代码门槛
- 条件分支、循环、子流程支持
- 工作流可导出为 JSON 分享
- 模板市场与技能市场联动

---

### 3.8 M8 管理控制塔

**定位**：管控层核心，系统的统一管理中枢。

**核心能力**（47 个路由模块）：

| 类别 | 模块数 | 说明 |
|------|--------|------|
| 基础管理 | 8 | 认证、用户、系统、审计、模块纳管、监控、部署、安全 |
| 算力调度 | 8 | 算力源、算力分组、模型配置、路由策略、监控统计、GPU管理 |
| 自进化引擎 | 3 | 进化规划、部署治理、安全审计 |
| 业务功能 | 14 | 聊天、知识库、Agent、记忆、成长中心、工作流、复盘等 |
| 场景与模式 | 3 | 场景管理、模式切换、上下文 |
| 语音与提醒 | 3 | TTS、ASR、提醒管理 |
| 硬件与设备 | 3 | 设备管理、数据采集、远程控制 |
| 工具与其他 | 5 | 通用工具、系统配置等 |

**技术选型**：FastAPI + SQLAlchemy + JWT + 进程管理器

**设计要点**：
- 所有模块通过 M8 标准接口纳管（health/metrics/config）
- 统一认证体系（JWT + API Key + Module Token）
- 自进化引擎：健康扫描 → 问题诊断 → 自动优化 → 一键回滚
- 算力调度：多模型路由、故障转移、熔断降级

---

### 3.9 M9 开发者工坊

**定位**：生产力层，为开发者提供一站式开发工具链。

**核心能力**：
- VSCode 管理（启动、状态、扩展）
- 工作区与项目管理
- 代码沙箱（安全执行代码）
- MCP 桥接（MCP 工具管理与调用）
- 项目模板与脚手架
- 开发者仪表盘

**技术选型**：FastAPI + 子进程管理 + MCP SDK

**设计要点**：
- 代码沙箱隔离，防止恶意代码
- 多语言支持（Python/JS/Shell 等）
- 与 M11 MCP 总线深度集成
- 项目文件管理与版本控制

---

### 3.10 M10 系统卫士

**定位**：基础设施层，7×24 小时守护系统运行。

**核心能力**：
- 系统资源监控（CPU/内存/磁盘/网络/GPU）
- 进程管理（启停、状态跟踪、异常检测）
- 阈值防护（告警阈值配置与触发）
- GPU 智能调度（潮汐调度、显存管理）
- 进程守护（自动重启异常进程）
- 性能数据采集与分析

**技术选型**：FastAPI + psutil + NVIDIA Management Library

**设计要点**：
- 轻量级，资源占用低
- 被动采集 + 主动告警
- GPU 潮汐调度：闲置时释放给其他任务
- 多级阈值：警告/严重/紧急

---

### 3.11 M11 MCP 总线

**定位**：基础设施层，统一的工具与服务调用总线。

**核心能力**：
- MCP 服务注册与发现
- 87 个 MCP 工具统一路由
- 多传输协议支持（HTTP/SSE/STDIO）
- 工具调用鉴权与限流
- 服务健康检查与故障转移
- API Key 管理

**技术选型**：FastAPI + MCP SDK + 注册中心模式

**设计要点**：
- 统一入口：所有 MCP 调用通过总线
- 多协议适配：HTTP、SSE、STDIO
- 工具聚合：跨服务的工具列表统一呈现
- 调用审计：所有工具调用可追溯

---

### 3.12 M12 安全盾

**定位**：安全防护层，提供全系统的安全防护能力。

**核心能力**：
- WAF 防护（SQL 注入、XSS、命令注入、路径遍历）
- JWT 认证与密钥管理
- IP 黑白名单控制
- 速率限制（多级限流、渐进式封禁）
- 威胁检测与防御
- 安全审计日志
- API Key 管理

**技术选型**：FastAPI 中间件 + WAF 引擎 + 令牌桶算法

**设计要点**：
- 纵深防御：多层安全防护叠加
- 零信任：每次请求都验证
- 可观测：所有安全事件可审计
- 降级策略：安全引擎异常时不阻塞正常请求

---

### 3.13 API 网关

**定位**：系统统一接入层，外部流量的唯一入口。

**核心能力**：
- 路由转发（基于路径前缀的模块路由）
- 认证鉴权（API Key + JWT）
- 速率限制（全局/IP/API Key 三级）
- 熔断降级（熔断器模式）
- CORS 处理
- 请求/响应头注入

**技术选型**：FastAPI + HTTPX 客户端 + 令牌桶 + 熔断器

**设计要点**：
- 从 M8 拆分出代理职责，解决 M8 单点问题
- 12 个模块统一接入
- 熔断器状态机：Closed → Open → Half-Open
- 自动恢复机制

## 5. 技术决策记录

### 5.1 后端框架：FastAPI

**决策**：选择 FastAPI 作为所有后端模块的 Web 框架。

**原因**：
- 高性能：基于 Starlette 和 Pydantic，异步支持好
- 自动文档：Swagger UI / ReDoc 自动生成
- 类型安全：Pydantic 模型提供强类型校验
- 生态成熟：Python 生态最主流的异步 Web 框架之一
- 开发效率高：代码量少，约定优于配置

**备选方案**：Flask、Django、Tornado

---

### 5.2 数据库：SQLite 本地优先

**决策**：采用 SQLite 作为各模块的默认数据库，本地优先。

**原因**：
- 零配置：无需安装数据库服务，开箱即用
- 本地优先：数据存在用户设备上，隐私保护
- 单用户场景：个人操作系统，并发量可控
- 可移植性：单文件数据库，备份迁移方便
- 足够快：对于单用户场景，SQLite 性能完全够用

**备选方案**：PostgreSQL、MySQL、MongoDB

**适用场景例外**：多用户部署时可切换到 PostgreSQL。

---

### 5.3 向量检索：FAISS

**决策**：采用 Facebook FAISS 作为向量检索引擎。

**原因**：
- 高性能：C++ 实现，检索速度快
- 本地运行：不需要独立服务，库调用即可
- 成熟稳定：工业界广泛使用
- 多索引类型：支持 Flat、IVF、HNSW 等

**备选方案**：ChromaDB、Milvus、Qdrant、Weaviate

---

### 5.4 模块间通信：HTTP + MCP

**决策**：模块间通信采用 HTTP REST + MCP 协议双轨制。

**原因**：
- HTTP REST：简单通用，调试方便，生态成熟
- MCP 协议：标准化工具调用，跨平台互操作
- 双轨制：业务调用用 HTTP，工具能力用 MCP
- 松耦合：通过标准接口通信，模块独立演进

**备选方案**：gRPC、消息队列（RabbitMQ/Kafka）

---

### 5.5 前端框架：Vue 3 + Vite

**决策**：前端采用 Vue 3 + Vite + SPA 架构。

**原因**：
- Vue 3 组合式 API：灵活、可复用
- Vite 构建快：开发体验好
- Pinia 状态管理：轻量、直观
- 生态丰富：Element Plus 等组件库成熟
- 渐进式：可逐步采用，适合模块独立开发

**备选方案**：React、Svelte、原生 JS

---

### 5.6 认证方案：JWT + API Key

**决策**：采用 JWT（用户认证）+ API Key（服务间调用）+ Module Token（M8 纳管）三层认证体系。

**原因**：
- JWT：无状态、跨域友好、适合用户认证
- API Key：简单高效、适合服务间调用
- Module Token：M8 纳管专用，权限隔离
- 三层体系：不同场景用不同认证方式，安全且灵活

**备选方案**：Session、OAuth2、mTLS

---

### 5.7 进程管理：原生进程

**决策**：采用原生进程方式启动各模块，而非容器化作为默认方式。

**原因**：
- 本地优先：个人设备上原生进程性能更好
- 资源占用低：没有容器的额外开销
- 调试方便：直接看进程、attach 调试
- Windows 友好：Docker 在 Windows 上体验不如 Linux
- Docker 可选：需要时可用 Docker Compose 部署

## 6. 设计原则

### 6.1 架构原则

1. **单一职责**：每个模块只负责一个核心领域
2. **松耦合高内聚**：模块间通过标准接口通信，内部高度聚合
3. **分层清晰**：严格遵循分层依赖规则，不越级调用
4. **可独立部署**：每个模块可独立启停、独立升级
5. **故障隔离**：单个模块故障不影响全局，支持降级运行

### 6.2 开发原则

1. **约定优于配置**：遵循统一的命名、结构、接口规范
2. **防御式编程**：输入校验、错误处理、边界条件
3. **可测试性**：核心逻辑可单元测试，依赖可 Mock
4. **可观测性**：日志、指标、链路追踪完善
5. **安全左移**：安全考虑融入开发每个环节

### 6.3 数据原则

1. **本地优先**：用户数据默认存储在本地
2. **隐私设计**：Privacy by Design，数据最小化
3. **数据主权**：用户完全掌控自己的数据
4. **端到端加密**：传输和存储都加密
5. **可移植性**：数据格式开放，可导出迁移

### 6.4 API 原则

1. **RESTful 规范**：遵循 REST 设计原则
2. **统一响应格式**：所有 API 响应格式一致
3. **统一错误码**：6 位错误码体系，模块+类别+序号
4. **版本化**：API 版本通过 URL 路径标识
5. **向后兼容**：尽量不破坏已有接口
