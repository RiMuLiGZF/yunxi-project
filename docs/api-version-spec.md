# API 版本前缀统一规范

> 文档版本：v1.0  
> 最后更新：2026-07-12  
> 适用范围：云汐项目全模块（M1 ~ M12）

---

## 1. 背景与目标

### 1.1 背景

随着云汐项目各模块的持续迭代，API 路由前缀逐渐呈现出多样化、不一致的现状：

- **版本号不统一**：部分模块使用 `/api/v1/`，部分使用 `/api/v2/`，还有部分直接使用 `/api/` 无版本前缀
- **业务接口与标准接口混杂**：M8 标准接口（health/metrics/config）与业务接口路径规则不一致
- **特殊协议端点缺少规范**：MCP 相关端点在不同模块中的路径定义存在差异（`/mcp/`、`/mcp/v1/` 等）
- **根路径用途不明确**：部分模块根路径 `/` 与 `/health` 职责重叠，探活入口不统一
- **跨模块调用成本高**：前端集成与网关路由配置需为每个模块单独适配，维护成本递增

### 1.2 目标

制定统一的 API 版本前缀规范，实现以下目标：

1. **一致性**：所有业务 API 遵循统一的版本前缀规则，降低跨模块集成成本
2. **可演进性**：通过版本号前缀为后续 API 迭代（v2、v3）预留扩展空间
3. **可识别性**：通过路径前缀即可区分业务接口、标准接口、协议端点
4. **兼容性**：在统一前缀的迁移过程中，保障现有调用方不受影响
5. **可治理性**：为 API 网关统一路由、鉴权、限流提供标准化依据

## 2. 统一规范

### 2.1 前缀分类总览

| 前缀类型 | 路径前缀 | 用途说明 | 适用模块 |
|---------|---------|---------|---------|
| 业务 API | `/api/v1/` | 各模块对外提供的业务功能接口 | 全部业务模块 |
| M8 标准接口 | `/m8/` | 健康检查、性能指标、配置查询三项标准接口 | 全部服务模块 |
| MCP 协议端点 | `/mcp` | MCP（Model Context Protocol）协议相关端点 | M4、M11 等 MCP 相关模块 |
| 探活端点 | `/`、`/health` | 负载均衡与容器编排的最简探活接口 | 全部服务模块 |

### 2.2 业务 API 前缀规范

**统一前缀：`/api/v1/`**

- 所有业务功能接口必须以 `/api/v1/` 作为版本前缀
- 路径结构：`/api/v1/{资源域}/{资源标识}/{子资源}`
- 示例：
  - `/api/v1/skills` —— 技能列表
  - `/api/v1/workflows/{id}/runs` —— 工作流运行记录
  - `/api/v1/tools/{tool_name}/call` —— 工具调用

> **说明**：当前各模块中存在的 `/api/v2/` 前缀（如 M2 技能集群的升级、测试相关接口）属于模块内部实验性接口，待功能稳定后应逐步合并至 v1 主干或按规范申请独立版本号。

### 2.3 M8 标准接口前缀规范

**统一前缀：`/m8/`**

M8 标准接口为云汐平台所有服务模块必须实现的三项基础设施接口，独立于业务版本体系：

| 接口路径 | 方法 | 说明 |
|---------|------|------|
| `/m8/health` | GET | 标准健康检查，返回服务状态与版本信息 |
| `/m8/metrics` | GET | 性能指标查询，返回 QPS、延迟、内存等指标 |
| `/m8/config` | GET | 配置查询，返回当前服务运行配置摘要 |

**约束**：
- M8 标准接口路径固定，**不得**添加 `/api/` 或版本号前缀
- 所有服务模块必须完整实现上述三个端点
- 响应格式遵循 M8 统一响应模型规范

### 2.4 MCP 协议端点前缀规范

**统一前缀：`/mcp`**

MCP（Model Context Protocol）协议相关端点使用独立前缀，与业务 API 隔离：

| 接口路径 | 方法 | 说明 |
|---------|------|------|
| `/mcp` | POST | MCP JSON-RPC 2.0 主端点 |
| `/mcp/tools` | GET | MCP 工具列表查询 |
| `/mcp/tools/{tool_name}` | GET | 单工具详情查询 |
| `/mcp/tools/{tool_name}/call` | POST | 工具调用（REST 风格封装） |
| `/mcp/health` | GET | MCP 服务健康检查 |
| `/mcp/config` | GET/PUT | MCP 服务配置查询与更新 |

**约束**：
- MCP 端点统一使用 `/mcp` 前缀，**不**附加版本号
- MCP 协议版本通过请求体中的 `jsonrpc` 字段或协议头标识，而非路径版本
- M11（MCP 总线）作为 MCP 协议的核心模块，是该规范的主要适用对象
- 其他模块如需暴露 MCP 能力，应统一对接 M11，而非各自实现 MCP 端点

### 2.5 探活端点规范

**保留路径：`/` 和 `/health`**

用于基础设施层（Kubernetes、Nginx、API 网关）的最简健康探测：

| 接口路径 | 方法 | 说明 |
|---------|------|------|
| `/` | GET | 根路径探活，返回 200 OK 即可，响应体可极简 |
| `/health` | GET | 简化健康检查，返回基本状态信息（比 `/m8/health` 更轻量） |

**约束**：
- 根路径 `/` 仅用于探活，不得承载业务逻辑或重定向到前端页面
- `/health` 响应时间应控制在 50ms 以内，不得依赖外部服务
- 如需完整健康信息（含依赖检查），应使用 `/m8/health`
- 两个端点均**不得**要求鉴权

## 3. 迁移策略

### 3.1 迁移原则

1. **新增优先**：所有新开发的业务接口直接使用 `/api/v1/` 前缀
2. **兼容过渡**：旧路径保留至少一个完整版本周期，标记为 deprecated
3. **渐进迁移**：按模块优先级分批迁移，避免一次性大规模变更
4. **可观测**：迁移过程中同时统计新旧路径的调用量，评估下线时机

### 3.2 迁移步骤

**阶段一：新增 v1 路径（双轨运行）**

- 在各模块中新增 `/api/v1/` 前缀的业务接口
- 旧路径保留，内部实现转发到新路径的处理函数（别名方式）
- 在旧接口的 OpenAPI 文档中添加 `Deprecated: true` 标记
- 在响应头中添加 `Deprecation` 和 `Sunset` 提示

```python
# 示例：旧路径兼容（FastAPI）
@router.get("/api/skills", deprecated=True, summary="【已废弃】请使用 /api/v1/skills")
async def get_skills_legacy():
    return await get_skills_v1()

@router.get("/api/v1/skills", summary="获取技能列表")
async def get_skills_v1():
    # 实际业务逻辑
    ...
```

**阶段二：调用方切换**

- 前端、网关、其他服务调用方逐步切换到 v1 路径
- 通过日志监控旧路径调用量，确认无关键调用依赖
- 同步更新 API 文档、SDK、接口测试用例

**阶段三：旧路径下线**

- 旧路径调用量持续为零超过 2 周后，发起下线评审
- 下线前发布变更公告，保留至少 1 周缓冲期
- 正式下线旧路径，返回 410 Gone 或 404 响应

### 3.3 重定向策略

对于部分纯查询类 GET 接口，可采用 HTTP 重定向方式过渡：

- **301 Moved Permanently**：适用于确认永久迁移的接口
- **307 Temporary Redirect**：适用于临时迁移或测试中的路径
- 重定向响应头中携带 `Location: /api/v1/...`

> **注意**：POST/PUT/DELETE 等有副作用的接口**不得**使用重定向过渡，必须采用别名方式（同一处理函数挂载多个路径）。

## 4. 各模块现状与迁移计划

### 4.1 模块总览表

| 模块编号 | 模块名称 | 当前业务前缀 | M8 标准接口 | MCP 端点 | 探活端点 | 迁移优先级 | 计划版本 | 备注 |
|---------|---------|------------|------------|---------|---------|-----------|---------|------|
| M1 | 多 Agent 调度 | 待确认 | 待确认 | 无 | 待确认 | 中 | v1.2 | 需梳理现有接口清单 |
| M2 | 技能集群 | `/api/v1/`、`/api/v2/` 混合 | 已合规（`/m8/`） | `/mcp/v1/`（需调整） | 已合规 | 高 | v1.2 | v2 实验接口需评审合并策略 |
| M3 | 场景协同 | `/api/v1/` | 待确认 | 无 | 待确认 | 中 | v1.2 | 主体已合规，需补全 M8 接口 |
| M4 | 场景引擎 | 无统一前缀 | 待确认 | `/mcp/`（已合规） | `/health` | 高 | v1.2 | 业务接口需统一加 v1 前缀 |
| M5 | 知识图谱 | 待确认 | 待确认 | 无 | 待确认 | 中 | v1.3 | 需梳理现有接口清单 |
| M6 | 硬件外设 | `/api/v1/` | 已合规（`/m8/`） | 无 | 已合规 | 低 | v1.2 | 基本合规，仅需核验完整性 |
| M7 | 工作流构建器 | `/api/v1/` | 已合规（`/m8/`） | 无 | 已合规 | 低 | v1.2 | 基本合规，仅需核验完整性 |
| M8 | 管理控制台 | `/api/`（无版本号） | 已合规（`/m8/`） | 无 | 已合规 | 高 | v1.2 | 大量接口需加 v1 前缀 |
| M9 | 编程开发 | 无统一前缀 | 已合规（`/m8/`） | 无 | 已合规 | 中 | v1.3 | 需梳理业务接口并统一前缀 |
| M10 | 系统守护 | `/api/`（无版本号） | 已合规（`/m8/`） | 无 | 已合规 | 中 | v1.3 | 业务接口需加 v1 前缀 |
| M11 | MCP 总线 | `/api/v1/`、`/api/console/` 混合 | 已合规（`/m8/`） | `/mcp`（已合规） | 已合规 | 高 | v1.2 | console 接口需迁移至 v1 |
| M12 | 端侧应用 | 待确认 | 待确认 | 无 | 待确认 | 低 | v1.3 | 端侧接口规范需单独评估 |

### 4.2 优先级说明

- **P0（高优先级）**：M2、M4、M8、M11
  - 接口数量多、调用方多、与其他模块耦合度高
  - 前缀不一致问题最突出，迁移收益最大
  - 安排在 v1.2 版本完成迁移

- **P1（中优先级）**：M1、M3、M5、M9、M10
  - 存在一定的前缀不规范问题，但影响范围可控
  - 可在核心模块迁移完成后跟进
  - 安排在 v1.2 ~ v1.3 版本完成

- **P2（低优先级）**：M6、M7、M12
  - 已基本合规或接口数量少
  - M12 端侧应用需结合端侧 SDK 发布节奏单独评估
  - 安排在 v1.3 版本完成核验

### 4.3 各模块迁移要点

**M2 技能集群**
- 现状：`/api/v1/` 与 `/api/v2/` 双版本共存，v2 包含升级、测试等实验性接口
- 策略：v2 接口评审后，稳定功能合并入 v1，实验性功能保留 v2 并明确标注
- MCP 端点 `/mcp/v1/` 需调整为 `/mcp/` 规范格式

**M8 管理控制台**
- 现状：业务接口使用 `/api/modules/`、`/api/system/` 等无版本前缀路径
- 策略：新增 `/api/v1/modules/`、`/api/v1/system/` 等对应路径，旧路径做别名兼容
- 管理台前端同步切换调用路径

**M11 MCP 总线**
- 现状：控制台接口使用 `/api/console/` 前缀，REST 工具接口已使用 `/api/v1/tools`
- 策略：`/api/console/*` 迁移至 `/api/v1/console/*`，API Key 管理接口同步迁移
- MCP 主端点 `/mcp` 已合规，无需调整

## 5. 兼容性保障方案

### 5.1 兼容层级

| 兼容层级 | 保障措施 | 适用场景 |
|---------|---------|---------|
| 路径兼容 | 旧路径作为别名保留，指向同一处理函数 | 所有迁移接口的默认策略 |
| 重定向兼容 | GET 接口返回 301/307 重定向到新路径 | 纯查询类、无副作用的 GET 接口 |
| 网关兼容 | API 网关层统一做路径改写 | 调用方无法快速修改的场景 |
| 响应兼容 | 响应结构保持一致，仅路径变更 | 所有迁移接口的基本要求 |

### 5.2 废弃标记规范

所有已废弃的旧接口必须包含以下标记：

1. **OpenAPI 标记**：`deprecated=True`，并在 description 中注明替代路径
2. **响应头标记**：添加 `Deprecation: true` 和 `Sunset: <日期>` 响应头
3. **日志告警**：旧路径被调用时记录 WARN 级别日志，便于统计调用量
4. **文档标注**：API 文档中使用醒目的废弃标识，指引使用新接口

```
# 响应头示例
Deprecation: true
Sunset: 2026-09-01
Link: </api/v1/skills>; rel="successor-version"
```

### 5.3 迁移验证方案

**接口一致性验证**
- 新旧路径调用同一接口，对比响应体是否完全一致（排除路径字段）
- 验证所有 HTTP 方法：GET、POST、PUT、DELETE、PATCH
- 验证异常场景：参数错误、鉴权失败、服务降级等

**性能验证**
- 迁移后接口的 P95 延迟不得劣化超过 5%
- 别名转发方式的额外开销应控制在 1ms 以内

**调用量监控**
- 分别统计新旧路径的调用 QPS
- 每日生成调用量趋势报告
- 旧路径调用量归零并稳定 2 周后方可启动下线流程

### 5.4 回滚机制

1. **快速回滚**：由于采用别名方式（新旧路径共存），如发现问题可立即恢复调用旧路径，无需代码回滚
2. **版本回滚**：如迁移引入代码缺陷，按 Git 版本快照回滚至上一稳定版本
3. **网关兜底**：API 网关配置回退规则，出现异常时可一键将流量切回旧路径
4. **灰度发布**：高风险模块的迁移采用灰度策略，先开放内部测试调用，再逐步放量

### 5.5 长期治理

- **新增接口审查**：代码评审环节检查新接口路径是否符合规范
- **定期审计**：每季度进行一次全模块 API 前缀合规性扫描
- **版本升级机制**：当 v2 版本需求出现时，按相同规范添加 `/api/v2/` 前缀，v1 进入废弃周期
- **网关统一管控**：长期目标是通过 API 网关统一管理所有模块的路由前缀与版本策略
