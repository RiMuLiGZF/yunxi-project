## 二、P0 级严重问题（必须修复）

### P0-1：M7 积木平台端口配置不一致
- **模块**：M7
- **问题**：M7 实际运行端口是 3001，但 shared/config.py 中配置的是 8007，M8 管理台也认为是 8007
- **影响**：M7 无法被 M8 统一纳管和调用，成为孤岛模块
- **根因**：M7 的 .env.example 和默认配置用的是 3001，与其他模块（800x 系列不一致
- **修复方案**：将 M7 端口统一为 8007，修改 server.py 和 .env.example

### P0-2：M8 仪表盘数据全部硬编码
- **模块**：M8 前端
- **问题**：仪表盘显示"在线模块数 8/8"、"今日任务数 128"等全部是硬编码假数据
- **影响**：用户无法了解系统真实运行状态，监控形同虚设
- **修复方案**：前端调用真实 API 接口获取数据

### P0-3：M3 Mock 回退机制过于宽松
- **模块**：M3
- **问题**：server.py 中组件初始化失败时静默降级为 Mock，调用方无法区分真实/模拟数据
- **影响**：生产环境可能在用户不知情的情况下返回假数据
- **修复方案**：响应中增加 mode 字段标识当前是 real 还是 mock

## 三、P1 级重要问题（建议修复）

### P1-1：多模块 M8 metrics 返回全零假数据
- **影响模块**：M4、M5、M6、M8
- **问题**：/m8/metrics 中 CPU、内存、请求数等硬编码为 0
- **影响**：M8 监控中心无法获取真实运行指标
- **修复方案**：接入 psutil 采集真实系统指标（M9/M10 已有实现可参考）

### P1-2：M6 业务 API 完全无鉴权
- **模块**：M6
- **问题**：/api/v1/devices、/api/v1/sensors、/api/v1/control 等业务 API 无任何鉴权保护
- **影响**：严重安全隐患，生产环境不可用
- **修复方案**：添加 M8 Token 鉴权中间件（M7 已有实现可参考）

### P1-3：M6 / M7 完全缺少单元测试
- **模块**：M6、M7
- **问题**：无 tests/ 目录，无任何单元测试
- **影响**：无法自动化回归验证，质量保障薄弱
- **修复方案**：补充核心功能单元测试（参考 M4/M10 的测试结构）

### P1-4：M4 缺少统一配置管理类
- **模块**：M4
- **问题**：配置散落在各模块直接读取 os.environ，无统一 Config 类
- **影响**：配置维护困难，缺少校验和默认值集中管理
- **修复方案**：添加 config.py 统一配置管理（参考 M6/M10 的实现模式）

### P1-5：M3 /m8/* 接口未复用 m8_api 实现
- **模块**：M3
- **问题**：m8_api/ 目录下有完整的 health_endpoints.py、config_endpoints.py，但 /m8/health、/m8/metrics、/m8/config 三个标准路径返回硬编码简化数据
- **影响**：代码重复，维护成本高，M8 获取的指标不完整
- **修复方案**：server.py 中挂载 m8_api 下的完整实现

### P1-6：M5 根目录缺少 README.md
- **模块**：M5
- **问题**：文档在 docs/README.md 中，根目录无 README，不符合常规项目结构
- **影响**：新人上手和自动化工具查找文档困难
- **修复方案**：在根目录创建 README.md

### P1-7：M1 / M2 / M3 缺少 .env.example
- **模块**：M1、M2、M3
- **问题**：无独立 .env.example 文件，环境变量配置入口不清晰
- **影响**：部署配置不直观
- **修复方案**：补充 .env.example 文件

### P1-8：shared 零测试覆盖
- **模块**：shared
- **问题**：作为公共基础组件，完全没有单元测试
- **影响**：基础组件改动风险高，回归无保障
- **修复方案**：为 config、logger、module_client 等核心模块补充测试

### P1-9：shared 缺少 requirements.txt
- **模块**：shared
- **问题**：无独立依赖声明，依赖由各使用方自行管理
- **影响**：新环境部署时容易遗漏依赖
- **修复方案**：添加 shared/requirements.txt

## 七、关键发现摘要

### 做得好的方面
1. M1/M2/M5/M10 核心功能完整度高，测试覆盖较好
2. M8 标准接口各模块基本都有实现
3. 前端设计系统较统一
4. 公共组件 shared 提供了配置/日志/模块调用/进程管理等基础能力
5. M10 系统卫士功能完整，真实指标采集做得最好
6. M7 的 M8 鉴权中间件实现规范，可作为范本

### 共性问题
1. M8 metrics 普遍假数据：大部分模块的 /m8/metrics 返回硬编码 0 值
2. 配置体系不统一：shared/config.py 与各模块自有 config.py 重叠
3. 测试覆盖率不均：M1/M2/M5/M10 较好，M6/M7/shared 为零
4. 前后端数据脱节：前端仪表盘显示假数据，不调用真实 API
5. 安全意识薄弱：M6 业务 API 完全无鉴权，M9 沙箱形同虚设
6. 端口不统一：M7 用 3001 而非 8007，与体系不一致
