## 一、现状与差距分析

### 1.1 当前实现概述

M5 潮汐记忆模块作为云汐项目的核心记忆系统，当前版本（M11 版）已实现四层潮汐存储架构与六步混合检索能力，在功能层面覆盖了从短期记忆到长期归档的完整链路。以下从代码规模、核心能力、已知问题三个维度进行概述。

#### 1.1.1 代码规模与结构

- **有效代码量**：约 7,266 行
- **文件数量**：47 个
- **主要模块分布**：
  - 存储层（L0/L1/L2/L3）：约占 45%
  - 检索引擎（关键词/向量/RRF/MMR/情绪重排）：约占 25%
  - 安全防护（权限/密级/脱敏/审计）：约占 15%
  - 工具与基础设施：约占 15%
- **测试覆盖率**：10.9%（远低于行业基线 60%）

#### 1.1.2 核心能力盘点

| 能力维度 | 实现状态 | 技术选型 | 备注 |
|---------|---------|---------|------|
| 存储架构 | 已实现 | 四层潮汐模型（L0沙滩/L1浅水/L2深水/L3深海） | 核心骨架完整 |
| 检索能力 | 已实现 | 六步混合检索（关键词+向量+RRF融合+MMR去重+情绪重排） | 链路完整但质量受限 |

#### 1.1.3 主要问题梳理

**问题一："不存原文"与向量检索的根本矛盾**

当前 M5 遵循"不存储原始对话文本"的隐私保护原则，仅存储语义向量与结构化元数据。这一设计在隐私保护层面有其合理性，但直接导致向量检索质量严重下降：
- 向量维度有限（通常 384~1024 维），无法完整还原语义细节
- 关键词检索退化为元数据字段匹配，精度不足
- 结果呈现时缺乏原文支撑，用户无法验证检索结果的准确性
- 无法基于原文进行摘要、蒸馏、联想等高级操作

**问题二：L1/L2 层代码重复率 > 80%**

L1 浅水层与 L2 深水层在存储结构、检索逻辑、元数据管理上高度相似，仅在访问频率、保留策略、压缩级别上有差异。当前实现为两套独立代码，导致：
- 维护成本倍增，Bug 修复需同步两处
- 行为不一致风险高，易出现"同一查询在 L1 和 L2 结果差异巨大"的问题
- 新增功能需双倍工作量

**问题三：M8 接口两套实现**

M8 作为 M5 的对外服务接口，存在两套独立实现路径，接口行为不完全一致，调用方需做兼容性适配，增加了集成复杂度。

**问题四：缺失批量 API、分页查询、备份恢复**

- 批量写入/删除/更新能力缺失，导致数据初始化、迁移、批量清理等运维操作效率极低
- 分页查询能力缺失，前端无法做列表展示，检索结果一次性返回全量数据
- 备份恢复机制缺失，数据安全无保障，无法应对误删除、磁盘损坏等场景

**问题五：FAISS 暴力搜索性能瓶颈**

当前使用 `IndexFlatIP` 暴力内积搜索，时间复杂度 O(n)。当向量规模增长到 10 万条以上时，单次检索耗时将突破秒级，严重影响用户体验。

**问题六：测试覆盖不足（10.9%）**

测试覆盖率仅 10.9%，核心检索链路、边界条件、异常场景均缺乏有效测试保障。在大规模重构与功能升级过程中，回归风险极高。

### 1.2 设计愿景回顾

从 v2.4 与 v3.0 两代设计文档中，可提取出 M5 模块的完整演进蓝图。以下从架构层、能力层、体验层三个维度进行回顾。

#### 1.2.1 v2.4 完整设计：Agent 化与游戏化

**架构范式：从单体模块到 Agent 集群**

v2.4 设计将 M5 从单一记忆模块升级为由 6 个子 Agent 协同工作的记忆系统：

| 子 Agent | 职责定位 | 核心能力 |
|---------|---------|---------|
| Recall Agent | 检索与回忆 | 多模态检索、上下文联想、情绪唤起 |
| Archive Agent | 归档与管理 | 潮汐迁移、压缩存储、生命周期管理 |
| Emotion Agent | 情绪与价值 | 情绪标签、价值评估、权重调整 |
| Narrative Agent | 叙事与组织 | 故事线生成、记忆串联、意义建构 |
| Talent Agent | 天赋与成长 | 能力建模、技能沉淀、成长路径 |
| Season Agent | 周期与节律 | 潮汐调度、赛季管理、节律控制 |

**通信协议：A2A（Agent-to-Agent）**

采用 JSON over HTTPS 的 A2A 协议替代 MCP，核心设计要点：
- 标准化消息信封（header + payload + signature）
- 支持同步请求/响应与异步事件通知两种模式
- 内置 QoS 优先级标记与超时控制
- 统一错误码与重试策略

**QoS 引擎：四级优先级 + 背压控制**

| 优先级 | 名称 | 典型场景 | 延迟目标 | 重试策略 |
|-------|------|---------|---------|---------|
| P0 | CRITICAL | 用户实时查询、身份验证 | < 100ms | 立即重试 × 3 |
| P1 | HIGH | 记忆写入、情绪更新 | < 500ms | 指数退避 × 5 |
| P2 | NORMAL | 潮汐迁移、定期巩固 | < 5s | 队列重试 × 3 |
| P3 | LOW | 统计报表、背景索引 | 异步/分钟级 | 失败累积后批量重试 |

背压控制机制：
- 基于令牌桶的流量整形
- 队列长度水位线告警（70%/85%/95% 三级）
- 低优先级任务自动降级或丢弃
- 过载保护：自动拒绝新请求并返回 Retry-After

**三层记忆组织模型**

1. **语义原子（Semantic Atom）**：最小记忆单元，单条事实/观点/体验，附带时间、来源、情绪、权重等元数据
2. **语义页表（Semantic Page Table）**：将语义原子按主题/时间/场景聚合成页，类似虚拟内存的页表机制，支持快速定位与批量操作
3. **原子图谱（Atom Graph）**：语义原子之间的关联网络，基于因果、相似、对比、时序等关系构建，支持联想推理

**潮汐四相循环控制器**

基于状态机驱动的四相循环：

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  涨潮期  │───▶│  高潮期  │───▶│  退潮期  │───▶│  低潮期  │
│ Flood   │    │ High    │    │ Ebb     │    │ Low     │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
      ▲                                                │
      └────────────────────────────────────────────────┘
```

- **涨潮期（Flood）**：记忆快速写入 L0/L1，检索活跃度高，以可用性为优先
- **高潮期（High）**：系统负载峰值，开启 QoS 限流，保证核心查询稳定
- **退潮期（Ebb）**：批量执行巩固、迁移、压缩，将 L0→L1→L2→L3 逐级下沉
- **低潮期（Low）**：系统空闲，执行深度索引、遗忘清理、备份归档等低优先级任务

**六大游戏化模块**

| 模块 | 英文代号 | 核心机制 | 预期效果 |
|-----|---------|---------|---------|
| 地球Online编年史 | storyline | 自动生成时间线叙事 | 增强记忆连贯性与意义感 |
| 成就勋章殿堂 | medals | 4 类 × 4 级 = 16 枚勋章 | 正向激励用户互动 |
| 心智天赋树 | talents | 4 分支 × 8 节点 × 3 层 = 96 天赋 | 可视化能力成长轨迹 |
| 潮汐专属历法 | calendar | 7 天小潮 / 30 天大潮 | 建立使用节律感 |
| 记忆回响对比 | echoes | 同一主题跨时间对比 | 呈现成长与变化 |
| 赛季征程系统 | seasons | 180 天 / 期，6 阶段 | 长期目标驱动 |

#### 1.2.2 v3.0 设计：自省进化系统

**系统定位**

v3.0 在 v2.4 基础上新增自我认知与进化能力，使 M5 不仅是被动存储系统，而是能够主动感知自身不足、寻求改进方案、自主完成升级的智能体。

**七大核心组件**

| 组件 | 职责 | 输入 | 输出 |
|-----|------|------|------|
| CapabilityInventory | 能力清单管理 | 模块元数据、依赖关系 | 当前能力图谱 |
| GapAnalyzer | 缺口分析 | 用户反馈、失败日志、对标数据 | 缺口优先级列表 |
| OpenSourceSearcher | 开源方案搜索 | 缺口描述、技术关键词 | 候选开源方案清单 |
| SecurityScanner | 安全扫描 | 候选方案代码 | 安全评估报告 |
| UpgradeReportGenerator | 升级报告生成 | 缺口+方案+评估 | 可执行升级方案 |
| InstallationEngine | 安装执行引擎 | 升级方案 | 安装结果与回滚点 |
| EvolutionScheduler | 进化调度器 | 系统负载、升级队列 | 进化执行计划 |

**三级安装体系**

| 级别 | 名称 | 适用场景 | 自动化程度 | 人工审核 |
|-----|------|---------|-----------|---------|
| L1 | 简单安装 | 配置项调整、参数优化、规则更新 | 全自动 | 无需 |
| L2 | 中等安装 | 新增模块、接口扩展、非核心功能 | 半自动 | 方案审核 |
| L3 | 复杂安装 | 核心架构变更、数据迁移、安全策略 | 手动执行 | 全面评审 |

| 信号源 | 权重 | 典型示例 | 采集方式 |

## 二、总体升级路线

### 2.1 升级原则

#### 原则一：务实优先，问题导向

不从设计文档倒推实现，而是从当前最痛的问题出发。L1/L2 代码重复、M8 接口不统一、批量/分页 API 缺失、测试覆盖率不足等"硬伤"优先解决，再谈 Agent 化与游戏化等"锦上添花"的功能。

#### 原则二：风险可控，小步快跑

每阶段变更范围可控，每阶段结束后有明确的验收标准与回滚方案。不追求"一步到位"式的大重构，而是通过 4 个阶段渐进式升级，每个阶段都能独立交付价值。

#### 原则三：渐进式，不破坏现有功能

所有升级均以"新增层/新增接口/可选开关"的方式引入，现有功能保持兼容。新能力通过 Feature Flag 控制灰度发布，验证稳定后再设为默认。不做"破坏性重构"。

#### 原则四：测试先行，质量保底

在每个 Phase 开始前，先补齐该 Phase 涉及模块的单元测试与集成测试，确保重构过程中有足够的测试守护。目标是在升级过程中将测试覆盖率从 10.9% 逐步提升至 60% 以上。

#### 原则五：文档同步，知识沉淀

代码变更与文档更新同步进行。每个 Phase 结束后，更新架构图、API 文档、运维手册，确保文档与代码一致。核心设计决策记录为 ADR（Architecture Decision Record）。

## 四、Phase 2：核心能力深化（P1，3-4周）

### 4.1 目标

Phase 2 的核心目标是**解决制约 M5 核心能力的根本性问题，深化记忆系统的核心能力，从"能用"迈向"好用"**。

具体目标拆解：

1. **检索质量突破**：通过可选加密原文存储解决"不存原文"的根本矛盾，大幅提升向量检索与关键词检索的精度与可用性
2. **检索性能跃升**：FAISS 从暴力搜索升级为 HNSW 索引，10w 向量检索延迟从秒级降至 50ms 以内
3. **记忆组织升级**：引入语义页表与原子图谱（基础版），从扁平存储升级为层次化组织
4. **潮汐控制自动化**：实现潮汐四相循环控制器，从手动触发升级为状态机驱动的自动化节律
5. **记忆加工能力**：梦境引擎上线，支持巩固、蒸馏、遗忘三种主动记忆加工
6. **性能全面优化**：L0-L1 KV 缓存联动 + SQLite WAL 模式，读写性能全面提升
7. **质量持续提升**：测试覆盖率从 30% 提升至 50%+

本阶段涉及核心存储结构与检索链路的深度改造，需在充分测试保障下稳步推进。所有新增能力均通过 Feature Flag 控制，默认关闭，验证稳定后再逐步开放。

## 五、Phase 3：游戏化系统（P2，3-4周）

### 5.1 目标

Phase 3 的核心目标是**在核心记忆系统稳定的基础上，构建游戏化体验层，将 M5 从"工具型记忆系统"升级为"陪伴型成长系统"**，显著提升用户粘性与使用深度。

具体目标拆解：

1. **成就激励体系**：16 枚成就勋章上线，覆盖探索、创造、社交、坚持四大维度，通过正向激励促进用户持续使用

7. **系统集成**：游戏化系统与核心记忆系统深度集成，数据互通，机制协同
8. **质量保障**：测试覆盖率从 50% 提升至 55%+

游戏化系统作为体验层，不影响核心记忆功能的稳定性。所有游戏化功能均可独立开关，用户可根据喜好选择启用或关闭。

**核心功能**

1. **自动时间线生成**：按日/周/月/年自动聚合记忆，生成时间线
2. **智能标题生成**：调用 LLM 为每个时间段生成有温度的标题（如"那个与咖啡为伴的夏天"）
3. **关键事件标注**：自动识别并标注重要事件（首次体验、里程碑、情绪高峰/低谷）
4. **主题叙事线**：按主题生成独立叙事线（如"我的项目管理成长之路"）
5. **随机回忆推送**："那年今日"功能，推送历史上的今天的记忆
6. **时间胶囊**：支持写给未来的自己，设定解锁时间

**时间线粒度**

| 视图 | 时间粒度 | 内容密度 | 典型使用场景 |
|------|---------|---------|-------------|
| 日视图 | 按小时 | 详细（全量记忆） | 当天回顾 |
| 周视图 | 按天 | 摘要（Top 10 关键记忆） | 周复盘 |
| 月视图 | 按周 | 精选（月度大事记） | 月度总结 |
| 年视图 | 按月 | 精华（年度里程碑） | 年终回顾 |
| 全量视图 | 按年 | 里程碑（人生节点） | 人生回顾 |

**技术实现要点**

1. 叙事生成引擎：基于模板 + LLM 辅助的混合生成策略
2. 重要性评估：综合情绪强度、访问频率、内容长度、用户标注等多维度打分
3. 主题聚类：基于语义相似度的自动主题聚类
4. 缓存策略：时间线内容缓存，仅增量更新
5. 数据结构：storylines 表 + storyline_items 表 + 	imeline_cache 表

#### 模块 3.4：记忆回响对比（Echoes）

**模块定位**

记忆回响对比是"成长可视化"的核心功能，通过对同一主题跨时间维度的记忆进行对比分析，让用户直观感知自己的成长、变化与进步。

**核心功能**

1. **主题回响**：选择一个主题，系统自动找出不同时期关于该主题的记忆，并排展示对比
2. **观点演变**：分析用户对同一事物的看法如何随时间变化
3. **能力成长曲线**：基于记忆内容分析，生成各维度能力的成长曲线
4. **情绪轨迹**：展示某段时间内的情绪变化轨迹
5. **成长报告**：定期（月/季/年）生成成长分析报告

**回响类型**

| 回响类型 | 触发方式 | 对比维度 | 输出形式 |
|---------|---------|---------|---------|
| 观点回响 | 手动选择主题 | 对同一事物的看法变化 | 对比卡片 + 分析文字 |
| 能力回响 | 自动生成（月度） | 各维度能力分变化 | 雷达图 + 成长曲线 |
| 情绪回响 | 自动生成（周度） | 情绪状态变化 | 情绪热力图 + 趋势线 |
| 知识回响 | 手动触发 | 知识广度与深度变化 | 知识图谱对比 |
| 习惯回响 | 自动生成（季度） | 行为模式变化 | 习惯追踪图 |

**技术实现要点**

1. 时间切片：将记忆按时间窗口切片，每个切片生成一个"时间点画像"
2. 对比算法：基于向量相似度 + 关键词差异 + 情绪差异的多维度对比
3. 可视化：对比卡片、双轴图、雷达图等多种可视化形式
4. 隐私保护：回响分析仅在本地进行，不上传原始数据
5. 数据结构：echo_analysis 表存储分析结果，支持缓存复用

#### 模块 3.5：潮汐专属历法（Calendar）

**模块定位**

潮汐专属历法是建立使用节律感的模块，通过 7 天小潮 / 30 天大潮的周期性设计，让用户的记忆使用形成自然的节律与仪式感。

**历法体系设计**

**潮汐周日（7 天小潮周期）**

| 星期 | 潮汐日 | 主题 | 推荐活动 |
|------|--------|------|---------|
| 周一 | 涨潮日 | 新启 | 设定本周目标，写入新计划 |
| 周二 | 潮升日 | 积累 | 大量输入，记录学习与思考 |
| 周三 | 高潮日 | 活跃 | 深度检索，回顾与联想 |
| 周四 | 退潮日 | 整理 | 整理记忆，归档与标签 |
| 周五 | 潮落日 | 反思 | 周复盘，输出总结 |
| 周六 | 低潮日 | 沉淀 | 轻度使用，让记忆自然沉淀 |
| 周日 | 平潮日 | 休整 | 完全休息，或随机漫步式回忆 |

**潮汐月历（30 天大潮周期）**

每月分为三个阶段，每阶段 10 天：

| 阶段 | 名称 | 主题 | 核心活动 |
|------|------|------|---------|
| 第 1-10 天 | 新月潮 | 播种期 | 设定月度目标，大量摄入新知 |
| 第 11-20 天 | 满月潮 | 全盛期 | 深度加工，记忆巩固，创造输出 |
| 第 21-30 天 | 残月潮 | 收获期 | 月度复盘，整理归档，遗忘清理 |

**历法功能**

1. 潮汐日历视图：以潮汐历法呈现的日历界面，每日有对应的潮汐状态与推荐活动
2. 潮汐节律提醒：根据潮汐阶段推送不同的使用建议（可关闭）
3. 潮汐仪式：每个潮汐周期切换时有专属的过渡动画与仪式感
4. 潮汐统计：按潮汐周期统计使用数据，观察自己的节律模式

**技术实现要点**

1. 历法计算：基于当前日期自动计算潮汐日/潮汐月阶段
2. 可配置性：用户可调整周期起始日，适配不同的生活节律
3. 与成就系统联动：完整经历一个潮汐周期解锁对应勋章
4. 与赛季系统联动：赛季进度与潮汐周期对齐
5. 数据结构：	ide_calendar_config 表 + 计算函数（无需持久化每日数据）

#### 模块 3.6：赛季征程系统（Seasons）

**模块定位**

赛季征程系统是长期目标驱动模块，以 180 天为一个赛季周期，通过 6 个阶段的目标设定与奖励机制，激励用户长期持续使用并见证自己的成长。

**赛季周期设计**

每个赛季 180 天（约 6 个月），分为 6 个阶段，每阶段 30 天：

| 阶段 | 名称 | 时长 | 核心主题 | 阶段目标 |
|------|------|------|---------|---------|
| S1 | 启程之潮 | 30 天 | 探索与适应 | 熟悉系统功能，建立使用习惯 |
| S2 | 奔涌之潮 | 30 天 | 积累与输入 | 大量记忆写入，构建知识体系 |
| S3 | 澎湃之潮 | 30 天 | 深度加工 | 记忆整理、关联、巩固 |
| S4 | 回响之潮 | 30 天 | 反思与输出 | 回顾总结，知识输出 |
| S5 | 沉淀之潮 | 30 天 | 凝练与升华 | 蒸馏核心认知，遗忘冗余 |
| S6 | 新生之潮 | 30 天 | 回顾与展望 | 赛季总结，规划下一季 |

**赛季奖励体系**

| 奖励类型 | 说明 | 获取方式 |
|---------|------|---------|
| 赛季勋章 | 每个赛季有专属限定勋章 | 完成赛季全部 6 阶段 |
| 赛季天赋点 | 额外的心智天赋点 | 各阶段目标达成 |
| 赛季皮肤 | 专属主题与界面样式 | 赛季排名前 10%（可选竞技模式） |
| 赛季报告 | 详细的个人成长分析报告 | 赛季结束自动生成 |
| 记忆胶囊 | 将赛季精华记忆封存，1 年后解锁 | 赛季结束时手动封存 |

**赛季功能**

1. 赛季看板：当前赛季进度、阶段目标、完成度可视化
2. 赛季任务：每个阶段有 3-5 个核心任务，完成获得奖励
3. 赛季统计：赛季期间的各类使用数据统计与对比
4. 赛季回顾：赛季结束时生成精美的回顾报告
5. 历史赛季：查看历史赛季的数据与成就

**技术实现要点**

1. 赛季引擎：基于时间周期的自动阶段切换与任务判定
2. 任务系统：赛季任务的定义、进度追踪、奖励发放
3. 赛季数据隔离：各赛季数据独立统计，支持跨赛季对比
4. 与成就/天赋/编年史联动：赛季数据反哺其他游戏化模块
5. 数据结构：seasons 表 + season_stages 表 + season_tasks 表 + user_season_progress 表

### 5.3 与核心记忆系统的集成点

游戏化系统不是独立存在的"皮肤"，而是与核心记忆系统深度集成、相互赋能的有机整体。以下是六大游戏化模块与核心记忆系统的主要集成点。

#### 集成点一：事件总线（Event Bus）

核心记忆系统的各类操作事件通过事件总线广播，游戏化系统订阅并响应这些事件，驱动成就判定、天赋进度、赛季任务等逻辑。

**核心事件清单：

| 事件类型 | 触发时机 | 订阅者 |
|---------|---------|--------|
| memory.created | 新记忆写入 | 成就（计数类）、天赋（成长类）、赛季（任务） |
| memory.retrieved | 记忆被检索 | 成就（探索类）、天赋（检索类） |
| memory.updated | 记忆被更新 | — |
| memory.deleted | 记忆被删除 | — |
| memory.linked | 记忆关联建立 | 成就（创造类）、天赋（联想类） |
| layer.migrated | 记忆跨层迁移 | 成就（坚持类）、天赋（存储类） |
| search.executed | 检索执行 | 成就（探索类）、天赋（检索类） |
| dream.completed | 梦境加工完成 | 成就（创造类）、天赋（加工类） |
| user.active | 用户活跃 | 成就（坚持类）、赛季（活跃任务） |

#### 集成点二：天赋效果回写

心智天赋树解锁的天赋效果不是"装饰"，而是实际影响核心记忆系统的运行参数。

**效果回写机制：

| 天赋效果类型 | 影响的核心模块 | 实现方式 |
|-------------|---------------|---------|
| 缓存容量调整 | L0 缓存层 | 动态修改 cache_size 配置 |
| 检索权重调整 | 检索引擎 | 修改 RRF 融合权重 |
| 迁移速度调整 | 潮汐迁移 | 修改迁移批量大小与频率 |
| 遗忘阈值调整 | 梦境引擎 | 修改遗忘曲线参数 |
| 页表容量调整 | 语义页表 | 修改每页原子数量上下限 |

#### 集成点三：数据互通

游戏化系统从核心记忆系统读取数据进行分析与展示，同时将游戏化产生的元数据写回记忆系统，丰富记忆的上下文。

| 数据流 | 方向 | 内容 |
|-------|------|------|
| 记忆内容 → 编年史 | 核心 → 游戏化 | 原始记忆数据用于生成叙事 |
| 记忆内容 → 回响对比 | 核心 → 游戏化 | 原始记忆数据用于对比分析 |
| 勋章解锁 → 记忆元数据 | 游戏化 → 核心 | 勋章解锁事件作为特殊记忆存入 |
| 天赋效果 → 系统参数 | 游戏化 → 核心 | 天赋效果影响系统运行参数 |
| 赛季报告 → 记忆 | 游戏化 → 核心 | 赛季报告作为长期记忆归档 |

#### 集成点四：潮汐节律协同

潮汐专属历法与核心系统的潮汐四相循环控制器协同运作，形成"用户体验节律"与"系统内部节律"的同频共振。

| 协同维度 | 潮汐历法（体验层） | 潮汐控制器（核心层） |
|---------|-------------------|---------------------|
| 日周期 | 每日潮汐状态 | 日内负载驱动的相位切换 |
| 周周期 | 7 天小潮周期 | 每周巩固/清理调度 |
| 月周期 | 30 天大潮周期 | 每月蒸馏/归档调度 |
| 赛季周期 | 180 天赛季 | 赛季级深度加工 |

## 六、Phase 4：自省进化系统（P2，4-6周）

### 6.1 目标与定位

Phase 4 是 M5 潮汐记忆模块的最高阶升级——从"被动工具"进化为"主动成长的智能体"。自省进化系统赋予 M5 自我认知、自我分析、自我升级的能力，使系统能够随着使用不断发现自身不足、寻求改进方案、自主完成优化升级。

**核心定位**

自省进化系统不是"自动写代码"的黑科技，而是一个**辅助决策与自动化执行的智能运维系统**。它的价值在于：
1. 主动发现问题，而不是等用户反馈
2. 自动搜索与评估方案，减少人工调研成本
3. 自动化执行低风险升级，释放人力
4. 提供结构化的升级建议，辅助人工决策

**设计原则**

1. **安全第一**：任何自动升级都必须在安全沙箱中执行，核心系统变更必须人工审核
2. **可解释性**：每一步决策都有清晰的依据与推理过程，用户可追溯、可干预
3. **渐进式自治**：从 L1 全自动（低风险）到 L3 全手动（高风险），逐步提升自动化程度
4. **失败可回滚**：任何升级操作都必须有明确的回滚方案，失败后可快速恢复
5. **人机协同**：系统做"体力活"（搜索、扫描、安装），人做"决策活"（审核、批准、监督）

---

### 6.2 七大组件设计概要

#### 组件 4.1：CapabilityInventory（能力清单管理）

**职责定位**

系统自我认知的基础组件，负责维护当前系统的能力图谱、模块清单、依赖关系与版本信息。是所有进化决策的"自我认知基础"。

**核心功能**

1. **模块注册与发现**：自动扫描系统内所有模块，注册模块元数据（名称、版本、职责、接口、依赖）
2. **能力图谱构建**：基于模块信息构建系统能力图谱，展示各模块提供的能力与依赖关系
3. **版本追踪**：记录每个模块的版本历史与变更记录
4. **能力评估**：定期评估各模块的健康度、性能指标、覆盖率等
5. **差距基线**：提供当前能力基线，供 GapAnalyzer 对比分析

**数据结构**

`
CapabilityInventory
├── modules（模块清单）
│   ├── module_id
│   ├── name / description
│   ├── version / status
│   ├── interfaces（暴露的接口）
│   ├── dependencies（依赖的模块）
│   └── health_score（健康度评分）
├── capabilities（能力清单）
│   ├── capability_id
│   ├── name / description
│   ├── provided_by（由哪些模块提供）
│   └── maturity_level（成熟度：实验/可用/稳定/成熟）
└── dependency_graph（依赖图谱）
    └── 有向图结构：模块 A → 依赖 → 模块 B
`

**技术要点**

- 模块注册方式：装饰器 + 配置文件双轨制
- 能力图谱可视化：可导出为 Mermaid / Graphviz 格式
- 健康度评分：多维度加权（性能 30% + 稳定性 30% + 覆盖率 20% + 活跃度 20%）

---

#### 组件 4.2：GapAnalyzer（缺口分析引擎）

**职责定位**

系统的"问题发现器"，负责从五类信号源中采集数据，分析系统存在的能力缺口，并按优先级排序输出缺口清单。

**五类信号源与权重**

| 信号源 | 权重 | 采集方式 | 分析维度 |
|-------|------|---------|---------|
| 用户未满足需求 | 35% | 对话语义分析、用户反馈采集 | 需求频次、情绪强度、影响面 |
| 任务失败记录 | 25% | 错误日志、监控告警、超时统计 | 失败频次、影响范围、严重程度 |
| 用户主动建议 | 20% | 反馈渠道、对话中提取的建议 | 支持人数、可行性、价值预估 |
| 对标竞品差距 | 15% | 竞品功能矩阵对比、行业趋势分析 | 差距大小、竞争压力、差异化价值 |
| 技术趋势预判 | 5% | 技术雷达、社区热点、论文追踪 | 技术成熟度、适配性、前瞻性 |

**缺口分析流程**

`
信号采集 → 信号清洗 → 缺口识别 → 缺口聚类 → 优先级评分 → 输出缺口清单
    ↑          ↑          ↑          ↑          ↑
  多源采集   去重降噪   模式匹配   主题聚合   加权评分
`

**缺口优先级评分公式**

`
缺口优先级 = 用户需求强度 × 0.35
            + 业务影响程度 × 0.25
            + 技术可行性 × 0.20
            + 竞争紧迫性 × 0.15
            + 战略匹配度 × 0.05
`

**输出格式**

每个缺口包含：缺口描述、影响范围、证据来源、优先级评分、建议解决方向、预估复杂度。

#### 组件 4.3：OpenSourceSearcher（开源方案搜索）

**职责定位**

系统的"方案研究员"，针对 GapAnalyzer 识别出的缺口，自动搜索开源社区中的候选解决方案，进行初步评估与筛选。

**搜索源**

| 来源 | 内容 | 搜索方式 |
|-----|------|---------|
| GitHub | 开源项目、代码仓库 | GitHub Search API + 关键词匹配 |
| PyPI | Python 包 | PyPI API + 名称/描述匹配 |

| HuggingFace | 模型与数据集 | HuggingFace API + 语义搜索 |
| Stack Overflow | 技术问答 | StackExchange API + 问题匹配 |

**评估维度**

对每个候选方案进行多维度评估：

| 维度 | 权重 | 评估指标 |
|-----|------|---------|
| 功能匹配度 | 30% | 满足缺口需求的程度 |
| 社区活跃度 | 20% | Star 数、贡献者数、最近更新时间 |
| 代码质量 | 15% | 测试覆盖率、文档完整度、Issue 响应速度 |
| 许可证友好度 | 15% | 许可证类型（MIT/Apache 优先，GPL 谨慎） |
| 集成难度 | 15% | API 友好度、依赖数量、学习曲线 |
| 安全性 | 5% | 已知漏洞、安全审计记录 |

**输出格式**

每个缺口输出 Top 3 候选方案，包含：方案名称、项目链接、核心功能、评估得分、优缺点分析、推荐度。

#### 组件 4.4：SecurityScanner（安全扫描器）

**职责定位**

系统的"安全守门员"，对 OpenSourceSearcher 筛选出的候选方案进行安全扫描与风险评估，确保引入的第三方代码不会带来安全隐患。

**扫描维度**

| 扫描类型 | 扫描内容 | 工具/方法 |
|---------|---------|----------|
| 已知漏洞扫描 | CVE 漏洞、安全公告 | OSV API / GitHub Advisory API |
| 许可证合规 | 许可证类型、传染性风险 | licensee / 手动分析 |
| 代码静态扫描 | 注入漏洞、硬编码密钥、危险函数 | Bandit / Semgrep |
| 依赖链分析 | 传递依赖的数量与安全性 | pip-audit / npm audit |
| 供应链安全 | 维护者可信度、发布历史 | 手动评估 + 社区指标 |

**安全评级**

| 评级 | 含义 | 处理方式 |
|-----|------|---------|
| A 级 | 安全，可放心使用 | 直接通过 |
| B 级 | 低风险，有小问题但不影响使用 | 记录风险，自动通过 |
| C 级 | 中风险，需人工评估 | 提交人工审核 |
| D 级 | 高风险，存在明显安全问题 | 自动拒绝 |

**输出格式**

安全评估报告：方案名称、安全评级、各维度得分、发现的问题、风险描述、建议措施。

`
升级方案报告

│   ├── 预期收益
│   ├── 风险概览

├── 缺口分析
│   ├── 缺口描述
│   ├── 影响范围
│   └── 优先级与紧迫性
├── 方案评估
│   ├── 候选方案对比
│   ├── 功能匹配度分析
│   ├── 安全评估结果
│   └── 推荐方案说明

│   ├── 安装级别（L1/L2/L3）

- 可执行性检查：自动检查方案的依赖是否满足、是否有冲突
- 人工审核接口：报告生成后自动提交审核队列，等待人工确认

#### 组件 4.6：InstallationEngine（安装执行引擎）

**职责定位**

系统的"动手执行者"，负责根据审核通过的升级方案执行实际的安装与升级操作。根据安装级别的不同，自动化程度也不同。

**三级安装体系**

| 级别 | 名称 | 适用场景 | 自动化程度 | 人工审核 | 回滚方式 |
|-----|------|---------|-----------|---------|---------|
| L1 | 简单安装 | 配置项调整、参数优化、规则更新 | 全自动 | 无需 | 配置回滚 |
| L2 | 中等安装 | 新增模块、接口扩展、非核心功能 | 半自动 | 方案审核 | 模块禁用 + 数据回滚 |
| L3 | 复杂安装 | 核心架构变更、数据迁移、安全策略 | 手动执行 | 全面评审 | 全量备份恢复 |

**L1 简单安装执行流程**

`
接收安装指令 → 验证参数合法性 → 备份当前配置 → 执行配置变更
     → 验证新配置有效性 → 生效新配置 → 记录安装日志 → 返回结果
     ↑ 失败则：回滚到备份配置 + 记录失败原因
`

**L2 中等安装执行流程**

`
人工审核方案 → 创建沙箱环境 → 沙箱安装测试 → 功能验证
     → 安全扫描 → 人工确认上线 → 生产环境安装 → 灰度发布
     → 全量生效 → 记录安装日志
     ↑ 任何环节失败：回滚到上一状态
`

**L3 复杂安装执行流程**

`
方案评审 → 全量备份 → 制定详细执行计划 → 人工逐步执行
     → 每步验证 → 全面回归测试 → 确认生效
     ↑ 任何环节出问题：从备份恢复
`

**技术要点**

- 沙箱执行：L2 及以上安装必须先在隔离沙箱中测试
- 原子操作：每个安装步骤都是原子的，要么成功要么回滚
- 安装日志：详细记录每一步操作与结果，支持审计追溯
- 版本快照：安装前自动创建 Git tag / 数据快照，支持回滚
- 灰度发布：L2/L3 安装支持灰度放量，逐步扩大范围

#### 组件 4.7：EvolutionScheduler（进化调度器）

**职责定位**

系统的"总指挥"，负责协调其他六个组件的工作，管理进化任务的生命周期，选择合适的时机执行升级，确保进化过程不影响系统正常运行。

**核心功能**

1. **任务队列管理**：管理待执行的进化任务队列，支持优先级排序
2. **时机选择**：根据系统负载、潮汐相位、用户活跃度等因素选择最佳执行时机
3. **冲突检测**：检测多个进化任务之间是否存在依赖冲突或资源竞争
4. **执行编排**：按依赖关系编排任务执行顺序，支持并行与串行执行
5. **进度追踪**：实时追踪进化任务的执行进度与状态
6. **结果汇总**：汇总进化结果，生成进化报告，更新能力清单

**进化时机策略**

| 时机类型 | 触发条件 | 适合的任务级别 |
|---------|---------|---------------|
| 低潮期执行 | 系统处于潮汐低潮期，用户活跃度低 | L2 / L3 大型升级 |
| 退潮期执行 | 系统处于潮汐退潮期，负载中等 | L1 / L2 中小型升级 |
| 立即执行 | 安全补丁、紧急修复 | 紧急安全更新 |
| 定时执行 | 预设的维护窗口 | 定期维护类任务 |
| 手动触发 | 管理员手动触发 | 任意级别 |

**任务状态机**

`
待审核 → 已批准 → 排队中 → 执行中 → 验证中 → 已完成
  ↓        ↓       ↓       ↓       ↓
  已拒绝   已取消   已暂停   失败回滚  验证失败
`

### 6.3 三级安装体系详解

三级安装体系是自省进化系统的核心安全机制，确保自动化升级的风险可控。以下对每一级别进行详细说明。

#### L1 简单安装（全自动）

**定义**：不涉及代码变更，仅修改配置参数或规则的低风险操作。

**典型场景**

- 调整 QoS 队列参数（令牌速率、队列长度等）
- 修改潮汐迁移的时间阈值
- 调整遗忘曲线参数
- 更新成就勋章的解锁条件
- 修改天赋效果数值
- 调整检索权重参数

**执行保障**

- 参数合法性校验（类型、范围、依赖关系）
- 配置变更前自动备份
- 变更后立即验证（如导致系统异常则自动回滚）
- 完整的操作审计日志

**回滚方式**：配置文件回滚，秒级恢复

#### L2 中等安装（半自动）

**定义**：新增独立模块或扩展非核心功能，不影响核心链路的中等风险操作。

**典型场景**

- 安装新的嵌入模型后端
- 新增一种检索策略（如 BM25 替代 TF-IDF）
- 新增游戏化模块
- 新增导出格式支持
- 新增通知渠道（如邮件、Webhook）

**执行保障**

- 沙箱环境先安装测试
- 功能验证测试自动执行
- 安全扫描通过
- 人工审核方案与测试结果
- 灰度发布（先 10% → 50% → 100%）
- 实时监控核心指标

**回滚方式**：禁用新模块 + 数据回滚（如有数据变更），分钟级恢复

#### L3 复杂安装（手动）

**定义**：涉及核心架构变更、数据结构迁移、安全策略调整等高风险操作。

**典型场景**

- 核心存储引擎替换（如 SQLite → PostgreSQL）
- 向量索引算法大版本升级
- 数据结构重构（如引入新的记忆组织模型）
- 加密算法变更或密钥体系升级
- 核心接口协议变更（如 A2A 协议大版本升级）

**执行保障**

- 全量数据备份
- 详细的执行计划与回滚方案
- 多人评审（技术负责人 + 安全负责人 + 产品负责人）
- 人工逐步执行，每步验证
- 全面回归测试
- 专人值守监控
- 预设回滚触发条件

**回滚方式**：从备份恢复 + 代码回滚，小时级恢复

### 6.4 安全与隐私保障

自省进化系统涉及代码执行与系统变更，安全是重中之重。以下从多个维度设计安全保障机制。

#### 沙箱执行机制

| 沙箱层级 | 适用场景 | 隔离方式 | 资源限制 |
|---------|---------|---------|---------|
| 进程级沙箱 | L1 配置变更 | 独立进程 + 权限降级 | CPU/内存限制 |
| 容器级沙箱 | L2 模块安装 | Docker 容器 + 网络隔离 | 完整资源限制 |
| VM 级沙箱 | L3 架构变更 | 独立虚拟机 + 快照 | 完全隔离 |

#### 权限最小化原则

- 进化系统运行在独立的低权限账户下
- 仅授予必要的文件系统访问权限
- 网络访问白名单制，仅允许访问预设的可信源
- 核心系统文件只读挂载
- 数据库访问仅授予必要的表权限

#### 代码安全审查

- 所有引入的第三方代码必须经过 SecurityScanner 扫描
- 自动生成的代码必须经过语法检查 + 静态分析
- 执行前进行哈希校验，确保代码未被篡改
- L2/L3 安装必须人工审核代码变更

#### 数据隐私保护

- 缺口分析时对用户数据进行脱敏处理
- 开源搜索不发送任何用户原始数据
- 升级过程中数据迁移使用加密传输
- 沙箱环境中的数据在测试结束后自动清理

#### 审计与追溯

- 所有进化操作完整记录审计日志
- 日志包含：操作人/操作时间/操作内容/执行结果/回滚点
- 日志不可篡改（追加写入 + 哈希链）
- 支持操作追溯与问题定位

#### 紧急制动机制

- 全局"暂停进化"开关，一键停止所有进化任务
- 异常检测：核心指标异常时自动中止正在执行的升级
- 回滚按钮：每个升级任务都有明确的回滚按钮
- 安全人员可随时介入并接管

1. **功能开关回滚**：全局关闭自省进化系统，所有进化任务停止，系统回退到 Phase 3 纯功能模式
2. **数据清理**：进化系统产生的数据与核心记忆数据物理隔离，可单独清理
3. **代码回滚**：Git revert 到 Phase 4 前的 commit
4. **系统恢复**：如 L3 安装导致严重问题，从备份完整恢复

## 七、技术风险与应对

### 7.1 架构风险

| 风险编号 | 风险描述 | 影响程度 | 发生概率 | 风险等级 | 应对措施 |
|---------|---------|---------|---------|---------|---------|

### 8.4 关键路径与依赖

关键路径（决定最短工期的任务链）：

```
M0 启动
  -> 任务 1.3 BaseSQLLayer 基类抽取
    -> 任务 1.5 批量操作 API
    -> 任务 1.6 分页查询 API
      -> 任务 2.1 可选加密原文存储
      -> 任务 2.3 语义页表与原子图谱
        -> 任务 2.5 梦境引擎
        -> 任务 2.4 潮汐四相循环控制器
          -> 任务 3.1~3.6 游戏化六大模块（并行）
            -> 任务 4.1 CapabilityInventory
              -> 任务 4.2 GapAnalyzer
                -> 任务 4.6 InstallationEngine
                -> 任务 4.7 EvolutionScheduler
                  -> M5 全量上线
```

关键依赖说明：
1. BaseSQLLayer 基类是 Phase 1 的核心，批量 API、分页 API、后续存储层改造都依赖它
2. 加密原文存储是 Phase 2 的基础能力，语义组织、梦境引擎都依赖原文数据
3. 潮汐四相控制器是游戏化与进化系统的时间基础，多个模块依赖潮汐节律
4. CapabilityInventory 是进化系统的自我认知基础，其他进化组件都依赖它
5. InstallationEngine 是进化系统的执行核心，需在其他组件成熟后才集成

可并行任务：
- Phase 1 中：A2A 协议 + QoS 引擎 与 BaseSQLLayer 可并行开发
- Phase 2 中：HNSW 索引升级、SQLite 性能优化、缓存联动 可与其他任务并行
- Phase 3 中：六大游戏化模块可并行开发（依赖统一的事件总线）
- Phase 4 中：GapAnalyzer + OpenSourceSearcher + SecurityScanner 可并行开发
