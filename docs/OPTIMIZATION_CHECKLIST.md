## 3. 优先级排序（按严重等级 + 影响范围）

### Top Priority（必须立即处理）

1. **SC-001** (P0) - 默认密钥明文存储 - 全系统安全风险
2. **SC-002** (P0) - CORS_ORIGINS=* - CSRF 攻击风险
3. **SC-003** (P1) - WAF 监控模式 - 生产环境攻击不拦截
4. **SC-004** (P1) - M8 默认弱密码 - 管理员账户接管风险
5. **AR-001** (P1) - 遗留代码未清理 - 维护成本+安全风险
6. **TS-001** (P1) - M0 测试覆盖率极低 - 核心管控台质量风险
7. **CQ-001** (P1) - 模块配置硬编码 - 扩展性差
8. **AR-002** (P1) - 双配置系统并行 - 维护成本高
9. **TS-004** (P1) - E2E 测试缺失 - 整体质量无保障
10. **OP-003** (P1) - 升级回滚机制不完善 - 生产风险

## 主要发现摘要

**最严重的问题集中在三个方面：**

1. **安全方面（最紧急）**：默认密钥明文存储在 yunxi.env 中包含所有模块的默认密钥（JWT secret、admin token 等），生产环境若未强制更换，存在严重安全隐患。CORS 允许所有来源 + WAF 监控模式不拦截，都是生产环境必须立即修复的 P0/P1 问题。

2. **架构方面（最大技术债）**：
   - 双配置系统（每个模块都有新旧两套配置）维护成本高
   - archive/ 和 legacy 目录下大量遗留代码未清理
   - 13 个模块的依赖管理分散，缺少统一的包管理和服务发现

3. **质量方面（最大风险）**：
   - M0 主理人管控台作为核心入口，测试覆盖率极低
   - 端到端集成测试几乎空白
   - 空异常处理吞掉大量异常，故障排查困难

**整体评估**：云汐系统经过四个阶段迭代后功能完备性较好，架构设计有前瞻性，但快速迭代过程中积累了较多技术债。安全和测试是最需要优先补齐的短板。
