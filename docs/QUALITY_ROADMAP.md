### 核心问题诊断

**架构层面**：M8 职责过重（737个API端点，占系统近一半），认证职责重叠（M0/M8/M12三套），双配置系统并行过渡。

**质量层面**：测试覆盖率两极分化（头部模块60-70%，尾部模块<10%），E2E测试缺失，587处TODO待处理。

**安全层面**：基础防护完善（WAF/限流/输入验证/数据脱敏），但 JWT 仍用 HS256，API Key 管理分散。

**功能层面**：硬件模块全模拟，可视化前端缺失多，API-Gateway 功能薄弱。

## 总体战略

### 指导原则

1. **安全优先**：生产就绪前必须消除所有已知安全风险
2. **质量打底**：测试覆盖率是质量的基础，不达标不上生产
3. **架构护航**：先解耦再扩展，避免在混乱架构上堆功能
4. **用户价值**：每个阶段都要有可感知的产品提升
5. **可回滚**：每个阶段设独立回滚锚点，风险可控

### 阶段划分

| 阶段 | 代号 | 核心目标 | 预计周期 | 关键产出 |
|------|------|---------|---------|---------|
| 第一阶段 | 安全筑基 | 消除所有P1级安全隐患，安全体系达标 | 2周 | 安全审计通过 |
| 第二阶段 | 质量攻坚 | 测试覆盖率突破60%，核心模块达标 | 3周 | 覆盖率报告 + E2E套件 |
| 第三阶段 | 架构解耦 | M8职责拆分，统一配置认证，架构清晰化 | 4周 | 架构评审通过 |
| 第四阶段 | 功能完善 | 薄弱模块补强，硬件/前端/网关升级 | 4周 | 功能完整度80%+ |
| 第五阶段 | 生产就绪 | 性能优化、运维体系、容灾演练 | 3周 | V1.0 发布 |

总周期：约 16 周（4 个月）

## 第一阶段：安全筑基（2周）

### 阶段目标

消除所有 P1 级安全风险，建立纵深防御体系，通过内部安全审计。

### 关键指标

- JWT 算法 100% 迁移到 RS256
- API Key 统一管理率 100%
- 审计日志覆盖率 95%+
- 安全测试用例 200+
- 代码安全扫描零高危

### 工作任务

#### 1.1 JWT 非对称加密迁移（SC-005, P2→P1）

**优先级**：P1 | **工作量**：M | **负责人**：安全组

将所有模块的 JWT 从 HS256（对称加密）迁移到 RS256（非对称加密）：

- 生成 RSA 密钥对（2048位以上），密钥文件安全存储
- 改造 shared/core/auth 中的 JWT 工具，支持 RS256
- 迁移 8 个模块的 JWT secret 配置
- 提供密钥轮换机制
- 兼容旧 Token 的过渡期方案

**验收标准**：
- 所有模块 JWT 默认使用 RS256
- 密钥轮换可在不中断服务的情况下完成
- 旧 Token 有 7 天过渡期自动失效

#### 1.2 API Key 统一管理（SC-010, P3→P1）

**优先级**：P1 | **工作量**：M | **负责人**：平台组

建立统一的 API Key 管理中心：

- 在 M12 安全盾或 M8 控制塔建立 API Key 管理模块
- 支持 Key 的创建、吊销、轮换、配额管理
- 所有服务间调用统一使用管理中心的 Key
- Key 分级：管理级 / 服务级 / 只读级
- 自动过期和轮换提醒

**验收标准**：
- 不再有任何硬编码的 API Key
- 所有服务间调用通过统一 Key 管理
- Key 使用审计可追溯

#### 1.3 审计日志全覆盖（SC-007, P2→P1）

**优先级**：P1 | **工作量**：L | **负责人**：安全组

补全审计日志覆盖：

- 所有管理操作（用户/角色/权限/配置修改）必须有审计记录
- 所有安全事件（登录失败/攻击检测/异常访问）必须有审计
- 所有数据变更（创建/更新/删除）关键操作有审计
- 审计日志独立存储，防篡改
- 支持审计日志查询和导出

**验收标准**：
- 管理操作审计覆盖率 100%
- 安全事件审计覆盖率 100%
- 审计日志查询响应 < 2s

#### 1.4 代码安全扫描集成

**优先级**：P2 | **工作量**：S | **负责人**：质量组

建立代码安全扫描机制：

- 集成 Bandit（Python 安全扫描）
- 集成 Semgrep 规则集
- CI/CD 流水线自动扫描
- 高危漏洞阻断合并
- 每周安全扫描报告

**验收标准**：
- CI 流水线安全扫描通过率 100%
- 零高危安全漏洞
- 中危漏洞 < 5 个

#### 1.5 安全测试补全

**优先级**：P1 | **工作量**：M | **负责人**：质量组

补充安全专项测试：

- SQL 注入测试套件（50+ 用例）
- XSS 攻击测试套件（50+ 用例）
- 认证绕过测试（30+ 用例）
- 权限提升测试（30+ 用例）
- 敏感数据泄露测试（20+ 用例）

**验收标准**：
- 安全测试用例总数 200+
- 全部通过
- OWASP Top 10 全覆盖

### 阶段产出

- 安全审计报告 V1.0
- JWT RS256 迁移指南
- API Key 管理系统
- 审计日志查询系统
- 安全测试套件（200+ 用例）

## 第五阶段：生产就绪（3周）

### 阶段目标

性能优化、运维体系完善、容灾演练，达到生产可用标准，发布 V1.0。

### 关键指标

- 核心接口 P95 延迟 < 200ms
- 系统可用性 99.9%
- 故障恢复时间 < 5 分钟
- 备份恢复成功率 100%
- 压测通过目标 QPS 的 150%

### 工作任务

#### 5.1 性能优化专项

**优先级**：P1 | **工作量**：L | **负责人**：性能组

全系统性能优化：

- **数据库优化**：
  - SQLite → PostgreSQL 迁移方案（生产环境）
  - 慢查询优化（Top 20 慢查询）
  - 索引优化
  - 连接池管理
  - 读写分离设计

- **缓存优化**：
  - 热点数据缓存策略
  - 多级缓存（内存 + Redis）
  - 缓存预热机制
  - 缓存穿透/击穿/雪崩防护

- **向量检索优化**：
  - FAISS 增量索引
  - 向量量化压缩
  - 检索性能基准测试

**验收标准**：
- 核心 API P95 延迟 < 200ms
- 数据库查询平均延迟 < 50ms
- 系统吞吐量提升 50%+

#### 5.2 运维体系完善

**优先级**：P1 | **工作量**：L | **负责人**：运维组

建立完善的运维体系：

- **滚动升级机制（OP-003, P1）**：
  - 蓝绿部署 / 金丝雀发布
  - 零停机升级
  - 自动回滚机制
  - 版本管理和回退

- **监控告警完善**：
  - Grafana 可视化面板（系统/业务/安全三类）
  - 告警分级和通知策略
  - 告警收敛和降噪
  - 值班和告警响应流程

- **日志体系完善**：
  - 集中式日志收集（ELK / Loki）
  - 日志查询和分析
  - 日志归档策略
  - 日志告警规则

**验收标准**：
- 支持滚动升级，零停机
- 监控指标覆盖率 95%+
- 告警响应时间 < 5 分钟
- 日志查询响应 < 3s

#### 5.3 容灾与高可用

**优先级**：P1 | **工作量**：L | **负责人**：运维组 + 架构组

建立高可用和容灾体系：

- **高可用架构**：
  - 关键模块多实例部署
  - 负载均衡
  - 服务发现
  - 健康检查和自动摘除

- **数据容灾**：
  - 数据库主从复制
  - 定时备份 + 增量备份
  - 异地备份
  - 备份恢复演练

- **故障演练**：
  - 模块宕机恢复演练
  - 数据库故障切换演练
  - 网络分区演练
  - 全系统故障恢复演练

**验收标准**：
- 单模块故障不影响整体服务
- 数据库故障切换时间 < 1 分钟
- 备份恢复成功率 100%
- 故障恢复时间 RTO < 5 分钟
- 数据丢失点 RPO < 5 分钟

#### 5.4 安全加固与合规

**优先级**：P2 | **工作量**：M | **负责人**：安全组

生产环境安全加固：

- 等保合规自查
- 数据分类分级
- 隐私保护合规
- 安全审计报告
- 渗透测试
- 安全应急预案

**验收标准**：
- 通过内部安全审计
- 渗透测试无高危漏洞
- 安全应急预案完整

**验收标准**：
- 全量测试通过率 100%
