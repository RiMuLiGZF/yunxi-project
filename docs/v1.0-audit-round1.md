## 执行摘要

本次大检查从 5 个维度全面审查了云汐系统 V1.0 的质量状态。共发现 **55 个问题**，其中 **6 个严重问题**需要立即处理，**15 个高优先级问题**需要在 V1.1 前修复。

**最紧急的 3 个问题**：
1. 🔴 默认 Token / JWT 密钥硬编码 — 生产环境将被直接接管
2. 🔴 终端命令技能命令注入漏洞 — 可执行任意系统命令
3. 🔴 M7 工作流 eval/exec 沙箱逃逸风险 — 恶意工作流可远程代码执行

**最大的架构债务**：
- M8-control-tower 职责严重过载（上帝模块），45 个 router 覆盖 15+ 业务域
- API 响应格式至少 3 套并存，前端对接成本高
- 配置管理至少 4 套方案并存

**最大的质量债务**：
- 前端 0 测试覆盖
- 根目录 tests/ 中 240 处 `pytest.skip` 掩盖真实失败
- 测试文件位置混乱，重复文件多

## 一、严重问题（Critical）— 必须立即修复

> 共 6 个，如不修复将导致系统被入侵或完全不可用

### SEC-001：全局模块默认 Token 硬编码

- **严重程度**：🔴 严重 (CVSS 9.8)
- **所在文件**：`shared/core/config.py` 第 959-1067 行
- **问题**：`GlobalModuleConfig` 为 13 个模块定义了默认 admin token，均为可预测值（如 `yunxi-m8-admin-token-2026`）
- **攻击场景**：攻击者直接用默认 Token 获得所有模块管理员权限
- **修复方案**：
  1. 删除所有默认 Token 值，改为空字符串
  2. 生产环境启动时强制校验必须通过环境变量显式配置
  3. 首次启动自动生成随机 Token 并输出到安全日志
- **预估工作量**：2 小时
- **负责人**：安全组

### SEC-002：全局 JWT 默认密钥硬编码

- **严重程度**：🔴 严重 (CVSS 9.8)
- **所在文件**：`shared/core/config.py` 第 1072 行
- **问题**：`jwt_secret` 默认值为 `yunxi-jwt-secret-key-2026`，公开弱密钥
- **攻击场景**：攻击者用该密钥伪造超级管理员 JWT，完全控制系统
- **修复方案**：
  1. 默认值设为空字符串
  2. 生产环境强制校验 JWT 密钥必须配置且长度 ≥ 32 字节
  3. 优先使用 RS256 非对称签名
- **预估工作量**：2 小时
- **负责人**：安全组

### SEC-003：终端命令技能命令注入漏洞

- **严重程度**：🔴 严重 (CVSS 9.8)
- **所在文件**：`M4-scene-engine/src/services/skills/terminal_command_skill.py` 第 263-300 行
- **问题**：`shell=True` 模式下命令字符串直接来自用户输入，可注入任意命令
- **攻击场景**：`command="rm -rf / ; echo done"` + `shell=true` 删除所有文件
- **修复方案**：
  1. 彻底移除 `shell=True` 选项，所有命令以列表形式传递
  2. 对命令和参数进行严格白名单校验
  3. 在沙箱环境中执行命令，限制权限和资源
- **预估工作量**：4 小时
- **负责人**：M4 模块组

### ARC-001：M7 工作流 eval/exec 沙箱逃逸风险

- **严重程度**：🔴 严重
- **所在文件**：
  - `M7-workflow-builder/src/services/executor.py` 第 1390 行
  - `M7-workflow-builder/src/services/nodes.py` 第 781 行
  - `shared/business/builtin_tools.py` 第 158 行
- **问题**：`exec()` 执行用户代码 + `eval()` 执行表达式，沙箱限制可能被绕过
- **攻击场景**：恶意构造的工作流可实现远程代码执行
- **修复方案**：
  1. 条件表达式使用 AST 解析白名单方案替代 eval
  2. 自定义积木执行引入 RestrictedPython 或 PyPy 沙箱
  3. 对所有用户提供的代码进行静态安全扫描
  4. 增加资源限制（内存、CPU时间）
- **预估工作量**：3-5 天
- **负责人**：M7 模块组

### QAL-001：测试文件位置混乱，大量重复测试文件

- **严重程度**：🔴 严重
- **所在位置**：`M8-control-tower/backend/` 与 `M8-control-tower/backend/tests/`
- **问题**：至少 8 组同名测试文件同时存在于 backend 根目录和 tests/ 子目录，内容高度相似
- **影响**：pytest 可能重复执行，修改时漏改，无法区分哪份有效
- **修复方案**：
  1. 清理 backend 根目录下的测试文件，统一收敛到 backend/tests/
  2. 通过 diff 对比两份文件，合并有价值的差异后删除重复文件
- **预估工作量**：1 天
- **负责人**：质量组

### TST-001：根目录 tests/ 中 240 处 pytest.skip 掩盖真实失败

- **严重程度**：🔴 严重
- **所在位置**：`tests/` 目录（特别是 test_m8/、test_m9/、test_m11/、integration/、e2e/）
- **问题**：大量测试用 try/except 包裹整个测试体，任何异常都导致测试被静默跳过
- **影响**：测试完全失去质量保障意义，"通过"的测试实际是被跳过的
- **修复方案**：
  1. 将需要真实服务运行的测试明确标记为 `@pytest.mark.e2e` 或 `@pytest.mark.integration`
  2. 使用 proper fixtures 和 mock 替代真实服务依赖来编写单元测试
  3. 移除 try/except + skip 模式，让测试真实失败
- **预估工作量**：3-5 天
- **负责人**：质量组

## 二、高优先级问题（High）— V1.1 前必须修复

> 共 15 个，严重影响系统安全、可维护性或用户体验

### 安全类

#### SEC-004：M8 控制塔 JWT 实现与统一认证体系割裂

- **严重程度**：🟠 高 (CVSS 7.5)
- **所在文件**：`m8-control-tower/backend/auth.py` 第 33-42 行
- **问题**：M8 有独立的 JWT 实现，缺少 Token 黑名单、密钥轮换等安全机制
- **修复方案**：迁移到 `shared/core/auth/jwt.py` 的统一 `JWTHandler`
- **预估工作量**：1 天

#### SEC-005：M2 技能集群 SQL 注入风险

- **严重程度**：🟠 高 (CVSS 7.5)
- **所在文件**：
  - `M2-skills-cluster/skill_cluster/db/skill_repository_base.py` 第 65-100 行
  - `M2-skills-cluster/skill_cluster/market/registry.py` 第 584、619 行
- **问题**：`order_by` 参数直接拼接到 SQL，未进行白名单校验
- **修复方案**：对 order_by 列名进行白名单校验，参考 `_validate_identifier` 模式
- **预估工作量**：4 小时

#### SEC-006：API 网关 JWT 开发模式不验证签名

- **严重程度**：🟠 高 (CVSS 7.5)
- **所在文件**：`API-Gateway/src/middleware/auth.py` 第 275-300 行
- **问题**：jose 库不可用时降级为仅解码 payload，完全不验证签名
- **修复方案**：移除降级模式，jose 不可用时直接拒绝所有 JWT 认证
- **预估工作量**：2 小时

#### SEC-007：密码哈希开发模式降级为 SHA256

- **严重程度**：🟠 高 (CVSS 7.5)
- **所在文件**：`shared/core/auth/password.py` 第 56-62 行
- **问题**：`YUNXI_DEV_MODE=1` 时密码哈希回退到 SHA256 + salt，无慢哈希
- **修复方案**：移除 fallback 模式，bcrypt 不可用时直接抛出异常
- **预估工作量**：2 小时

#### SEC-008：测试配置文件泄露敏感信息

- **严重程度**：🟠 高 (CVSS 7.2)
- **所在文件**：`.env.test` 第 14-19 行
- **问题**：明文存储测试账号密码 `admin123456` 和测试 JWT 密钥
- **修复方案**：测试环境密码使用随机生成值，添加 CI 检查防止泄露
- **预估工作量**：2 小时

### 架构类

#### ARC-002：M8-control-tower 职责严重过载（上帝模块）

- **严重程度**：🟠 高
- **所在位置**：`M8-control-tower/backend/routers/`（45 个 router 文件）
- **问题**：M8 承担了 15+ 个完全不同的业务域，违反单一职责原则
- **影响**：任何业务变更都影响 M8，维护成本高，发布风险大
- **修复方案**：
  1. 业务类 router 迁移至 M4 场景引擎
  2. 算力调度中台独立为子模块
  3. 自进化引擎独立为专用服务
  4. M8 仅保留平台级能力（监控、模块管理、部署、权限、审计）
- **预估工作量**：2-3 周（分阶段）
- **负责人**：架构组

#### ARC-003：API 响应格式不统一，3+ 套体系并存

- **严重程度**：🟠 高
- **问题**：M8 模式 / ApiResponse 模式 / API-Gateway 模式 / M1 模式，4 套响应格式
- **影响**：前端对接成本高，跨模块调用不一致，错误处理复杂
- **修复方案**：
  1. 明确标准响应格式（推荐 ApiResponse + 6位错误码）
  2. 所有模块逐步迁移，旧格式通过兼容层支持
  3. 在 shared 中提供标准 FastAPI 响应模型和全局异常处理器
- **预估工作量**：1-2 周
- **负责人**：架构组

#### ARC-004：shared 目录三层冗余，迁移不彻底

- **严重程度**：🟠 高
- **所在位置**：`shared/` 根目录 vs `shared/core/` vs `shared/business/`
- **问题**：shared 根目录 30 个文件都是兼容存根 + DeprecationWarning，但 M8 等模块仍大量使用旧路径
- **修复方案**：
  1. 全项目批量替换导入路径
  2. 删除 shared 根目录存根文件
  3. 清理 5 个空壳目录
- **预估工作量**：2-3 天

#### ARC-005：M8 main.py 过长（969 行），职责混杂

- **严重程度**：🟠 高
- **所在文件**：`M8-control-tower/backend/main.py`
- **问题**：create_app() 混合了模块导入、CORS、45 个 router 注册、健康检查、系统检测、静态文件等
- **修复方案**：
  1. 健康检查抽离为独立模块
  2. 系统检测抽离为独立服务
  3. router 注册通过配置列表 + 循环实现
- **预估工作量**：1 天
